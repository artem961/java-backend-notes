# Авторизация

Авторизация отвечает на вопрос «**что тебе можно?**» — проверка прав уже
установленной личности. В Spring Security она работает на двух уровнях:
правила по URL в конфигурации и аннотации на методах.

## Роли и authorities

Права пользователя — коллекция `GrantedAuthority` в `Authentication`.
**Роль** — это authority с префиксом `ROLE_`: роль `ADMIN` хранится как
`ROLE_ADMIN`. Отсюда вечная путаница:

- `hasRole("ADMIN")` — проверяет authority `ROLE_ADMIN` (префикс добавляется сам);
- `hasAuthority("ROLE_ADMIN")` — то же самое, но префикс пишется явно;
- `hasAuthority("orders:read")` — «мелкое» право без префикса.

Роли — крупные группы прав (админ, оператор), authorities — точечные
разрешения. В JWT-мире authorities обычно строятся из scope/claims токена.

## Уровень 1: правила по URL

```java
.authorizeHttpRequests(auth -> auth
    .requestMatchers("/api/public/**").permitAll()
    .requestMatchers("/api/admin/**").hasRole("ADMIN")
    .requestMatchers(HttpMethod.POST, "/api/orders").hasAuthority("orders:write")
    .anyRequest().authenticated())
```

Проверяет `AuthorizationFilter` до контроллера. Правила применяются
**сверху вниз до первого совпадения** — специфичные раньше общих;
`anyRequest()` — последним. Ошибка порядка — классический источник
«дырок» и лишних 403.

## Уровень 2: аннотации на методах

Включаются `@EnableMethodSecurity`; работают через AOP-прокси на любых
бинах (обычно на сервисах):

```java
@PreAuthorize("hasRole('ADMIN')")
public void deleteUser(long id) { ... }

// SpEL: доступ к аргументам метода и principal
@PreAuthorize("#userId == authentication.name or hasRole('ADMIN')")
public ProfileDto getProfile(String userId) { ... }
```

`@PreAuthorize` — проверка до вызова, самая используемая; есть
`@PostAuthorize` (по результату) и `@PreFilter`/`@PostFilter` (фильтрация
коллекций — осторожно с большими списками). Выражения — SpEL, со ссылками
на аргументы (`#userId`) и текущего пользователя (`authentication`).

Поскольку это AOP-прокси — работают **все прокси-грабли**: самовызов мимо
прокси, не-public методы, объект через `new`, и главное — **без
`@EnableMethodSecurity` аннотации молча игнорируются**. «`@PreAuthorize`
стоит, а доступ открыт» — почти всегда одно из этих.

## Как уровни сочетаются

Практичная схема:

- **URL-правила** — грубый периметр: что вообще требует входа, что только
  для админов. Срабатывают раньше и дешевле.
- **Методные аннотации** — тонкая логика у данных: «свой профиль или
  админ», проверки на сервисном слое, куда можно прийти не только по HTTP.
- **Владение объектом** (пользователь видит только свои заказы) — это тоже
  авторизация: проверяется в сервисе/запросе (`WHERE user_id = :current`),
  а не только аннотацией. Забытая проверка владения — классическая уязвимость
  IDOR: «поменял id в URL и увидел чужой заказ».

Отказ в авторизации — `AccessDeniedException` → 403 (`AccessDeniedHandler`).
Отличать от 401: 401 — «не знаю, кто ты», 403 — «знаю, но нельзя».

## Как ответить на интервью

Коротко: авторизация — «что можно» уже аутентифицированному; права —
authorities, роль — authority с префиксом `ROLE_` (`hasRole("ADMIN")` ==
`hasAuthority("ROLE_ADMIN")`). Два уровня: правила по URL
в `authorizeHttpRequests` (сверху вниз до первого совпадения — порядок
критичен) и `@PreAuthorize` с SpEL на методах (требует
`@EnableMethodSecurity` и живёт на AOP-прокси со всеми его граблями).
Плюс проверка владения данными в запросах — иначе IDOR. 401 — не
аутентифицирован, 403 — аутентифицирован, но прав нет.
