# Сессии, токены и JWT

После входа сервер должен узнавать пользователя в последующих запросах.
Два подхода: **сессии** (состояние на сервере) и **токены** (состояние
у клиента). Выбор между ними определяет архитектуру всей аутентификации.

## Сессии: состояние на сервере

Классика: при входе сервер создаёт запись сессии у себя, клиенту ставится
кука `JSESSIONID`. Каждый запрос — кука → поиск сессии → пользователь.

Плюсы: **отзыв мгновенный** (удалил сессию — пользователь разлогинен),
данные сессии не покидают сервер. Минусы: сервер **stateful** — при
нескольких инстансах нужна либо липкость балансировщика, либо общее
хранилище сессий (Spring Session + Redis); куки требуют защиты от CSRF.

## Токены: состояние у клиента

При входе сервер выдаёт **подписанный токен**; клиент шлёт его в заголовке
`Authorization: Bearer <token>`. Сервер не хранит ничего — вся информация
в токене, подлинность гарантирует подпись.

## JWT: устройство

JWT (JSON Web Token) — три части через точку, каждая в base64:

```
header.payload.signature
```

- **header** — алгоритм подписи (`RS256`...);
- **payload** — **claims**: `sub` (кто), `exp` (до когда), `iat` (когда
  выдан), плюс свои — роли, id;
- **signature** — подпись первых двух частей ключом сервера.

Проверка токена = проверка подписи + `exp`. Никакого похода в БД —
поэтому JWT идеален для микросервисов: сервис проверяет токен сам,
зная только публичный ключ выдавшего (при асимметричной подписи RS256;
ключи публикуются как JWKS-эндпоинт).

Две вещи, которые обязательно спрашивают:

- **Payload не зашифрован** — только закодирован base64: любой прочитает.
  Секретов в claims не бывает. Подпись гарантирует **целостность**,
  не конфиденциальность.
- **JWT нельзя отозвать**: он валиден до `exp` — сервер про него ничего
  не хранит. Компрометация — токен работает до истечения. Это плата
  за statelessness.

## Access + refresh: стандартная схема

Проблему отзыва смягчают парой токенов:

- **access-токен** — короткоживущий (минуты), им подписывается каждый запрос;
- **refresh-токен** — долгоживущий, хранится бережно и **известен серверу**:
  им получают новый access, и его можно отозвать.

Компрометация access ограничена минутами жизни; управление сессией
(разлогин, бан) работает через refresh. Это компромисс между stateless
(access проверяется без БД) и контролем (refresh — под контролем сервера).

## В Spring: resource server

Приём JWT в API включается конфигурацией ресурс-сервера:

```java
http.oauth2ResourceServer(o -> o.jwt(withDefaults()));
```

```yaml
spring:
  security:
    oauth2:
      resourceserver:
        jwt:
          issuer-uri: https://auth.example.com   # откуда брать ключи (JWKS)
```

Фильтр валидирует подпись и exp, строит `Authentication` из claims
(маппинг ролей настраивается конвертером). Для stateless API отключают
сессии (`STATELESS`) и CSRF — CSRF-атака работает на куках, которые браузер
шлёт сам; Bearer-заголовок руками подставляет только твой код.

## Как выбрать

Практическое правило: браузерное приложение с одним бэкендом — сессии
(+ Spring Session при нескольких инстансах) проще и контролируемее;
публичный API, микросервисы, мобильные клиенты — JWT access + refresh.

## Как ответить на интервью

Коротко: сессии — состояние на сервере, кука-идентификатор, мгновенный
отзыв, но stateful (нужен общий стор при масштабировании) и CSRF-защита.
JWT — подписанный набор claims у клиента: проверяется подписью без БД
(отлично для микросервисов), но payload читаем всеми (не шифрование!)
и токен не отозвать до `exp` — поэтому схема access (короткий, stateless) +
refresh (долгий, хранится сервером, отзываемый). В Spring — resource server
с `issuer-uri`, сессии `STATELESS`, CSRF для Bearer-API отключается.
