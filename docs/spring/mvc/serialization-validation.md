# Сериализация и валидация

Между JSON в теле запроса и объектом в аргументе контроллера стоят два
механизма: Jackson (превращает байты в объект и обратно) и Bean Validation
(проверяет, что объект корректен). Оба декларативные, оба — источник
типовых вопросов.

## Jackson: JSON ↔ объект

Spring MVC сериализует через `HttpMessageConverter`, и для JSON это Jackson.
Работает по соглашению: имена полей ↔ ключи JSON, геттеры/сеттеры или
конструктор. Настройки поверх соглашения — аннотациями:

```java
public record OrderDto(
    long id,
    @JsonProperty("order_status") String status,       // другое имя в JSON
    @JsonFormat(pattern = "yyyy-MM-dd") LocalDate date, // формат даты
    @JsonIgnore String internalNote                     // не сериализуется
) { }
```

Что важно понимать:

- **Десериализация не валидирует**: JSON без поля просто даст `null`
  (или 0 для примитива — и это ловушка: отличить «не прислали» от «прислали 0»
  у примитива нельзя, поэтому в DTO — обёртки). Неизвестные поля Boot
  по умолчанию игнорирует.
- **Ошибка формата** (кривой JSON, строка вместо числа) — это
  `HttpMessageNotReadableException` → 400 ещё до вызова контроллера.
- **Даты**: `LocalDate`/`Instant` поддерживаются модулем JavaTime
  (в Boot из коробки), формат по умолчанию — ISO-8601; `@JsonFormat` —
  точечные отклонения.
- Глобальная настройка — через проперти `spring.jackson.*` или свой бин
  `ObjectMapper` (помня, что он переопределит автоконфигурационный).

## Bean Validation: декларативные проверки

Проверки входных данных описываются аннотациями на полях DTO,
а включаются `@Valid` на параметре:

```java
public record CreateOrderRequest(
    @NotBlank String customerName,
    @Email String email,
    @Positive BigDecimal amount,
    @Size(max = 100) List<@Valid OrderItemDto> items   // каскад на вложенные
) { }

@PostMapping
public OrderDto create(@RequestBody @Valid CreateOrderRequest request) { ... }
```

Ходовой набор: `@NotNull` (не null), `@NotBlank` (строка не пуста
и не пробельна), `@NotEmpty` (коллекция не пуста), `@Size`, `@Min`/`@Max`/
`@Positive`, `@Email`, `@Pattern`. Вложенные объекты проверяются только
при `@Valid` на поле — частая ошибка «аннотации есть, а не проверяется».

Провал валидации `@RequestBody` — `MethodArgumentNotValidException` → 400;
в обработчике ошибок из него достаются пофидовые сообщения для клиента.

Кастомное правило — своя аннотация + `ConstraintValidator`; для проверок,
требующих контекста (уникальность email), чаще честнее проверка в сервисе
с доменным исключением.

## Граница ответственности

Слои проверок стоит разделять — это признак зрелого ответа:

- **Формат** (валидный JSON, типы) — Jackson, 400 автоматически.
- **Синтаксическая корректность полей** (не пусто, диапазон, формат
  email) — Bean Validation на DTO, 400 со списком ошибок.
- **Бизнес-правила** (заказ можно отменить только в статусе NEW) — сервисный
  слой, доменные исключения → 409/422 через обработчик ошибок.

Валидация на контроллере не заменяет проверок инвариантов в домене —
она лишь отсекает мусор на границе.

## Как ответить на интервью

Коротко: JSON ↔ объект делает Jackson через `HttpMessageConverter` —
по соглашению об именах, с настройкой аннотациями (`@JsonProperty`,
`@JsonFormat`, `@JsonIgnore`); ошибка формата — 400 до контроллера,
отсутствующее поле — молча `null`, поэтому нужна валидация. Её даёт
Bean Validation: `@NotNull`/`@NotBlank`/`@Size`... на полях DTO + `@Valid`
на параметре (и на вложенных полях — иначе каскада нет); провал — 400
со списком ошибок. Разделение: формат — Jackson, поля — Bean Validation,
бизнес-правила — сервисный слой.
