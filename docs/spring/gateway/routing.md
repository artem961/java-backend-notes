# Маршрутизация

Маршрут (route) — базовая единица конфигурации шлюза: **куда** проксировать
запрос и **при каких условиях**. Здесь — из чего маршрут состоит и как
устроен выбор маршрута для запроса.

## Анатомия маршрута

Маршрут = **id + uri + предикаты + фильтры**:

```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: orders
          uri: http://orders-service:8080        # куда проксировать
          predicates:                             # когда маршрут подходит
            - Path=/api/orders/**
          filters:                                # что сделать с запросом
            - StripPrefix=1                       # /api/orders/1 -> /orders/1

        - id: legacy-users
          uri: http://legacy:8080
          predicates:
            - Path=/api/users/**
            - Method=GET
            - Header=X-Api-Version, v1            # только с этим заголовком
```

Обработка запроса: шлюз перебирает маршруты, находит **первый, чьи
предикаты совпали все**, применяет его фильтры и проксирует на `uri`.
Не совпал ни один — 404.

## Предикаты

Условия совпадения; комбинируются по «И»:

| Предикат | Условие |
|---|---|
| `Path` | шаблон пути (`/api/orders/**`) — основной |
| `Method` | HTTP-метод |
| `Host` | домен запроса (`api.example.com`) |
| `Header` / `Query` / `Cookie` | наличие/значение заголовка, параметра, куки |
| `Before` / `After` / `Between` | время (включение маршрута по расписанию) |
| `Weight` | весовое распределение между маршрутами |

`Weight` — практически интересный: две записи одной группы с весами 90/10
распределяют трафик — так делают **канареечные выкатки** на уровне шлюза:

```yaml
- id: orders-stable
  uri: http://orders-v1:8080
  predicates: [ "Path=/api/orders/**", "Weight=orders, 9" ]
- id: orders-canary
  uri: http://orders-v2:8080
  predicates: [ "Path=/api/orders/**", "Weight=orders, 1" ]
```

Порядок маршрутов важен там, где шаблоны пересекаются: более специфичные
пути должны проверяться раньше (управляется свойством `order`).

## Куда проксировать: uri и discovery

- Статический `uri` — прямой адрес (в Kubernetes — имя сервиса, балансировку
  даёт сам k8s).
- `lb://orders-service` — через **service discovery** (Eureka/Consul):
  шлюз получает список живых инстансов из реестра и балансирует сам
  (Spring Cloud LoadBalancer).
- Возможна автогенерация маршрутов из discovery: каждый зарегистрированный
  сервис доступен по `/{serviceId}/**` — удобно для старта, в проде обычно
  маршруты явные.

## Java DSL

Те же маршруты кодом — когда нужны условия/логика при построении:

```java
@Bean
RouteLocator routes(RouteLocatorBuilder rb) {
    return rb.routes()
        .route("orders", r -> r.path("/api/orders/**")
                               .filters(f -> f.stripPrefix(1))
                               .uri("http://orders-service:8080"))
        .build();
}
```

Yml-конфигурация предпочтительна: маршруты видны как конфигурация,
меняются без пересборки (с Config Server — и без рестарта).

## Как ответить на интервью

Коротко: маршрут — это uri назначения плюс набор предикатов (условий)
и фильтров; запрос получает первый маршрут, у которого совпали все
предикаты. Основной предикат — `Path`, есть `Method`/`Host`/`Header`
и `Weight` — весовое распределение для канареечных выкаток. Назначение —
прямой адрес, а с service discovery — `lb://service` с балансировкой
по живым инстансам. Конфигурируется в yml (или Java DSL), порядок важен
при пересекающихся шаблонах.
