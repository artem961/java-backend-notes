# Реактивность и эксплуатация

Spring Cloud Gateway построен на WebFlux и Netty — и это не деталь
реализации, а причина его эффективности и источник главных правил
эксплуатации.

## Почему шлюз реактивный

Работа шлюза — почти чистое **ожидание**: принять запрос, отправить
в сервис, ждать ответ, отдать клиенту. В классической модели
(Tomcat, thread per request) каждый висящий запрос занимал бы поток;
у шлюза «в полёте» одновременно тысячи запросов — тысячи потоков только
на ожидание.

Реактивная модель (WebFlux/Netty): **несколько event-loop-потоков**
(по числу ядер) обрабатывают события ввода-вывода; пока ответ от сервиса
не пришёл, поток не ждёт, а обслуживает другие запросы. Для прокси-нагрузки
это идеально: минимум потоков и памяти, высокая плотность соединений.

Следствие — **жёсткое правило: не блокировать event loop**. Блокирующий
вызов (JDBC, `RestTemplate`, `Thread.sleep`, тяжёлое вычисление) в фильтре
останавливает поток, обслуживающий сотни соединений, — деградирует весь
шлюз. Нужен блокирующий вызов — только через `Mono.fromCallable(...)
.subscribeOn(Schedulers.boundedElastic())` (отдельный пул), а лучше —
неблокирующие клиенты (`WebClient`, реактивный Redis).

Это же означает: в одном приложении Gateway и Spring MVC не живут —
стеки разные (Netty vs Tomcat, `ServerWebExchange` vs `HttpServletRequest`);
Security для шлюза — реактивные варианты конфигурации.

## Надёжность: таймауты и предохранители

Шлюз стоит перед всеми — его дисциплина таймаутов защищает всю систему:

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        connect-timeout: 1000        # мс на установку соединения
        response-timeout: 5s         # ожидание ответа сервиса
```

- **Таймауты обязательны и короче клиентских**: иначе зависший сервис
  копит соединения на шлюзе, и деградация одного сервиса кладёт периметр.
- **Circuit breaker + fallback** на маршрутах к нестабильным сервисам:
  быстрый отказ вместо очереди ожиданий.
- **Retry** — точечно и только идемпотентно; ретраи по всем маршрутам
  умножают нагрузку в инцидент (retry storm).
- Rate limiting на периметре — защита сервисов от всплесков и злоупотреблений.

## Наблюдаемость

Шлюз — лучшая точка обзора трафика:

- **Access-логи и метрики по маршрутам**: RPS, латентность, коды ответов
  каждого маршрута (`spring.cloud.gateway.metrics.enabled=true`,
  метрика `spring.cloud.gateway.requests` в Prometheus) — дашборд
  «здоровье всех API» в одном месте.
- **Трейсинг**: шлюз генерирует/пробрасывает `traceparent` — корреляция
  запроса через все сервисы начинается здесь; сюда же удобен request-id.
- Actuator-эндпоинт `/actuator/gateway/routes` — фактические маршруты
  работающего шлюза (первый инструмент при «куда реально ушёл запрос»).

## Эксплуатационные правила

- Шлюз **stateless** → горизонтальное масштабирование тривиально;
  общее состояние (rate limit счётчики) — в Redis.
- Несколько реплик за L4-балансировщиком/ingress — обязательны:
  единая точка отказа.
- Шлюз должен оставаться **тонким**: каждая добавленная логика — латентность
  и риск для всего трафика; тяжёлое — в сервисы.

## Как ответить на интервью

Коротко: шлюз реактивный (WebFlux/Netty), потому что его работа — ожидание
чужих ответов: event loop из нескольких потоков держит тысячи запросов
в полёте без потока на каждый. Главное правило — не блокировать event loop:
блокирующие вызовы только в отдельном scheduler'е, клиенты — неблокирующие.
Эксплуатация: обязательные таймауты (короче клиентских), circuit breaker
с fallback, осторожные ретраи, rate limiting; метрики и трейсинг
по маршрутам — центральная точка наблюдаемости; сам шлюз stateless,
масштабируется репликами и остаётся тонким.
