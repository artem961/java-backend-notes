# Скоупы

Скоуп бина отвечает на вопрос «**сколько экземпляров и как долго живут**».
На практике почти всё — синглтоны; остальные скоупы нужно знать, чтобы
понимать их ловушки.

## Какие бывают

| Скоуп | Экземпляров | Когда используется |
|---|---|---|
| `singleton` (дефолт) | один на контекст | всё обычное: сервисы, репозитории, контроллеры |
| `prototype` | новый на каждое обращение | редкo: stateful-объекты, «фабрика через контейнер» |
| `request` | один на HTTP-запрос | данные текущего запроса (текущий пользователь) |
| `session` | один на HTTP-сессию | состояние пользователя между запросами |
| `application` | один на ServletContext | практически не встречается |

```java
@Component
@Scope("prototype")
public class ReportBuilder { ... }
```

Веб-скоупы (`request`/`session`) существуют только внутри обработки
запроса — внедрить такой бин в синглтон напрямую нельзя без прокси
(об этом ниже).

## Главная ловушка: prototype внутри синглтона

Классический вопрос: prototype-бин внедрён в синглтон — сколько будет
экземпляров?

```java
@Service                                  // синглтон
public class ReportService {
    private final ReportBuilder builder;  // prototype

    public ReportService(ReportBuilder builder) { this.builder = builder; }
}
```

**Один.** Внедрение происходит один раз — при создании синглтона. Контейнер
создал один `ReportBuilder`, положил в поле, и дальше все вызовы работают
с ним. «Новый экземпляр на каждое обращение» относится к обращениям
**к контейнеру**, а поле синглтона к контейнеру больше не обращается.

Если действительно нужен новый экземпляр на каждый вызов:

```java
// 1. ObjectProvider — обращение к контейнеру в момент использования
private final ObjectProvider<ReportBuilder> builders;

public Report build() {
    ReportBuilder b = builders.getObject();   // каждый раз новый
    ...
}
```

Другие варианты: `@Lookup`-метод (Spring переопределит его в наследнике,
чтобы он ходил в контейнер) или scoped proxy. Но честный вопрос — а нужен ли
тут вообще бин: часто правильный ответ — обычный `new ReportBuilder()`,
если у объекта нет зависимостей из контейнера.

## Scoped proxy: как request-бин попадает в синглтон

Та же проблема с веб-скоупами решается прокси:

```java
@Component
@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)
public class CurrentUser { ... }
```

В синглтон внедряется не сам бин, а **прокси-заглушка**; при каждом
обращении к её методам прокси находит бин текущего запроса и делегирует ему.
Так синглтон-сервис безопасно пользуется request-скоупным `CurrentUser`.

## Скоупы и состояние

Связка с правилом stateless-бинов: скоуп — это ответ Spring на вопрос
«куда девать состояние». Состояние на время запроса — request-скоуп
(или просто параметры методов); состояние пользователя — session
(в современных stateless-API чаще токен + БД/Redis); всё остальное —
синглтоны без состояния. Если хочется `@Scope("prototype")` ради
изменяемых полей — чаще всего объект просто не должен быть бином.

## Как ответить на интервью

Коротко: скоуп определяет число и время жизни экземпляров. Дефолт —
`singleton` (один на контекст, поэтому бины stateless), `prototype` — новый
на каждое обращение к контейнеру, веб-скоупы `request`/`session` — на время
запроса/сессии. Ловушка: prototype, внедрённый в синглтон, создаётся
**один раз** — внедрение происходит при создании синглтона; нужен свежий
экземпляр — `ObjectProvider`, `@Lookup` или scoped proxy. Request-бины
внедряются в синглтоны через scoped proxy, который делегирует бину
текущего запроса.
