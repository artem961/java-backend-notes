# IoC и внедрение зависимостей

Spring в основе — это **IoC-контейнер**: фабрика, которая создаёт объекты
приложения, связывает их между собой и управляет их жизнью. Всё остальное
(MVC, Data, Security) построено поверх этой идеи.

## Проблема, которую решает IoC

Без контейнера объекты сами добывают свои зависимости:

```java
public class OrderService {
    private final OrderRepository repository = new JdbcOrderRepository(); // сам создал
}
```

Чем это плохо:

- **Жёсткая связанность**: `OrderService` прибит к конкретной реализации;
  заменить её (на другую БД, на мок в тесте) нельзя без правки кода.
- **Дублирование сборки**: каждый, кому нужен репозиторий, должен знать,
  как его построить (а тому нужен `DataSource`, а тому — конфигурация...).
- **Нетестируемость**: в юнит-тесте не подсунуть заглушку.

**Inversion of Control** — переворот ответственности: объект не создаёт
зависимости сам, а **получает их снаружи**. Кто-то внешний собирает граф
объектов. **Dependency Injection** — конкретный механизм этого переворота:
зависимости передаются через конструктор или сеттер.

```java
@Service
public class OrderService {
    private final OrderRepository repository;

    public OrderService(OrderRepository repository) {   // получил снаружи
        this.repository = repository;
    }
}
```

Теперь `OrderService` зависит от **интерфейса** и не знает, кто и как
создал реализацию. В тесте — `new OrderService(mock)`, в проде Spring
подставит реальный бин.

## Что делает контейнер

При старте Spring:

1. **Находит классы-компоненты** — сканированием пакетов на аннотации
   (`@Component`, `@Service`...) и по `@Bean`-методам конфигураций.
2. **Строит граф зависимостей**: смотрит на конструкторы и понимает,
   кто от кого зависит.
3. **Создаёт объекты в правильном порядке** — сначала зависимости, потом
   зависящих, — и внедряет их друг в друга.
4. Дальше **управляет их жизненным циклом**: синглтоны живут в контейнере
   до остановки приложения, на завершении вызываются destroy-колбэки.

Слово «инверсия» именно про это: не твой код вызывает фреймворк, чтобы
получить объект, а фреймворк вызывает твой код (конструкторы, колбэки),
собирая приложение.

## Способы внедрения

```java
// 1. Через конструктор — стандарт
public OrderService(OrderRepository repository) { ... }

// 2. Через сеттер — для необязательных зависимостей, редко
@Autowired
public void setClock(Clock clock) { ... }

// 3. В поле — компактно, но плохо
@Autowired
private OrderRepository repository;
```

**Стандарт — конструктор**, и на интервью полезно объяснить почему:

- поле может быть `final` — зависимость гарантированно есть и неизменна;
- объект **невозможно создать в невалидном состоянии** — все зависимости
  обязательны по сигнатуре;
- в тестах объект создаётся обычным `new` без всякого Spring;
- раздутый конструктор на 8 зависимостей **виден** и сигналит, что класс
  делает слишком много — полевое внедрение эту боль прячет.

Полевое внедрение (`@Autowired` на поле) живо в старом коде и тестах,
но считается плохой практикой: скрывает зависимости и требует рефлексии
или контейнера для создания объекта. `@Autowired` на единственном
конструкторе можно не писать — Spring найдёт его сам.

## Как ответить на интервью

Коротко: IoC — объекты не создают свои зависимости сами, их создаёт
и связывает контейнер; DI — механизм передачи зависимостей снаружи, через
конструктор. Это даёт слабую связанность (зависимость от интерфейса,
а не реализации), тестируемость (подставить мок) и единое место сборки
приложения. Spring на старте сканирует компоненты, строит граф зависимостей
и создаёт бины в нужном порядке. Внедрять стандартно через конструктор:
`final`-поля, валидность с рождения, тесты без контейнера; полевое
внедрение — антипаттерн.
