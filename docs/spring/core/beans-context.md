# Бины и контекст

**Бин** — объект, которым управляет Spring-контейнер. **ApplicationContext**
(контекст) — сам контейнер: реестр всех бинов с их определениями, фабрика,
которая их создаёт, и точка, откуда их можно получить.

## Откуда берутся бины

Два способа объявить бин:

**1. Аннотации-стереотипы + сканирование.** Spring сканирует пакеты
(начиная с пакета main-класса) и регистрирует классы, помеченные:

- `@Component` — базовая аннотация «это компонент»;
- `@Service` — семантика «сервисный слой» (технически то же самое);
- `@Repository` — слой доступа к данным + трансляция исключений БД
  в спринговую иерархию `DataAccessException`;
- `@Controller` / `@RestController` — веб-слой;
- `@Configuration` — класс с определениями бинов.

Разница между ними в основном **семантическая** — обозначить роль класса;
контейнер обрабатывает их одинаково (плюс упомянутые бонусы).

**2. `@Bean`-методы в конфигурации** — для объектов, чей класс не твой
(библиотечный) и на который аннотацию не повесить:

```java
@Configuration
public class AppConfig {
    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper().registerModule(new JavaTimeModule());
    }
}
```

Правило выбора простое: свой класс — стереотип, чужой — `@Bean`.

## Что такое контекст

`ApplicationContext` хранит не сами классы, а **bean definitions** —
описания «какой класс, какой скоуп, какие зависимости, как создать».
На старте контейнер по этим описаниям создаёт бины (синглтоны — сразу,
энергично) и складывает готовые экземпляры внутри себя.

Получить бин можно тремя путями: внедрением (нормальный путь),
`context.getBean(...)` (в прикладном коде — запах: сервис-локатор вместо DI)
и через события/колбэки.

Контекст — это ещё и:

- **источник конфигурации** — `Environment` с проперти и профилями;
- **шина событий** — `ApplicationEventPublisher`/`@EventListener`;
- **иерархия** — контексты могут иметь родителя (встречается в тестах
  и старых MVC-приложениях, знать достаточно факт существования).

## Синглтон по умолчанию и его следствие

По умолчанию каждый бин — **синглтон**: один экземпляр на контекст, который
делят все, кто его внедрил. Отсюда правило, которое проверяют на интервью:
**бины должны быть stateless** — без изменяемого состояния в полях.
Один и тот же `OrderService` одновременно выполняет запросы десятков
потоков; изменяемое поле в нём — классическая гонка. Всё изменяемое —
в локальные переменные, в БД или в явно потокобезопасные структуры.

## BeanFactory и ApplicationContext

Классический вопрос-примечание: `BeanFactory` — минимальный интерфейс
контейнера («дай бин по имени»), `ApplicationContext` расширяет его
событиями, `Environment`, интернационализацией и энергичной инициализацией
синглтонов. На практике всегда используется `ApplicationContext`.

## Как ответить на интервью

Коротко: бин — объект под управлением контейнера, контекст
(`ApplicationContext`) — сам контейнер: реестр определений бинов и фабрика
экземпляров. Бины объявляются стереотипами (`@Component`/`@Service`/
`@Repository` — разница семантическая) для своих классов и `@Bean`-методами
конфигураций — для библиотечных. По умолчанию бин — синглтон, создаётся
энергично на старте, поэтому бины держат stateless: один экземпляр
обслуживает все потоки. `getBean` в прикладном коде — запах, нормальный
путь — внедрение.
