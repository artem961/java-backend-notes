# AOP и прокси

AOP (аспектно-ориентированное программирование) — способ вынести сквозную
логику (транзакции, кэширование, метрики, безопасность) из бизнес-кода
в отдельное место и «навешивать» её декларативно. В Spring это фундамент,
на котором стоят `@Transactional`, `@Async`, `@Cacheable`, `@PreAuthorize` —
и все их знаменитые ловушки.

## Идея

Без AOP сквозная логика дублируется в каждом методе: открой транзакцию,
выполни, закоммить, при ошибке откати; замерь время; проверь права...
AOP выносит это в **аспект** — класс с логикой «вокруг» методов,
а места применения описываются декларативно.

Терминология (нужен минимум): **аспект** — класс со сквозной логикой;
**advice** — сама логика (что делать до/после/вокруг вызова);
**pointcut** — выражение, к каким методам применять.

```java
@Aspect
@Component
public class TimingAspect {

    @Around("@annotation(Timed)")                     // pointcut: методы с @Timed
    public Object time(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.nanoTime();
        try {
            return pjp.proceed();                     // вызов реального метода
        } finally {
            long ms = (System.nanoTime() - start) / 1_000_000;
            log.info("{} took {} ms", pjp.getSignature(), ms);
        }
    }
}
```

## Как это работает: прокси

Spring реализует AOP через **динамические прокси**. При создании бина
`BeanPostProcessor` видит, что к бину применимы аспекты (или аннотации вроде
`@Transactional`), и кладёт в контекст **не сам объект, а обёртку**:

```mermaid
flowchart LR
    C["Вызывающий код"] --> P["Прокси:<br/>логика аспекта<br/>(открыть транзакцию...)"]
    P --> T["Реальный объект:<br/>твой метод"]
    T --> P2["Прокси: завершение<br/>(commit/rollback)"]
```

Все, кто внедряет этот бин, получают прокси и вызывают методы через него —
поэтому логика аспекта срабатывает. Два вида прокси: JDK dynamic proxy
(реализует интерфейсы бина) и CGLIB (наследник класса — дефолт в Boot).

## Ограничения прокси — главное на интервью

Из механизма «обёртка вокруг объекта» следуют все классические грабли:

- **Самовызов (self-invocation)**: метод `a()` вызывает `b()` того же
  класса напрямую через `this` — **мимо прокси**. Аннотации на `b()`
  (`@Transactional`, `@Async`, `@Cacheable`) не сработают. Лечение —
  вынести `b()` в отдельный бин (правильно) или внедрить себя же/
  `ObjectProvider` (костыль).
- **`private`/`final`/`static` методы** не перехватываются: CGLIB
  не может переопределить их в наследнике.
- **Вызов не через контейнер**: объект, созданный `new`, — не бин,
  прокси у него нет, аннотации мертвы.

Эти три пункта — ответ на добрую половину вопросов вида «аннотация есть,
а эффекта нет».

## Когда писать свой аспект

Реальные кандидаты: тайминги/метрики вокруг группы методов, аудит вызовов,
ретраи, проверка кастомных аннотаций. Альтернативы по границам: HTTP-граница —
фильтр/интерсептор; один-два метода — обычная композиция без магии. Аспекты
дают невидимую в call-стеке логику, поэтому злоупотребление ими затрудняет
чтение кода — используются точечно.

## Как ответить на интервью

Коротко: AOP выносит сквозную логику (транзакции, кэш, метрики) в аспекты,
применяемые к методам декларативно. Spring реализует это прокси: контейнер
подменяет бин обёрткой (JDK proxy по интерфейсам или CGLIB-наследник),
и логика выполняется при вызове через прокси. На этом стоят
`@Transactional`, `@Async`, `@Cacheable` — и отсюда их ограничения:
самовызов через `this` идёт мимо прокси, `private`/`final`/`static`
не перехватываются, объекты через `new` не проксируются. «Аннотация
не работает» — почти всегда одна из этих трёх причин.
