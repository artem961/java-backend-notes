# Транзакции в Spring

Транзакция гарантирует: группа операций с БД выполнится **вся или никак**.
В Spring управление транзакциями декларативное — `@Transactional` на методе,
остальное делает инфраструктура. Здесь — что именно она делает.

## Что делает @Transactional

```java
@Service
public class TransferService {

    @Transactional
    public void transfer(long fromId, long toId, BigDecimal amount) {
        accountRepository.withdraw(fromId, amount);   // одна БД-операция
        accountRepository.deposit(toId, amount);      // вторая
        // выход без исключения -> commit; исключение -> rollback
    }
}
```

Под капотом — AOP-прокси:

1. Вызов метода проходит через прокси; тот просит
   `PlatformTransactionManager` **начать транзакцию**: взять соединение
   из пула, выключить autocommit, привязать соединение к текущему потоку
   (через `ThreadLocal`).
2. Все обращения к БД внутри метода (репозитории, JPA) получают **то же
   соединение** — потому и попадают в одну транзакцию.
3. Метод завершился нормально — **commit**; вылетело исключение —
   **rollback**. Соединение возвращается в пул.

Привязка к потоку — ключевая деталь: «текущая транзакция» — это соединение
в `ThreadLocal` текущего потока. Отсюда следствие: код, выполняющийся
**в другом потоке** (`@Async`, свой executor, параллельный стрим),
в транзакцию вызывающего **не попадает**.

## Правила отката

По умолчанию откат происходит только на **unchecked**-исключениях
(`RuntimeException`, `Error`). Checked-исключение — commit!
Историческое решение, о котором нужно знать:

```java
@Transactional(rollbackFor = Exception.class)   // откатывать и на checked
public void process() throws IOException { ... }
```

Вторая ловушка — **поймать и не пробросить**: если исключение перехвачено
внутри метода, прокси его не увидит и закоммитит. А если исключение вылетело
из вложенного транзакционного вызова, транзакция уже помечена
rollback-only — поймав его снаружи, получишь
`UnexpectedRollbackException` на коммите.

## Границы транзакции: где ставить

`@Transactional` ставится на **сервисный метод** — единицу бизнес-операции:

- Не на контроллер: держать транзакцию на время сериализации и сети — лишнее.
- Не на каждый репозиторный вызов: тогда каждая операция — своя транзакция,
  и атомарности между ними нет.
- Метод должен быть **коротким**: транзакция держит соединение из пула
  и блокировки в БД. HTTP-вызовы, Kafka, долгие вычисления внутри
  транзакции — типовая причина исчерпания пула под нагрузкой.

`@Transactional(readOnly = true)` — для чтения: подсказка Hibernate
(не отслеживать изменения — экономия памяти и CPU) и драйверу.
Идиома Spring Data: `readOnly = true` на классе сервиса, обычный
`@Transactional` — на изменяющих методах.

## Программные транзакции

Когда границы не совпадают с методом (транзакция на кусок метода, в цикле) —
`TransactionTemplate`:

```java
transactionTemplate.executeWithoutResult(tx -> {
    // атомарный кусок
});
```

Тот же менеджер транзакций, но границы — явные. Нужен нечасто, но снимает
вопрос «а как без аннотации».

## Как ответить на интервью

Коротко: `@Transactional` через AOP-прокси начинает транзакцию перед методом
(соединение из пула, autocommit off, привязка к потоку через ThreadLocal),
и коммитит на нормальном выходе или откатывает на исключении — по умолчанию
только на unchecked; для checked нужен `rollbackFor`. Все БД-вызовы внутри
работают через одно соединение — это и есть «одна транзакция», поэтому
другой поток в неё не попадает. Ставится на сервисный метод — короткую
бизнес-операцию; долгие внешние вызовы внутри транзакции истощают пул.
Для чтения — `readOnly = true`, для явных границ — `TransactionTemplate`.
