# Распространение и изоляция

Два параметра `@Transactional`, вокруг которых строятся уточняющие вопросы:
**propagation** — что делать, если транзакция уже идёт, и **isolation** —
насколько транзакции видят друг друга.

## Propagation: транзакционный метод вызывает транзакционный

Когда метод с `@Transactional` вызывает (через прокси) другой метод
с `@Transactional`, поведение определяет режим распространения:

| Режим | Поведение |
|---|---|
| `REQUIRED` (дефолт) | транзакция есть — присоединиться; нет — начать новую |
| `REQUIRES_NEW` | всегда **новая** транзакция; текущая приостанавливается |
| `SUPPORTS` | есть — присоединиться, нет — выполниться без транзакции |
| `MANDATORY` | транзакция обязана быть, иначе исключение |
| `NEVER` | транзакции быть не должно, иначе исключение |
| `NESTED` | savepoint внутри текущей: можно откатить только вложенную часть |

На практике нужны два: **`REQUIRED`** — дефолт, «всё в одной транзакции»,
и **`REQUIRES_NEW`** — независимая транзакция, которая закоммитится, даже
если внешняя откатится. Классический кейс `REQUIRES_NEW` — **аудит**:
запись «попытка операции» должна остаться в БД, даже если сама операция
откатилась.

```java
@Transactional
public void process(Order order) {
    auditService.logAttempt(order);   // REQUIRES_NEW: закоммитится сам
    riskyOperation(order);            // откат не тронет запись аудита
}
```

Важно понимать цену `REQUIRES_NEW`: это **второе соединение из пула**
на время вложенной транзакции — злоупотребление быстро исчерпывает пул.
И помнить про rollback-only: исключение из вложенного `REQUIRED`-вызова
помечает **общую** транзакцию на откат — поймать его и «продолжить» не выйдет.

## Isolation: что транзакции видят друг у друга

Уровень изоляции — компромисс между корректностью и конкурентностью.
Аномалии, которые уровни по очереди запрещают:

- **dirty read** — чтение незакоммиченных чужих изменений;
- **non-repeatable read** — повторное чтение строки даёт другой результат;
- **phantom read** — повторный запрос по условию возвращает новые строки.

| Уровень | Запрещает |
|---|---|
| `READ_UNCOMMITTED` | ничего |
| `READ_COMMITTED` | dirty read |
| `REPEATABLE_READ` | + non-repeatable read |
| `SERIALIZABLE` | + phantom read (полная изоляция) |

`@Transactional(isolation = ...)` просто передаёт уровень в БД —
**реальное поведение определяет база**. Дефолт `DEFAULT` — уровень БД:
у PostgreSQL это `READ_COMMITTED`. Подробный разбор уровней, MVCC
и блокировок — в разделе про БД; со стороны Spring важно знать, что
параметр есть, и что менять его стоит осознанно и редко — обычно проблемы
конкурентности решаются не поднятием изоляции, а явными блокировками
(`SELECT FOR UPDATE`, оптимистичные версии).

## Таймаут

Третий параметр той же группы: `@Transactional(timeout = 5)` — транзакция,
не уложившаяся в лимит, откатывается. Защита от «зависших» транзакций,
которые держат соединение и блокировки.

## Как ответить на интервью

Коротко: propagation определяет судьбу вызова внутри существующей
транзакции: дефолтный `REQUIRED` присоединяется, `REQUIRES_NEW` открывает
независимую (кейс — аудит, который должен пережить откат; цена — второе
соединение из пула). Isolation — уровень изоляции, передаваемый в БД:
`READ_COMMITTED` (дефолт PostgreSQL) убирает грязное чтение,
`REPEATABLE_READ` — неповторяющееся, `SERIALIZABLE` — фантомы; поднимать
уровень — редкое решение, чаще правильны явные блокировки или оптимистичная
версия. Плюс `timeout` против зависших транзакций.
