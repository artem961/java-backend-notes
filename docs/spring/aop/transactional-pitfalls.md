# Где @Transactional ломается

Сводка случаев, когда `@Transactional` стоит, а транзакции нет (или она
не такая, как ожидалось). Это, возможно, самый популярный вопрос по Spring
на интервью — потому что каждый случай проверяет понимание механизма,
а не заучивание.

## Карта отказов

Почти все случаи — следствия двух фактов: транзакция работает через
**AOP-прокси** и привязана к **потоку**.

### 1. Самовызов — лидер по частоте

```java
@Service
public class OrderService {
    public void processAll(List<Order> orders) {
        orders.forEach(this::processOne);      // вызов через this — мимо прокси!
    }

    @Transactional
    public void processOne(Order order) { ... } // транзакции НЕТ
}
```

Вызов `this.processOne()` — прямой вызов метода объекта, прокси в нём
не участвует. Лечение: вынести `processOne` в отдельный бин — заодно
это обычно улучшает дизайн.

### 2. Метод не перехватывается

`private`, `final`, `static` — CGLIB не может переопределить такой метод
в наследнике-прокси. Аннотация молча не работает (IDE и статический анализ
обычно предупреждают). `@Transactional` — только на public-методах бинов.

### 3. Объект — не бин

Создан через `new`, не прошёл через контейнер — прокси нет, аннотации
мертвы.

### 4. Checked-исключение — commit вместо rollback

По умолчанию откат только на `RuntimeException`/`Error`. Метод бросил
`IOException` — транзакция **закоммитится**. Нужен
`rollbackFor = Exception.class`.

### 5. Исключение поймано внутри

```java
@Transactional
public void process() {
    try {
        riskyDbWork();
    } catch (Exception e) {
        log.warn("ignored", e);    // прокси исключения не увидит -> commit
    }
}
```

Обратная сторона: если исключение вылетело из **вложенного**
транзакционного вызова, транзакция уже помечена rollback-only — и ловля
исключения снаружи даст `UnexpectedRollbackException` при коммите.

### 6. Другой поток

```java
@Transactional
public void process() {
    executor.submit(() -> repository.save(...));   // другой поток — другая
}                                                  // (точнее, никакая) транзакция
```

Транзакция — соединение в `ThreadLocal` потока. `@Async`-метод, свой
executor, `CompletableFuture`, параллельный стрим — всё это выполняется
вне транзакции вызывающего. Кстати, у `@Async` тот же прокси-механизм —
и та же ловушка самовызова.

### 7. Транзакция «не видит» изменений — слишком долгая

Не отказ, но родственная проблема: долгая транзакция с HTTP-вызовом внутри
держит соединение из пула и блокировки; под нагрузкой пул кончается,
и сервис встаёт. Внешние вызовы — **вне** транзакции.

## Как диагностировать

- Лог `TransactionInterceptor` (`logging.level.org.springframework.transaction.interceptor=TRACE`) —
  видно, где транзакции открываются и закрываются.
- `TransactionSynchronizationManager.isActualTransactionActive()` —
  проверить в отладке, есть ли транзакция прямо сейчас.
- Простой тест: бросить исключение в конце метода и посмотреть, откатились
  ли изменения.

## Как ответить на интервью

Коротко: `@Transactional` ломается там, где ломается его механика — прокси
и привязка к потоку. Чек-лист: самовызов через `this` (мимо прокси);
`private`/`final`/`static` метод; объект не бин (`new`); checked-исключение
без `rollbackFor` (по умолчанию коммит!); исключение поймано внутри метода;
работа ушла в другой поток (`@Async`, executor). Плюс дисциплина: транзакции
короткие, внешние вызовы — за их пределами, иначе пул соединений исчерпается.
