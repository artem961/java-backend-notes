# Пагинация и сортировка

Ни один список в реальном API не отдаётся целиком — «выведи все заказы»
на миллионе строк уронит и БД, и сервис. Spring Data делает пагинацию
первоклассной: параметром запроса и типом результата.

## Pageable и Page

```java
public interface OrderRepository extends JpaRepository<Order, Long> {
    Page<Order> findByStatus(OrderStatus status, Pageable pageable);
}

// использование
Page<Order> page = repository.findByStatus(
    OrderStatus.NEW,
    PageRequest.of(0, 20, Sort.by("createdAt").descending()));

page.getContent();        // сами элементы
page.getTotalElements();  // всего строк (отдельный count-запрос!)
page.getTotalPages();
```

- **`Pageable`** — параметр «какая страница»: номер, размер, сортировка.
  Добавляется к любому методу репозитория.
- **`Page<T>`** — результат с метаданными. Для него Spring выполняет
  **два запроса**: выборку страницы и `count(*)` всего результата.
- **`Slice<T>`** — то же без count: известно лишь «есть ли следующая
  страница» (запрашивается size+1 элементов). Для бесконечной ленты,
  где точное число страниц не нужно, — дешевле.
- `List<T>` с `Pageable` — просто срез без метаданных.

В контроллере `Pageable` собирается из query-параметров автоматически:
`GET /orders?page=2&size=20&sort=createdAt,desc` → готовый объект
в аргументе метода. Обязательная практика — **ограничить максимальный
size** (защита от `?size=1000000`).

## Как это работает в SQL

`PageRequest.of(2, 20)` превращается в `LIMIT 20 OFFSET 40` (+ `ORDER BY`).
Отсюда два фундаментальных свойства offset-пагинации:

- **Глубокие страницы дороги**: `OFFSET 100000` заставляет БД прочитать
  и отбросить сто тысяч строк. Для UI-пагинации на десятки страниц — ок,
  для выгрузок и бесконечных лент — нет.
- **Страницы плывут**: вставили строку между запросами страниц — элементы
  сдвинулись, что-то покажется дважды или пропадёт.

Альтернатива — **keyset-пагинация** («после элемента X»):
`WHERE created_at < :lastSeen ORDER BY created_at DESC LIMIT 20` —
работает по индексу одинаково быстро на любой глубине и не плывёт.
Цена — нельзя прыгнуть на произвольную страницу. Правило: UI со страницами —
offset, лента/выгрузка/API-курсор — keyset.

## Сортировка

`Sort` — часть `Pageable` или самостоятельный параметр. Важные детали:

- **Пагинация без явной сортировки — баг**: без `ORDER BY` порядок строк
  в SQL не гарантирован, страницы могут пересекаться. Всегда задавать
  сортировку, и лучше с **уникальным хвостом** (`createdAt, id`) — иначе
  равные значения между страницами тасуются.
- Сортировка из query-параметров — по **полям сущности**; поля, по которым
  сортируют, должны быть покрыты индексом, иначе `ORDER BY` + `LIMIT`
  на большой таблице — сортировка всего результата.
- Белый список сортируемых полей на границе API — хорошая практика:
  не давать клиенту сортировать по любому полю.

## Наружу — не Page

`Page` в JSON сериализуется с кучей внутренних полей и нестабилен между
версиями (Spring сам предупреждает). Наружу — свой DTO страницы:

```java
public record PageDto<T>(List<T> items, long total, int page, int size) {
    public static <T> PageDto<T> from(Page<T> p) {
        return new PageDto<>(p.getContent(), p.getTotalElements(),
                             p.getNumber(), p.getSize());
    }
}
```

## Как ответить на интервью

Коротко: пагинация — `Pageable` (страница, размер, сортировка; собирается
из query-параметров) и `Page` в ответе — но `Page` делает дополнительный
count-запрос; `Slice` — без него, для лент. В SQL это `LIMIT/OFFSET`:
глубокий offset дорог и страницы плывут — альтернатива keyset-пагинация
(«после элемента X», по индексу). Сортировка обязательна (без `ORDER BY`
порядок не гарантирован) и с уникальным хвостом; размер страницы
ограничивать, наружу отдавать свой DTO, а не `Page`.
