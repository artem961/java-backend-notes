# Модификация данных

Изменять данные через JPA можно двумя принципиально разными путями:
через **сущности** (загрузил — изменил — dirty checking сохранит) и
**массовыми запросами** (`@Modifying` — UPDATE/DELETE сразу в БД).
Понимание разницы — источник и ответов, и багов.

## Путь 1: через сущность

```java
@Transactional
public void cancel(long orderId) {
    Order order = repository.findById(orderId)
        .orElseThrow(() -> new OrderNotFoundException(orderId));
    order.setStatus(CANCELLED);          // просто меняем объект
    // никакого save: dirty checking сам сделает UPDATE при коммите
}
```

Внутри транзакции загруженная сущность **managed**: Hibernate отслеживает
изменения и на коммите генерирует UPDATE по фактически изменённым полям.
Явный `save` для managed-сущности избыточен (не вреден — он просто
вернёт тот же объект).

Этот путь — по умолчанию: работают проверки бизнес-логики, колбэки,
оптимистичная блокировка, кэш. Минус — на N строк это N загрузок
и N UPDATE'ов.

## Путь 2: массовые запросы — @Modifying

```java
@Modifying(clearAutomatically = true)
@Query("update Order o set o.status = 'EXPIRED' where o.status = 'NEW' and o.createdAt < :deadline")
int expireStale(@Param("deadline") Instant deadline);
```

- `@Query` с UPDATE/DELETE требует **`@Modifying`** (без него — ошибка:
  метод ждёт результат-выборку). Возвращает число затронутых строк.
  Нужна активная транзакция.
- Запрос идёт **напрямую в БД, мимо persistence context**: уже загруженные
  сущности **не узнают** об изменении и останутся устаревшими. Отсюда
  `clearAutomatically = true` — очистить контекст после запроса
  (а `flushAutomatically` — протолкнуть отложенные изменения до него).
- Колбэки сущностей, `@Version`-инкременты, каскады — **не срабатывают**:
  это голый SQL-эффект.

Когда использовать: массовые операции над многими строками —
«просрочить все старые заказы», «удалить логи старше месяца». На порядок
быстрее цикла по сущностям, но требует понимания последствий.

## Удаление

- `deleteById`/`delete(entity)` — через сущность: загрузка + каскады + колбэки.
- `deleteAllInBatch` / `@Modifying`-DELETE — одним запросом, мимо контекста
  и каскадов JPA (каскады на уровне БД `ON DELETE CASCADE` — сработают).
- Практика продовых систем — **soft delete** (флаг `deleted_at` вместо
  физического удаления): история цела, ссылочная целостность не страдает;
  реализуется вручную или `@SQLDelete`/`@Where` в Hibernate.

## Конкурентные обновления

Два обновления одной строки «одновременно» — вторая перезапишет первую
(lost update). Инструменты:

- **Оптимистичная блокировка** — поле `@Version`: UPDATE проверяет версию,
  при несовпадении — `OptimisticLockException` → повторить или отдать 409.
  Дефолт для редких конфликтов.
- **Пессимистичная** — `@Lock(PESSIMISTIC_WRITE)` на методе репозитория
  (SQL `SELECT ... FOR UPDATE`): строка блокируется до конца транзакции.
  Для горячих строк типа счетов/остатков.
- **Атомарный UPDATE** — `set balance = balance - :amount where balance >= :amount`:
  проверка и изменение одним запросом, без чтения в приложение.

Подробный разбор блокировок — в разделе про БД; здесь важно знать
JPA-обёртки над ними.

## Как ответить на интервью

Коротко: обычная модификация — загрузить сущность в транзакции и изменить:
dirty checking сам сделает UPDATE по изменённым полям, `save` для managed
не нужен. Массовые изменения — `@Modifying` + UPDATE/DELETE-запрос: одним
SQL, но мимо persistence context (загруженные сущности устаревают — 
`clearAutomatically`), без колбэков и версий. Конкурентные обновления:
`@Version` (оптимистично, 409 при конфликте), `@Lock(PESSIMISTIC_WRITE)`
(`FOR UPDATE`) или атомарный UPDATE с условием. Продовое удаление — чаще
soft delete.
