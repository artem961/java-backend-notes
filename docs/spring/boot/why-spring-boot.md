# Зачем нужен Spring Boot

Spring Boot — не замена Spring, а **надстройка над ним**, убирающая
конфигурационную рутину. Spring Framework даёт механизмы (DI, MVC,
транзакции), но до Boot их нужно было вручную собирать в приложение.

## Проблема «чистого» Spring

Приложение на Spring до Boot требовало явно сконфигурировать всё:
`DispatcherServlet` и его маппинг, `DataSource` с пулом, `EntityManagerFactory`,
менеджер транзакций, конвертеры JSON, — десятки бинов boilerplate-конфигурации,
одинаковой от проекта к проекту. Плюс войну версий: подобрать совместимые
версии Spring, Hibernate, Jackson и десятков библиотек. Плюс внешний Tomcat,
в который деплоится WAR.

## Что даёт Boot

Три кита:

1. **Стартеры** — готовые наборы зависимостей под задачу.
   `spring-boot-starter-web` = MVC + Jackson + встроенный Tomcat;
   `spring-boot-starter-data-jpa` = Spring Data + Hibernate + пул HikariCP.
   Версии всех транзитивных зависимостей согласованы в BOM —
   война версий решена: указываешь версию Boot, остальное подобрано.

2. **Автоконфигурация** — Boot смотрит, что лежит в classpath, и сам
   создаёт нужные бины: увидел драйвер PostgreSQL и JPA — сконфигурировал
   `DataSource`, `EntityManagerFactory` и транзакции; увидел web-стартер —
   поднял Tomcat и `DispatcherServlet`. Своей конфигурацией любой кусок
   можно переопределить (подробно — в теме про автоконфигурацию).

3. **Встроенный сервер и исполняемый jar** — приложение содержит Tomcat
   внутри и запускается `java -jar`. Это перевернуло деплой: вместо
   «WAR в сервер приложений» — самодостаточный артефакт, который одинаково
   запускается локально, в CI и в Docker-контейнере.

Плюс продакшен-обвязка из коробки: **Actuator** — готовые эндпоинты
здоровья (`/actuator/health`), метрик и информации о приложении,
на которые опираются Kubernetes-пробы и Prometheus.

## Минимальное приложение

```java
@SpringBootApplication
public class App {
    public static void main(String[] args) {
        SpringApplication.run(App.class, args);
    }
}
```

`@SpringBootApplication` — композиция трёх аннотаций:

- `@ComponentScan` — сканировать компоненты от текущего пакета вниз
  (поэтому main-класс кладут в корневой пакет проекта);
- `@EnableAutoConfiguration` — включить автоконфигурацию;
- `@Configuration` — сам класс может объявлять `@Bean`.

## Boot и Spring: как отвечать на «в чём разница»

Частый вопрос-ловушка. Точный ответ: Spring Framework — сами механизмы
(контейнер, MVC, транзакции), Spring Boot — слой поверх: стартеры
с согласованными версиями, автоконфигурация, встроенный сервер, Actuator.
Boot не добавляет новой функциональности уровня «транзакции» — он добавляет
**сборку по умолчанию**, которую без него писали руками. Convention over
configuration: разумные дефолты везде, любое отклонение — явной настройкой.

## Как ответить на интервью

Коротко: Boot — надстройка над Spring, решающая проблему конфигурационной
рутины. Стартеры дают согласованные наборы зависимостей (без войны версий),
автоконфигурация создаёт стандартные бины по содержимому classpath,
встроенный Tomcat превращает приложение в самодостаточный `java -jar`
артефакт — идеально под Docker, Actuator даёт health/метрики из коробки.
`@SpringBootApplication` = `@ComponentScan` + `@EnableAutoConfiguration` +
`@Configuration`. Разница со Spring: фреймворк — механизмы, Boot — их
автоматическая сборка с разумными дефолтами.
