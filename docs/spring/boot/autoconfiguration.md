# Автоконфигурация

Автоконфигурация — механизм, которым Boot «догадывается», какие бины нужны
приложению. Понимание, как она устроена, отделяет «пользуюсь Boot»
от «понимаю Boot» — и помогает, когда магия ломается.

## Как это работает

Автоконфигурация — это обычные `@Configuration`-классы, поставляемые
с Boot (и стартерами), но применяемые **по условиям**:

```java
@AutoConfiguration
@ConditionalOnClass(DataSource.class)          // в classpath есть JDBC
@ConditionalOnMissingBean(DataSource.class)    // и пользователь не создал свой
public class DataSourceAutoConfiguration {
    @Bean
    DataSource dataSource(DataSourceProperties props) { ... }
}
```

Механика по шагам:

1. Boot читает **список автоконфигурационных классов** из файла
   `AutoConfiguration.imports` в каждом jar (свой список есть у каждого
   стартера).
2. Для каждого класса проверяются **условия** `@Conditional...`:
   - `@ConditionalOnClass` — класс есть в classpath (главный триггер:
     добавил зависимость — включилась конфигурация);
   - `@ConditionalOnMissingBean` — пользователь **не объявил** такой бин сам;
   - `@ConditionalOnProperty` — включено проперти.
3. Прошедшие условия конфигурации создают бины — **после** пользовательских
   определений.

`@ConditionalOnMissingBean` — сердце принципа «дефолты, которые можно
переопределить»: объяви свой `ObjectMapper` — и автоконфигурационный
просто не создастся. Не нужно ничего выключать — достаточно предоставить своё.

## Следствия, которые проверяют вопросами

**«Добавил зависимость — поведение изменилось».** Зависимость в classpath —
это триггер: добавил `spring-boot-starter-security` — все эндпоинты
закрылись формой логина; добавил драйвер и JPA — Boot требует настройки
`DataSource` и не стартует без URL базы. Зависимости в Boot — не пассивные
библиотеки, а включатели поведения.

**Отключение куска автоконфигурации**: точечно —
`@SpringBootApplication(exclude = DataSourceAutoConfiguration.class)`
или проперти `spring.autoconfigure.exclude`. Но чаще правильнее
переопределить бин, а не выключать конфигурацию целиком.

## Как посмотреть, что сконфигурировалось

Когда «магия» повела себя неожиданно, у Boot есть штатная диагностика:

- **`--debug`** (или `debug=true` в проперти) — при старте печатается
  **conditions evaluation report**: какие автоконфигурации применились
  (positive matches), какие нет и **почему** (negative matches — «не было
  класса X», «бин уже определён»). Первый инструмент при вопросе «почему
  не создался/создался бин».
- **Actuator `/actuator/conditions`** — тот же отчёт у работающего
  приложения; `/actuator/beans` — все бины контекста.

## Свой стартер — чтобы понимать устройство

Внутренний шаринг кода между сервисами устроен так же: библиотека +
автоконфигурационный класс + `AutoConfiguration.imports`. Подключил
зависимость — получил настроенные бины. Писать своё на интервью не попросят,
но знание, что стартер = «зависимости + условная конфигурация», показывает
понимание механизма.

## Как ответить на интервью

Коротко: автоконфигурация — набор готовых `@Configuration`-классов
с условиями: `@ConditionalOnClass` включает конфигурацию по наличию
библиотеки в classpath, `@ConditionalOnMissingBean` отступает перед
пользовательским бином — поэтому дефолт переопределяется простым
объявлением своего бина. Список конфигураций Boot читает из
`AutoConfiguration.imports` каждого стартера. Диагностика — запуск
с `--debug` (отчёт, какие условия совпали и почему) или
`/actuator/conditions`.
