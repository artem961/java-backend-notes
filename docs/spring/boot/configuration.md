# Конфигурация приложения

Всё, что отличает окружения — адреса БД, таймауты, фичефлаги, — выносится
из кода во внешнюю конфигурацию. У Boot для этого стройная система:
проперти, профили и типизированные настройки.

## application.yml и источники конфигурации

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/app
    username: app

app:
  orders:
    page-size: 20
    sync-enabled: true
```

Boot собирает конфигурацию из множества источников с **приоритетом**;
упрощённо, от слабого к сильному:

1. `application.yml` внутри jar — дефолты;
2. `application-{profile}.yml` — профильные переопределения;
3. **переменные окружения** — `SPRING_DATASOURCE_URL` (автоматическое
   преобразование имени: точки → подчёркивания, верхний регистр);
4. аргументы командной строки `--spring.datasource.url=...`.

Практический смысл: в jar — разумные дефолты, а окружение (Docker,
Kubernetes) переопределяет своё через env-переменные, не пересобирая
артефакт. Секреты (пароли, ключи) в репозиторий не кладут никогда —
только через окружение или секрет-хранилище.

## Профили

Профиль — именованный набор конфигурации и бинов под окружение:
`dev`, `test`, `prod`.

- `application-prod.yml` активируется при `spring.profiles.active=prod`
  (обычно через переменную окружения).
- `@Profile("prod")` на бине/конфигурации — бин существует только
  в этом профиле (реальный SMS-шлюз в prod, заглушка в dev).

Антипаттерн — логика на `if (profile == "prod")` в коде: окружения должны
отличаться **конфигурацией**, а не ветками кода.

## Доступ к значениям: @Value и @ConfigurationProperties

```java
// точечно — @Value
@Value("${app.orders.page-size}")
private int pageSize;

// группой — типизированный класс
@ConfigurationProperties(prefix = "app.orders")
public record OrdersProperties(int pageSize, boolean syncEnabled) { }
```

`@ConfigurationProperties` — предпочтительный способ для связанных настроек:

- **типизация и структура**: настройки группы — один объект, а не россыпь
  строковых ключей по коду;
- **валидация на старте**: `@Validated` + `@Min`/`@NotBlank` — кривая
  конфигурация валит запуск, а не всплывает на проде;
- удобное внедрение: класс — обычный бин, тестируется без контекста;
- IDE-подсказки в yml (через annotation processor).

`@Value` оставляют для одиночных значений. Важно: оба читаются **при
создании бина** — изменение файла на диске работающее приложение не увидит.

## Типовой расклад по окружениям

- **Локально**: `application.yml` с дефолтами под docker-compose
  (localhost-адреса), профиль `dev`.
- **Тесты**: `application-test.yml`, Testcontainers переопределяет адреса
  динамически.
- **Прод**: тот же jar; адреса, лимиты и секреты — env-переменными
  из Kubernetes (ConfigMap/Secret).

Один артефакт проходит все окружения — меняется только конфигурация. Это
принцип, который стоит проговорить на интервью.

## Как ответить на интервью

Коротко: конфигурация собирается из источников по приоритету —
`application.yml` в jar (дефолты) < профильные файлы < переменные окружения
< аргументы запуска; поэтому один артефакт едет через все окружения,
а различия задаёт среда. Профили (`spring.profiles.active`) включают
профильные файлы и `@Profile`-бины. Доступ — `@Value` для одиночных
значений и `@ConfigurationProperties` для групп: типизированный класс
с валидацией на старте. Секреты — только через окружение, не в репозитории.
