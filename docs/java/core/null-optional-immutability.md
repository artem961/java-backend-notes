# `null`, `Optional` и неизменяемость

Три темы об одном: как писать код, в котором меньше сюрпризов. `null` —
главный источник ошибок в Java, `Optional` — способ честно сказать «значения
может не быть», неизменяемость — способ гарантировать, что объект не изменится
у вас за спиной.

## Проблема null

`null` — «ссылка ни на что». Любая операция над ним — `NullPointerException`,
самое частое исключение в Java-приложениях. Коварство в том, что `null`
возникает в одном месте, а взрывается в другом: метод вернул `null`, его
передали дальше, положили в поле — и NPE случился через три вызова от настоящей
причины.

Со времён Java 14 сообщение NPE стало полезным — оно говорит, *что именно*
было `null`: `Cannot invoke "String.length()" because "user.name" is null`.

Как защищаться:

- **Проверять на границах** — данные снаружи (параметры API, ответы чужих
  сервисов, результаты поиска) проверяются сразу при получении, а не в глубине
  логики. Fail fast: `Objects.requireNonNull(user, "user")` в начале метода.
- **Не возвращать `null` из своих методов.** Для «не нашли» есть `Optional`,
  для пустого результата — пустая коллекция (`List.of()`), но не `null`.
- **Не передавать `null` как аргумент** — перегрузка или отдельный метод
  лучше «волшебного» null-параметра.

## Optional

`Optional<T>` — контейнер: либо значение, либо пусто. Его смысл — **сделать
отсутствие значения частью сигнатуры**. `User findUser(id)` молчит о том,
что юзера может не быть; `Optional<User> findUser(id)` заставляет вызывающего
обработать этот случай — забыть не даст компилятор.

```java
// создание
Optional.of(value);         // value точно не null (иначе NPE прямо здесь)
Optional.ofNullable(value); // value может быть null -> будет empty
Optional.empty();

// использование — цепочкой, без if
String city = findUser(id)
        .map(User::getAddress)
        .map(Address::getCity)
        .orElse("не указан");

findUser(id).ifPresent(user -> sendEmail(user));

User user = findUser(id)
        .orElseThrow(() -> new UserNotFoundException(id));
```

Разница `orElse` и `orElseGet`: аргумент `orElse(создатьДефолт())` вычисляется
**всегда**, даже когда значение есть; `orElseGet(() -> создатьДефолт())` —
только при пустом Optional. Для дорогих дефолтов — только `orElseGet`.

### Как Optional использовать не надо

- **`if (opt.isPresent()) opt.get()`** — тот же null-check, только длиннее.
  Для этого есть `map`/`ifPresent`/`orElse`.
- **Поля класса и параметры методов.** `Optional` спроектирован как
  **возвращаемый тип**. В полях он не сериализуется нормально и добавляет
  обёртку без пользы; параметр `Optional` заставляет вызывающего писать
  `Optional.ofNullable(...)` вместо простой перегрузки.
- **`opt.get()` без проверки** — кидает `NoSuchElementException`; это тот же
  отказ думать об отсутствии значения, от которого Optional должен защищать.
- **Возвращать `null` вместо `Optional`** — худшее из возможного: вызывающий
  доверяет типу и получает NPE на самом Optional. Метод с возвращаемым типом
  `Optional` не должен возвращать `null` никогда.

## Неизменяемость

Неизменяемый (immutable) объект после создания не меняется. Это снимает целый
класс проблем: объект можно свободно отдавать куда угодно и класть в ключи
хеш-коллекций, его состояние не «уедет» из другого места кода, и он
**потокобезопасен без всякой синхронизации** — потому что гонки бывают только
вокруг изменяемого состояния.

Рецепт неизменяемого класса:

```java
public final class Money {                    // final: нельзя добавить мутабельного наследника
    private final long amount;                // все поля private final
    private final String currency;
    private final List<String> tags;

    public Money(long amount, String currency, List<String> tags) {
        this.amount = amount;
        this.currency = currency;
        this.tags = List.copyOf(tags);        // защитная копия на входе
    }

    public List<String> getTags() {
        return tags;                          // List.copyOf уже неизменяем;
    }                                         // для мутабельных полей — копия и на выходе

    public Money add(long delta) {
        return new Money(amount + delta, currency, tags); // «изменение» = новый объект
    }
}
```

!!! warning "final ≠ неизменяемость"
    `final` на поле фиксирует **ссылку**, а не объект: `final List<String> tags`
    спокойно позволяет `tags.add(...)`. Класс с final-полями, хранящими
    мутабельные объекты (списки, массивы, `Date`), — не immutable: тот, кто
    передал список в конструктор или получил его из геттера, может менять
    «неизменяемый» объект снаружи. Отсюда защитные копии на входе и выходе.

Так работают все «образцовые» неизменяемые классы: `String`, обёртки,
`BigDecimal`, `LocalDate`. И `record` — современный способ получить
неизменяемый класс одной строкой (но копии мутабельных компонентов record
за вас не сделает — это по-прежнему ваша забота).

### Неизменяемые коллекции

- `List.of(...)`, `Map.of(...)`, `Set.of(...)` — по-настоящему неизменяемые
  коллекции; `add`/`put` кидают `UnsupportedOperationException`.
- `List.copyOf(list)` — неизменяемая копия существующей коллекции.
- `Collections.unmodifiableList(list)` — **не копия, а обёртка-вид**: через неё
  писать нельзя, но если менять исходный `list`, изменения видны сквозь обёртку.

## Как ответить на интервью

Коротко: `null` опасен тем, что взрывается далеко от места возникновения,
поэтому — проверки на границах, fail fast и «не возвращать null». `Optional` —
явный контракт «может отсутствовать» для возвращаемых значений; работать с ним
цепочками `map`/`orElse`, не тащить в поля и параметры. Неизменяемость: final
поля + защитные копии мутабельного + новые объекты вместо изменения; даёт
безопасный шаринг, надёжные ключи и потокобезопасность бесплатно. `final` на
ссылке сам по себе неизменяемости не даёт.
