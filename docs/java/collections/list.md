# List

`List` — упорядоченная последовательность с доступом по индексу и дубликатами.
Двух реализаций достаточно, чтобы закрыть тему: `ArrayList` (используется
практически всегда) и `LinkedList` (нужно знать, почему почти никогда).

## ArrayList: массив с запасом

Внутри — обычный массив. Когда место кончается, создаётся новый массив
**в полтора раза больше**, и элементы копируются в него.

Отсюда вся его арифметика:

- `get(i)` / `set(i, x)` — **O(1)**: прямое обращение к ячейке массива.
- `add(x)` в конец — **амортизированное O(1)**: обычно просто запись в ячейку,
  изредка — дорогое расширение, но в среднем по всем вставкам получается
  константа.
- `add(i, x)` / `remove(i)` в середину — **O(n)**: весь «хвост» массива
  сдвигается на одну позицию.
- `contains(x)` / `indexOf(x)` — **O(n)**: линейный перебор с `equals`.
  Если проверки на вхождение частые — это сигнал, что нужен `Set`.

Практическая деталь: если размер известен заранее, задавайте ёмкость —
`new ArrayList<>(10_000)` — чтобы избежать серии промежуточных расширений
с копированием.

## LinkedList: почему он почти не нужен

Внутри — двусвязный список: каждый элемент обёрнут в узел со ссылками
на соседей. В теории вставка/удаление — O(1), но есть два «но»:

- O(1) — только **если вы уже стоите на месте вставки** (через итератор).
  Добраться до позиции `i` — это O(n) прыжков по ссылкам: `get(i)` у
  `LinkedList` линейный.
- На практике `ArrayList` быстрее почти во всех сценариях, включая вставки:
  узлы `LinkedList` разбросаны по куче, и процессорный кэш работает с ними
  плохо, плюс каждый элемент несёт накладные расходы двух ссылок и заголовка
  объекта.

Честный ответ на интервью: «по умолчанию всегда `ArrayList`; `LinkedList`
теоретически выигрывает на частых вставках/удалениях через итератор в середине
больших списков, но на практике я таких случаев не встречал».

| Операция | ArrayList | LinkedList |
|---|---|---|
| `get(i)` | O(1) | O(n) |
| `add` в конец | O(1)* | O(1) |
| `add`/`remove` в середине | O(n) — сдвиг | O(n) — поиск позиции |
| Память на элемент | ячейка массива | узел: элемент + 2 ссылки |

*амортизированное

## Ловушки API

!!! warning "`Arrays.asList` — не обычный список"
    `Arrays.asList(1, 2, 3)` возвращает обёртку **фиксированного размера**
    над массивом: `set` работает, а `add`/`remove` кидают
    `UnsupportedOperationException`. `List.of(...)` ещё строже — неизменяемый
    полностью. Нужен обычный список — оборачивайте:
    `new ArrayList<>(List.of(1, 2, 3))`.

!!! warning "`remove(int)` против `remove(Object)`"
    У `List<Integer>` две перегрузки конфликтуют:

    ```java
    list.remove(5);            // remove(int) — удалит ПЯТЫЙ ЭЛЕМЕНТ
    list.remove(Integer.valueOf(5)); // remove(Object) — удалит значение 5
    ```

    Классический тихий баг при работе со списками чисел.

- Удаление элементов во время `for-each` — `ConcurrentModificationException`;
  правильные способы — `removeIf(predicate)` или явный итератор
  (разбирается в теме про итерирование).
- `subList` возвращает **вид** на исходный список, а не копию: изменения
  видны в обе стороны, а структурное изменение исходного списка ломает subList.

## Как ответить на интервью

Коротко: `ArrayList` — массив с полуторным ростом: индексный доступ O(1),
вставка в конец амортизированная O(1), в середину O(n) из-за сдвига;
`contains` линейный — частые проверки вхождения означают, что нужен `Set`.
`LinkedList` — двусвязный список, теоретически хорош на вставках через
итератор, но из-за поиска позиции за O(n) и недружелюбности к кэшу процессора
на практике всегда проигрывает — выбор по умолчанию `ArrayList`. Из ловушек:
`Arrays.asList` фиксированного размера, `remove(int)` vs `remove(Object)`,
удаление в цикле — только через `removeIf` или итератор.
