# Лямбды и функциональный стиль

Лямбда — способ передать **поведение** как значение: не «объект с методом»,
а сам метод. До Java 8 для этого писали анонимные классы; лямбда делает то же
самое в одну строку.

```java
// анонимный класс — как было
button.onClick(new Handler() {
    @Override
    public void handle(Event e) { log(e); }
});

// лямбда — как стало
button.onClick(e -> log(e));

// ссылка на метод — когда лямбда только вызывает готовый метод
button.onClick(this::log);
```

## Функциональные интерфейсы

Лямбда сама по себе не имеет типа — её тип определяет **функциональный
интерфейс**: интерфейс с ровно одним абстрактным методом. Лямбда — это его
компактная реализация:

```java
@FunctionalInterface
public interface Validator {
    boolean isValid(Order order);   // единственный абстрактный метод
}

Validator v = order -> order.getTotal() > 0;
```

`@FunctionalInterface` не обязателен, но полезен: компилятор не даст случайно
добавить второй абстрактный метод и сломать все лямбды.

Свои интерфейсы нужны редко — в `java.util.function` уже есть стандартный набор:

| Интерфейс | Сигнатура | Смысл | Где встречается |
|---|---|---|---|
| `Function<T, R>` | `T -> R` | преобразование | `stream.map`, `Optional.map` |
| `Predicate<T>` | `T -> boolean` | проверка | `stream.filter`, `removeIf` |
| `Consumer<T>` | `T -> void` | действие | `forEach`, `ifPresent` |
| `Supplier<T>` | `() -> T` | отложенное создание | `orElseGet`, `Optional.orElseThrow` |
| `UnaryOperator<T>` | `T -> T` | преобразование в тот же тип | `list.replaceAll` |
| `BiFunction<T, U, R>` | `(T, U) -> R` | два аргумента | `map.merge`, `map.compute` |

Умение читать эти сигнатуры — практический навык: увидев параметр
`Supplier<X>`, вы знаете, что значение будет вычислено **лениво** и, возможно,
не будет вычислено вообще.

## Ссылки на методы

Четыре формы, все — просто сокращение лямбды, которая ничего не делает,
кроме вызова:

```java
String::valueOf        // статический метод:      x -> String.valueOf(x)
user::getName          // метод конкретного объекта: () -> user.getName()
User::getName          // метод экземпляра по первому аргументу: u -> u.getName()
ArrayList::new         // конструктор:            () -> new ArrayList<>()
```

Правило стиля: ссылка на метод — когда она читается лучше лямбды
(`map(User::getName)`), лямбда — когда внутри есть хоть какая-то логика.

## Захват переменных

Лямбда «видит» переменные окружающего метода, но захватывать может только
**effectively final** — те, что не переприсваиваются:

```java
int limit = config.getLimit();
orders.removeIf(o -> o.getTotal() > limit); // ок: limit не меняется

int count = 0;
orders.forEach(o -> count++); // ошибка компиляции: count изменяется
```

Причина: лямбда может выполниться позже и в другом потоке — захват
изменяемой локальной переменной означал бы гонки и чтение «из прошлого».
Это ограничение — намеренный дизайн: счётчики и накопление — задача
`reduce`/`collect` в стримах, а не побочных эффектов из лямбд.

Отличия от анонимного класса, которые стоит знать: лямбда не создаёт отдельный
класс-файл (компилируется через `invokedynamic`), и `this` внутри лямбды —
это внешний объект, а не сама лямбда.

## Функциональный стиль на практике

Лямбды поменяли повседневный вид Java-кода далеко за пределами стримов:

```java
// коллекции
users.removeIf(u -> u.isInactive());
map.computeIfAbsent(key, k -> new ArrayList<>());

// Optional
findUser(id).map(User::getEmail).ifPresent(mailer::send);

// отложенные вычисления
value.orElseGet(() -> expensiveDefault());

// Spring: колбэки и конфигурация
transactionTemplate.execute(status -> repository.save(order));
http.authorizeHttpRequests(auth -> auth.anyRequest().authenticated());
```

Общая идея одна: методы принимают «что сделать» параметром, и код из
«как обходить и проверять» превращается в «что мы хотим получить».

## Как ответить на интервью

Коротко: лямбда — компактная реализация функционального интерфейса
(один абстрактный метод); стандартный набор — `Function`, `Predicate`,
`Consumer`, `Supplier`. Ссылки на методы — сокращение для лямбд-делегатов.
Захватывать можно только effectively final переменные — защита от гонок
и отложенного выполнения. `Supplier` в сигнатуре означает ленивость —
разница `orElse`/`orElseGet` именно в этом.
