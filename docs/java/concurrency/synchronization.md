# Синхронизация и блокировки

Инструменты для работы с общим изменяемым состоянием — от `synchronized`
до атомиков. Смысл у всех один: сделать критическую секцию **атомарной**
и **видимой** другим потокам; отличаются они гибкостью и ценой.

## synchronized

У каждого объекта Java есть **монитор** — встроенная блокировка. `synchronized`
захватывает её на входе и отпускает на выходе (в том числе при исключении);
пока монитор занят, остальные потоки ждут в состоянии `BLOCKED`.

```java
public class Counter {
    private int count = 0;
    private final Object lock = new Object();

    public synchronized void increment() { count++; }  // монитор this

    public void incrementExplicit() {
        synchronized (lock) {                          // монитор отдельного объекта
            count++;
        }
    }
}
```

Что важно понимать:

- `synchronized`-метод синхронизируется на **`this`**, статический — на
  **`Класс.class`**. Это два разных монитора: статический и обычный
  synchronized-методы друг другу не мешают.
- Взаимное исключение работает только между потоками, использующими
  **один и тот же монитор**. Синхронизация «каждый на своём объекте» —
  не синхронизация вовсе.
- Монитор **реентерабелен**: поток, уже держащий его, может войти повторно
  (иначе рекурсия и вызов одного synchronized-метода из другого зависали бы).
- Хорошая практика — приватный `final`-объект-замок вместо `this`: на `this`
  может синхронизироваться и внешний код, устраивая неожиданные конфликты.

!!! warning "Почему «поставил synchronized, а гонка осталась»"
    Типовые причины: синхронизированы **не все** места доступа к полю (запись
    под замком, чтение без); потоки синхронизируются на **разных мониторах**
    (разные объекты, или смесь статических и нестатических методов); атомарны
    отдельные операции, но не их **композиция** — `if (map.containsKey(...))`
    и `map.put(...)` синхронизированы порознь, а вклиниться можно между ними.

## volatile

`volatile` решает ровно одну проблему — **видимость**: запись в volatile-поле
немедленно видна всем потокам, чтение всегда возвращает последнюю запись
(плюс запрещает переупорядочивать операции вокруг).

```java
private volatile boolean stopped;      // флаг остановки — идеальный случай

void run() { while (!stopped) doWork(); }
void stop() { stopped = true; }        // теперь рабочий поток увидит
```

Чего volatile **не** даёт — атомарности: `volatile int count` не чинит
`count++`, это по-прежнему три операции. Область применения узкая: флаги
и статусы, которые **один** поток пишет, а другие читают. Для
read-modify-write нужны атомики или блокировки.

## Атомики и CAS

`AtomicInteger`, `AtomicLong`, `AtomicReference` делают read-modify-write
атомарным **без блокировок** — через процессорную инструкцию CAS
(compare-and-swap): «запиши новое значение, только если там всё ещё старое»;
не вышло — прочитай заново и повтори.

```java
private final AtomicInteger count = new AtomicInteger();

count.incrementAndGet();                      // атомарный ++
count.compareAndSet(expected, newValue);      // CAS напрямую
count.updateAndGet(x -> x * 2);               // атомарное преобразование
```

Потоки не засыпают и не ждут — при конфликте просто повторяется попытка.
Для счётчиков и одиночных значений это быстрее и проще блокировок; счётчик
из вопроса «почему не 2000» чинится заменой `int` на `AtomicInteger`.
Под очень высокой конкуренцией за один счётчик есть `LongAdder` — он
раскладывает инкременты по ячейкам и суммирует при чтении.

Граница возможностей: атомик защищает **одно** значение. Инвариант из
нескольких полей (баланс + история операций) атомиками не защитить —
нужна блокировка.

## ReentrantLock

Явная блокировка из `java.util.concurrent.locks` — тот же смысл, что
у `synchronized`, но в виде объекта с API:

```java
private final ReentrantLock lock = new ReentrantLock();

public void transfer() {
    lock.lock();
    try {
        // критическая секция
    } finally {
        lock.unlock();   // ОБЯЗАТЕЛЬНО в finally — иначе замок навсегда
    }
}
```

Что умеет сверх `synchronized`:

- **`tryLock()`** / `tryLock(timeout)` — попытаться и не зависнуть: основа
  стратегий избегания deadlock.
- **`lockInterruptibly()`** — ожидание, которое можно прервать `interrupt()`.
- **Честный режим** `new ReentrantLock(true)` — очередь FIFO, защита от
  starvation (ценой пропускной способности).
- **Условия** `newCondition()` — несколько «комнат ожидания» на один замок
  (await/signal — структурированная замена wait/notify).

Правило выбора простое: по умолчанию — `synchronized` (короче, нельзя забыть
отпустить); `ReentrantLock` — когда нужны его фичи: tryLock, таймауты,
прерываемость, честность.

### ReadWriteLock

`ReentrantReadWriteLock` разделяет доступ: **много читателей одновременно**,
писатель — эксклюзивно. Выгоден при частом чтении и редкой записи
(конфигурация, справочники в памяти). Если запись частая — выгоды нет,
проще одиночный замок или concurrent-коллекция.

## wait / notify

Низкоуровневая координация на мониторе: `wait()` отпускает монитор и усыпляет
поток, `notify()/notifyAll()` будят ждущих. Вызываются только внутри
`synchronized` по тому же объекту, ждать нужно **в цикле** (`while`, не `if` —
из-за ложных пробуждений).

Практический статус: знать, что это фундамент, на котором построены высокоуровневые
средства, но в прикладном коде не использовать — очереди (`BlockingQueue`),
`CountDownLatch`, `Condition` и пулы решают те же задачи без ручной возни.

## Как ответить на интервью

Коротко: `synchronized` захватывает монитор объекта и даёт атомарность +
видимость; работает, только если все потоки используют один монитор и под
замком вся составная операция. `volatile` — только видимость, для флагов;
`count++` он не чинит. Атомики — неблокирующий CAS для одиночных значений,
`AtomicInteger.incrementAndGet` — правильный счётчик. `ReentrantLock` — тот же
монитор, но с `tryLock`, таймаутами и честностью; `unlock` строго в `finally`.
Выбор: по умолчанию synchronized, атомики для счётчиков, ReentrantLock — когда
нужны его возможности.
