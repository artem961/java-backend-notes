# Потокобезопасные коллекции

Обычные коллекции в многопоточном коде ломаются: гонки при изменении, порча
структуры, `ConcurrentModificationException` при обходе. Пакет
`java.util.concurrent` даёт коллекции, спроектированные под конкурентный
доступ, — и они почти всегда лучше «синхронизированных обёрток».

## Почему не Collections.synchronizedMap

Старый способ — обернуть коллекцию: каждый метод становится `synchronized`
на одном общем мониторе. Проблемы:

- **Один замок на всё** — потоки выстраиваются в очередь даже на чтение;
  под нагрузкой это узкое место.
- **Составные операции не атомарны**: «проверил — положил» — это два вызова,
  между ними вклинивается другой поток. Синхронность отдельных методов
  не складывается в синхронность сценария.
- **Итерирование всё равно вручную**: обход нужно оборачивать в
  `synchronized (map) { ... }`, иначе CME.

Вывод: обёртки — легаси-ответ; правильный ответ — коллекции, спроектированные
под конкурентность.

## ConcurrentHashMap

Главная конкурентная структура — потокобезопасный `HashMap` **без глобальной
блокировки**:

- **Чтения вообще без блокировок** — `get` никого не ждёт никогда.
- Записи: вставка в пустой бакет — атомарным CAS; при коллизии —
  `synchronized` на **первом узле конкретного бакета**. Потоки конфликтуют,
  только если бьются в один и тот же бакет; карта из тысяч бакетов
  практически не создаёт конкуренции.
  (До Java 8 использовались сегменты — 16 замков на карту; это уже история,
  но о ней иногда спрашивают.)

Атомарные составные операции — то, чего не хватает обёрткам:

```java
map.putIfAbsent(key, value);                       // «положи, если нет» — атомарно
map.computeIfAbsent(userId, id -> new ArrayList<>()) // ленивое создание значения
   .add(order);
map.merge(word, 1, Integer::sum);                  // атомарный счётчик частот
```

Особенности, о которых спрашивают:

- **`null` запрещён** и для ключей, и для значений: в конкурентной карте
  `get() == null` не отличить от «другой поток как раз удаляет», поэтому
  неоднозначность убрали на уровне API.
- Итераторы **слабо согласованы** (weakly consistent): CME не кидают, обходят
  снимок «примерно текущего» состояния — параллельные изменения могут попасть
  или не попасть в обход. Для конкурентной структуры это норма: «строго
  согласованного снапшота» без остановки мира не существует.
- `size()` — оценка на движущейся цели; строить логику на точном размере
  конкурентной карты — ошибка дизайна.

## CopyOnWriteArrayList

Список, у которого **каждая запись копирует весь внутренний массив**,
а чтения работают по неизменяемому снимку без всяких блокировок.

- Итератор получает снимок на момент создания: никакого CME, но и изменений,
  сделанных после начала обхода, он не увидит.
- Цена — O(n) память и время на **каждую** запись.

Ниша: **читают постоянно, пишут почти никогда** — списки слушателей,
подписчиков, снапшоты конфигурации. Для частых записей — категорически
не подходит.

## BlockingQueue: производитель — потребитель

Блокирующая очередь — стандартный канал передачи работы между потоками:

```java
BlockingQueue<Task> queue = new ArrayBlockingQueue<>(1000);

// производитель
queue.put(task);        // блокируется, если очередь ПОЛНА

// потребитель
Task task = queue.take(); // блокируется, если очередь ПУСТА
```

`put`/`take` сами решают всю координацию — никаких wait/notify руками.
Ограниченная ёмкость — это **backpressure**: быстрый производитель
притормаживается, вместо того чтобы утопить потребителя (и память).

Реализации: `ArrayBlockingQueue` — ограниченная, на массиве;
`LinkedBlockingQueue` — по умолчанию неограниченная (осторожно: «очередь
никогда не переполняется» означает «память когда-нибудь закончится»);
`SynchronousQueue` — без ёмкости, передача из рук в руки (внутри cached-пула).
Плюс неблокирующая `ConcurrentLinkedQueue` — когда ждать не нужно.

На этой же идее стоят очереди задач в `ThreadPoolExecutor` — пул это
и есть производитель-потребитель.

## Правила выбора

| Ситуация | Выбор |
|---|---|
| Словарь с конкурентным доступом | `ConcurrentHashMap` |
| Кэш/счётчики | `ConcurrentHashMap` + `computeIfAbsent`/`merge` |
| Часто читаемый, редко меняемый список | `CopyOnWriteArrayList` |
| Передача задач между потоками | `BlockingQueue` |
| Коллекция живёт в одном потоке / под общим замком | обычные коллекции |

И общий приём нулевой стоимости: **неизменяемые коллекции** (`List.of`,
`Map.copyOf`) потокобезопасны сами по себе — если данные можно не менять,
а пересоздавать, проблема конкурентности исчезает целиком.

## Как ответить на интервью

Коротко: синхронизированные обёртки — один замок на всё и неатомарные
составные операции, поэтому вместо них — конкурентные коллекции.
`ConcurrentHashMap`: чтения без блокировок, записи через CAS и точечный
`synchronized` на бакет, атомарные `putIfAbsent`/`computeIfAbsent`/`merge`,
null запрещён, итераторы слабо согласованы. `CopyOnWriteArrayList` — снимок
на каждую запись, для «читают много, пишут редко». `BlockingQueue` — канал
производитель-потребитель с backpressure через ограниченную ёмкость.
А лучший потокобезопасный вариант — неизменяемость.
