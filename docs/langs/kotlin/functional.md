# Функциональные возможности

Функциональный стиль в Kotlin — не отдельный API, как Stream в Java,
а часть языка: лямбды компактнее, функции — полноценные значения,
а операции над коллекциями встроены в стандартную библиотеку.

## Лямбды и функциональные типы

```kotlin
val square: (Int) -> Int = { x -> x * x }
val isPositive: (Int) -> Boolean = { it > 0 }   // it — единственный параметр
```

Отличия от Java:

- **Функциональные типы встроены в язык**: `(Int) -> Boolean` вместо подбора
  интерфейса (`Predicate`, `Function`, ...). Любая функция — значение.
- **`it`** — неявное имя единственного параметра, убирает шум в коротких лямбдах.
- **Лямбда за скобками**: если лямбда — последний аргумент, её выносят
  за скобки вызова: `list.filter { it > 0 }`. Отсюда «DSL-ный» вид Kotlin-кода.
- **Захват переменных без ограничений**: в отличие от Java, лямбда может
  читать и **изменять** локальные переменные — требования effectively final нет.

## Коллекции: операции без Stream

В Java для `filter`/`map` нужно открыть стрим и собрать результат коллектором.
В Kotlin это методы прямо на коллекциях:

```kotlin
val names = users
    .filter { it.active }
    .sortedBy { it.createdAt }
    .map { it.name }
    .take(10)
```

Важное отличие от стримов: эти операции **энергичные** — каждая сразу
возвращает новый список. Для длинных конвейеров и больших коллекций есть
ленивый аналог — `asSequence()`, работающий как Stream (элементы проходят
конвейер по одному, без промежуточных списков):

```kotlin
val result = hugeList.asSequence()
    .filter { it.valid }
    .map { it.toDto() }
    .first()                     // ленивость + короткое замыкание
```

Практическое правило: обычный код — операции на коллекциях (проще читать,
дешевле на маленьких данных), большие данные или ранний выход — `asSequence()`.

Богатство стандартной библиотеки — то, что в Java собирается коллекторами
вручную: `groupBy`, `associateBy` (список → map по ключу), `partition`,
`zip`, `chunked`, `flatMap`, `sumOf`, `distinctBy`.

## Scope-функции

Пять маленьких функций (`let`, `apply`, `also`, `run`, `with`), выполняющих
блок в контексте объекта. Реально нужны две идиомы:

```kotlin
// let + ?. — выполнить блок, только если не null
user?.let { sendEmail(it.email) }

// apply — настроить объект и вернуть его же
val props = Properties().apply {
    setProperty("host", "localhost")
    setProperty("port", "5432")
}
```

Различия между всеми пятью — как объект доступен внутри (`it` или `this`)
и что возвращается (объект или результат блока). Злоупотребление
scope-функциями делает код хуже, чем их отсутствие, — цепочки из
`let.run.also` читать невозможно.

## Функции высшего порядка и inline

Функция может принимать и возвращать функции. Стандартная библиотека
помечает такие функции `inline` — компилятор подставляет тело лямбды
в место вызова, и абстракция становится бесплатной (без создания объекта
лямбды на каждый вызов). Поэтому `filter`/`map`/`forEach` в Kotlin
не дороже цикла.

## Как ответить на интервью

Коротко: лямбды и функциональные типы встроены в язык — `(Int) -> Boolean`
вместо функциональных интерфейсов, `it` для единственного параметра, лямбда
за скобками. Операции `filter`/`map`/`groupBy` живут прямо на коллекциях
и выполняются энергично; ленивый конвейер как Stream — `asSequence()`.
Scope-функции — идиомы `?.let` для null и `apply` для настройки объекта.
Стандартные функции высшего порядка — `inline`, поэтому накладных расходов
на лямбды нет.
