# Null safety

Главная «фишка» Kotlin: возможность `null` встроена в **систему типов**.
В Java любая ссылка может оказаться `null`, и компилятор об этом молчит —
отсюда вечные NPE. В Kotlin тип честно говорит, может ли там быть `null`,
и компилятор не даст обратиться к nullable-значению без проверки.

## Nullable и non-null типы

```kotlin
var name: String = "Артём"
name = null                 // ошибка компиляции: String не допускает null

var nick: String? = null    // String? — «String или null», это другой тип
nick.length                 // ошибка компиляции: сначала проверь на null
```

`String` и `String?` — разные типы. Всё, что может быть `null`, обязано
объявляться со знаком `?`, и компилятор заставляет обработать этот `null`
до использования. NPE из «забыл проверить» превращается из ошибки времени
выполнения в ошибку компиляции.

## Операторы работы с null

```kotlin
val len = nick?.length            // safe call: null, если nick == null
val len2 = nick?.length ?: 0      // элвис: значение по умолчанию вместо null
val city = user?.address?.city    // цепочка: null, если оборвалась на любом шаге

val len3 = nick!!.length          // "уверен, что не null" — если null, будет NPE
```

- **`?.`** (safe call) — вызвать, если не `null`, иначе вернуть `null`.
- **`?:`** (элвис) — «если слева `null`, взять правое». Обычная пара к safe
  call: `user?.name ?: "гость"`.
- **`!!`** — принудительное разыменование. Единственный оператор, который
  явно может бросить NPE; в нормальном коде его быть не должно — каждое `!!`
  это место, где программист спорит с компилятором.

**Smart cast**: после проверки компилятор сам сужает тип — внутри
`if (nick != null)` переменная уже `String`, а не `String?`, никаких
дополнительных приведений не нужно.

## Откуда NPE всё-таки прилетает

Null safety защищает только внутри мира Kotlin. Дыры — на границах:

- **Java-код.** Типы, пришедшие из Java, — **платформенные** (`String!`):
  компилятор не знает, может ли там быть `null`, и не проверяет. Java-метод
  вернул `null` — Kotlin упадёт с NPE как обычная Java. Помогают аннотации
  `@Nullable`/`@NotNull` в Java-коде — Kotlin их читает и превращает
  платформенный тип в честный nullable/non-null.
- **Оператор `!!`** — сам попросил.
- **Отложенная инициализация**: `lateinit var` до первого присваивания
  бросит исключение при обращении.
- **Десериализация и рефлексия** — Jackson или JPA могут записать `null`
  в non-null поле в обход компилятора, проверок в рантайме по умолчанию нет.

Это стандартный ответ на вопрос «в Kotlin же null safety, откуда NPE?» —
из интеропа с Java, `!!` и рефлексии.

## Сравнение с Java

В Java ту же задачу решают соглашениями: `Optional` для возвращаемых
значений, аннотации `@Nullable`, дисциплина проверок. Разница в том, что
в Java это **опционально** и держится на аккуратности команды, а в Kotlin —
**обязательно** и проверяется компилятором. `Optional` в Kotlin не нужен:
`String?` выражает то же самое без обёртки и аллокаций.

## Как ответить на интервью

Коротко: в Kotlin nullability — часть типа: `String` не может быть `null`,
`String?` может, и компилятор требует обработать `null` до обращения.
Инструменты — safe call `?.`, элвис `?:`, smart cast после проверки;
`!!` — аварийный люк, которого в хорошем коде нет. NPE остаётся возможным
на границах: платформенные типы из Java-кода без аннотаций, `!!`, `lateinit`,
десериализация через рефлексию. По сути Kotlin делает обязательным то, что
в Java держится на `Optional` и договорённостях.
