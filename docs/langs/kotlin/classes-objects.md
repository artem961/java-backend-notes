# Классы и объекты

Классы в Kotlin решают ту же задачу, что в Java, но с меньшим количеством
церемоний: свойства вместо полей с геттерами, первичный конструктор в одной
строке с объявлением класса, data-классы вместо ручных `equals`/`hashCode`.

## Свойства и конструкторы

```kotlin
class User(
    val id: Long,            // параметр конструктора сразу становится свойством
    var name: String,
    val email: String? = null
)

val user = User(1, "Артём")
user.name = "Артем"          // выглядит как поле, работает через сеттер
```

- **Свойство** = поле + геттер (+ сеттер для `var`) в одном объявлении.
  Снаружи обращение выглядит как к полю, но идёт через аксессоры — можно
  в любой момент добавить кастомный `get()`/`set()`, не ломая вызывающий код.
- **Первичный конструктор** пишется в заголовке класса; `val`/`var` перед
  параметром превращает его в свойство. 15 строк Java-бойлерплейта — в 3.

## data class

Класс-носитель данных: компилятор сам генерирует `equals`, `hashCode`,
`toString`, `copy` и деструктуризацию по свойствам первичного конструктора:

```kotlin
data class Point(val x: Int, val y: Int)

val p = Point(1, 2)
val moved = p.copy(x = 5)      // новый объект с изменённым полем
val (x, y) = p                 // деструктуризация
```

Это прямой аналог `record` из Java, но появившийся раньше и гибче: свойства
могут быть `var`, класс может иметь тело, поля вне конструктора.
Идиома та же, что с рекордами: **неизменяемый носитель данных + `copy`**
вместо мутации.

!!! warning "data class и JPA-сущности — плохая пара"
    JPA-сущность data-классом делать не стоит, и это популярный вопрос:

    - JPA требует **конструктор без аргументов** — у data-класса его нет
      (решается плагином `kotlin-jpa`, который его генерирует).
    - Сгенерированный `equals`/`hashCode` считается **по всем полям**, включая
      `id`, который присваивается при сохранении, — объект «меняет хэш» после
      `persist`, ломая контракт для `HashSet` и сравнение до/после сохранения.
    - `equals` по всем полям трогает все свойства — при ленивых связях это
      случайная инициализация lazy-коллекций.
    - Все классы Kotlin по умолчанию `final`, а Hibernate нужны прокси
      (наследники) — решается плагином `kotlin-allopen`/`kotlin-spring`.

    Идиома: сущности — обычные классы с ручным `equals` по `id`,
    а data-классы — для DTO, где они и хороши.

## object и companion object

В Kotlin нет `static`. Вместо него:

```kotlin
object AppConfig {                  // синглтон на уровне языка
    val version = "1.0"
}

class User(val id: Long) {
    companion object {              // "статическая" часть класса
        fun fromRow(row: Row): User = User(row.getLong("id"))
    }
}

val u = User.fromRow(row)           // вызов как статического метода
```

- **`object`** — объявление класса и его единственного экземпляра сразу:
  синглтон без ручной реализации паттерна.
- **`companion object`** — объект-компаньон внутри класса; его члены
  вызываются через имя класса, как статические. Обычное место для фабричных
  методов и констант.

## Наследование: final по умолчанию

Ещё одно отличие от Java: **классы и методы по умолчанию `final`**.
Чтобы разрешить наследование, класс помечают `open`; чтобы переопределить
метод — `open` у родителя и обязательный `override` у наследника.

Это язык подталкивает к «composition over inheritance»: наследование —
осознанное решение автора класса, а не возможность по умолчанию.
Для фреймворков, которым нужны прокси-наследники (Spring, Hibernate),
классы открывает компиляторный плагин `kotlin-spring`/`kotlin-allopen`.

Иерархии с фиксированным набором вариантов выражаются `sealed class` /
`sealed interface` — прямой аналог sealed-классов Java, в паре с `when`
даёт проверяемую компилятором полноту разбора.

## Как ответить на интервью

Коротко: свойства объединяют поле и аксессоры, первичный конструктор
объявляется в заголовке класса. `data class` — аналог record: авто-`equals`/
`hashCode`/`copy`, идеален для DTO, но плох для JPA-сущностей (equals по всем
полям, нет no-arg конструктора, final). Вместо `static` — `object` (синглтон)
и `companion object`. Классы по умолчанию `final`, наследование — через
`open`/`override`; для Spring/Hibernate-прокси классы открывает плагин
`kotlin-spring`.
