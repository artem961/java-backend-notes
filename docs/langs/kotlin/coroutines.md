# Корутины

Корутины — механизм Kotlin для асинхронного и конкурентного кода, который
**пишется как обычный последовательный**. Решают ту же проблему, что
виртуальные потоки Java: тысячи одновременных операций ввода-вывода без
тысяч дорогих платформенных потоков и без колбэков/реактивщины.

## Идея: приостановка вместо блокировки

Обычный поток, ожидая ответа БД или HTTP, **блокируется** — стоит и держит
ресурсы. Корутина в этот момент **приостанавливается** (suspend): её поток
освобождается и выполняет другие корутины, а когда ответ пришёл —
выполнение продолжается с того же места. Корутины — лёгкие задачи поверх
небольшого пула потоков, их можно запускать сотнями тысяч.

```kotlin
suspend fun loadUser(id: Long): User {
    val profile = api.fetchProfile(id)     // приостановка, поток не заблокирован
    val orders = api.fetchOrders(id)       // последовательно, но без блокировок
    return User(profile, orders)
}
```

Ключевое слово **`suspend`** помечает функцию, которая умеет
приостанавливаться. Вызывать её можно только из другой suspend-функции или
из корутины — так компилятор гарантирует, что блокирующий стиль не смешается
с приостанавливаемым. Под капотом компилятор превращает функцию в конечный
автомат с колбэками — но пишется и читается она как обычный код.

## Запуск и параллельность

```kotlin
suspend fun loadDashboard(id: Long): Dashboard = coroutineScope {
    val profile = async { api.fetchProfile(id) }   // запустить параллельно
    val orders = async { api.fetchOrders(id) }
    Dashboard(profile.await(), orders.await())     // дождаться обоих
}
```

- **`launch`** — запустить корутину «выстрелил и забыл» (аналог `execute`).
- **`async` / `await`** — запустить и получить результат (аналог
  `CompletableFuture`, но код остаётся линейным).
- **`runBlocking`** — мост из блокирующего мира: запускает корутину
  и блокирует текущий поток до её конца (main, тесты — не прод-код).

**Структурная конкурентность** — важное отличие от голых потоков
и `CompletableFuture`: корутины запускаются внутри области (`coroutineScope`),
и область не завершится, пока не завершатся все её дети. Упал один `async` —
отменяются остальные и ошибка летит наверх обычным исключением. «Потерянных»
фоновых задач, о которых никто не узнает, не бывает.

Отмена — кооперативная, как прерывание потоков: suspend-функции проверяют
отмену в точках приостановки и бросают `CancellationException`.

## Диспетчеры

На каких потоках выполняются корутины, решает **диспетчер**:

- `Dispatchers.Default` — пул по числу ядер, для CPU-работы;
- `Dispatchers.IO` — расширяемый пул для **блокирующего** IO (JDBC, файлы);
- `withContext(Dispatchers.IO) { ... }` — переключить контекст на время блока:
  так оборачивают блокирующие вызовы, чтобы не занимать основной пул.

## Корутины и виртуальные потоки

Одна задача — два решения:

| | Корутины | Виртуальные потоки (Java 21) |
|---|---|---|
| Уровень | компилятор Kotlin + библиотека | JVM |
| Код | помечается `suspend` | обычный блокирующий код |
| Работает с | suspend-совместимыми библиотеками | любым Java-кодом |
| Бонусы | структурная конкурентность, отмена, `Flow` | ничего менять не надо |

Виртуальные потоки делают лёгким сам блокирующий вызов, корутины — дают
язык поверх: структурную конкурентность, отмену, потоки данных (`Flow`).
В Kotlin-проектах корутины остаются идиоматичным выбором; на виртуальных
потоках их диспетчеры тоже умеют работать.

## Как ответить на интервью

Коротко: корутины — лёгкие задачи, которые при ожидании IO приостанавливаются
и отпускают поток вместо блокировки, поэтому их можно запускать сотнями тысяч.
`suspend` помечает приостанавливаемую функцию, `launch`/`async`+`await`
запускают, `withContext(Dispatchers.IO)` изолирует блокирующие вызовы.
Структурная конкурентность: корутины живут в области, ошибки и отмена
распространяются по дереву — фоновые задачи не теряются. Решают ту же
проблему, что виртуальные потоки Java, но дают ещё структуру и отмену.
