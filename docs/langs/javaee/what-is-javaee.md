# Что такое JavaEE и Jakarta EE

Java EE (Enterprise Edition) — **набор стандартов** для серверной Java:
спецификации, описывающие, как должны выглядеть API для веба, транзакций,
персистентности, DI и прочего энтерпрайза. Это не фреймворк-продукт,
а документы + эталонные API, которые реализуют разные вендоры.

## Идея: спецификация отдельно, реализация отдельно

Каждая часть Java EE — отдельная спецификация: JPA (персистентность),
Servlet (веб), JAX-RS (REST), CDI (DI), JMS (очереди), Bean Validation
(валидация). У спецификации может быть несколько реализаций:

| Спецификация | Реализации |
|---|---|
| JPA | Hibernate, EclipseLink |
| Servlet | Tomcat, Jetty, Undertow |
| JAX-RS | Jersey, RESTEasy |
| CDI | Weld, OpenWebBeans |
| Bean Validation | Hibernate Validator |

Смысл — переносимость: код пишется против стандартного API
(`jakarta.persistence.*`), а реализацию можно менять. Именно поэтому
«JPA — спецификация, Hibernate — реализация» — дежурная фраза на интервью.

## Java EE → Jakarta EE

История, объясняющая два названия одного и того же:

1. **J2EE → Java EE** — платформа развивалась в Sun/Oracle с конца 90-х.
2. В 2017 Oracle передала платформу фонду **Eclipse**. Права на бренд «Java»
   остались у Oracle, поэтому платформу переименовали в **Jakarta EE**.
3. Практическое следствие переименования — **смена пакетов
   `javax.*` → `jakarta.*`** (Jakarta EE 9+). Это была болезненная миграция
   всей экосистемы: Spring Boot 3 и Hibernate 6 уже на `jakarta.*`,
   старые библиотеки с `javax.*` с ними несовместимы. Ошибки вида
   «класс `javax.persistence.Entity` не найден» — обычно именно конфликт
   этих миров.

Сегодня «Java EE» — историческое название, актуальное — Jakarta EE.

## Классическая модель: сервер приложений

Изначальная модель развёртывания Java EE отличалась от современной:
приложение собиралось в **WAR/EAR** и деплоилось в **сервер приложений**
(WildFly, WebLogic, WebSphere, GlassFish), который предоставлял всю
инфраструктуру — контейнер сервлетов, EJB-контейнер, пулы соединений,
транзакции, безопасность. Один сервер мог держать несколько приложений.

Современный мир перевернул модель: **приложение само содержит сервер**
(Spring Boot с встроенным Tomcat, Quarkus) и деплоится как самодостаточный
jar/контейнер. Тяжёлые серверы приложений остались в легаси корпоративного
сектора.

## Зачем это знать, если пишешь на Spring

Spring не реализует Jakarta EE целиком, но **стоит на её спецификациях
и совместим с ними**:

- JPA — персистентность в Spring Data JPA (это Hibernate под капотом);
- Servlet API — фундамент Spring MVC (`DispatcherServlet` — это сервлет);
- Bean Validation — `@Valid`, `@NotNull` в контроллерах;
- аннотации CDI/DI (`@Inject`) Spring тоже понимает.

То есть Jakarta EE — это «система стандартов», из которой современные
фреймворки берут куски: Spring — прагматично выборочно, Quarkus —
как основу.

## Как ответить на интервью

Коротко: Java EE (теперь Jakarta EE) — набор спецификаций серверной Java:
JPA, Servlet, JAX-RS, CDI, Bean Validation и другие; реализации делают
вендоры (JPA → Hibernate, Servlet → Tomcat). В 2017 платформа переехала
от Oracle в Eclipse Foundation и переименована в Jakarta EE, что дало смену
пакетов `javax.*` → `jakarta.*` (Spring Boot 3 уже на них). Классическая
модель «WAR в сервер приложений» уступила встроенным серверам,
но спецификации живы: Spring MVC стоит на Servlet API, Spring Data JPA —
на JPA.
