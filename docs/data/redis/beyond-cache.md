# Redis не только кэш

Кэш — самое частое применение, но структуры данных и атомарность команд
делают Redis удобным для целого ряда задач. Полезно назвать несколько — это
показывает, что понимаешь инструмент шире «быстрого кэша».

## Счётчики и rate limiting

`INCR` атомарно увеличивает число — идеально для счётчиков (просмотры, лайки)
без гонок. На этом же строят **ограничение частоты запросов**: ключ на
пользователя с `INCR` и `EXPIRE` — «не больше N запросов в минуту». Именно так
работает rate limiter в API-шлюзе (см. Spring Cloud Gateway).

## Сессии

Хранилище пользовательских сессий: быстро, с TTL (сессия сама протухает), и —
главное — **общее для всех инстансов** приложения. Тогда неважно, на какой
инстанс попал запрос, сессия одна. Spring Session умеет складывать сессии в
Redis из коробки.

## Очереди и отложенные задачи

- **List** (`LPUSH`/`BRPOP`) — простая очередь задач между сервисами.
- **Sorted Set** с временем как весом — отложенные задачи: достаём те, у кого
  «время пришло».
- **Redis Streams** — более серьёзный лог событий с группами потребителей
  (ближе к мини-Kafka). Для настоящей шины сообщений всё же берут Kafka, но
  для лёгких сценариев Streams хватает.

## Pub/Sub

Публикация сообщений подписчикам в реальном времени. Простой механизм
уведомлений между инстансами (например, «сбросьте локальный кэш ключа X»).
Оговорка: Pub/Sub **не хранит** сообщения — кто не был подписан в момент
отправки, их не получит; для гарантий доставки это не подходит.

## Распределённая блокировка

Redis используют как **распределённый замок** между инстансами приложения:
кто поставил ключ — тот владеет ресурсом. Но это **сложнее, чем просто
поставить ключ** (см. вопрос в подборке): нужен атомарный
`SET key val NX PX <ttl>` (захват с TTL за одну команду), уникальный токен
владельца и снятие только своим токеном (через Lua-скрипт), а TTL — компромисс
между «зависший владелец не заблокирует навечно» и «не истечь раньше, чем
доделал работу». Надёжную реализацию берут готовую (Redlock/Redisson), а не
пишут руками.

## Как ответить на интервью

Коротко: кроме кэша, Redis — это атомарные счётчики и rate limiting (`INCR` +
`EXPIRE`), общее для всех инстансов хранилище сессий с TTL, лёгкие очереди
(List/Sorted Set) и отложенные задачи, Pub/Sub для уведомлений (без гарантии
доставки), Streams как мини-лог событий, и распределённые блокировки — но
последнее тоньше, чем кажется, и берут готовый Redisson. Общая причина, почему
это удобно, — скорость, TTL и атомарность команд.
