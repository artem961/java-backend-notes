# Персистентность и надёжность

Redis держит данные в памяти, но умеет сохранять их на диск, чтобы пережить
перезапуск. Понимать это важно, потому что от режима зависит, что произойдёт с
данными при падении.

## Что будет при перезапуске

Ответ на частый вопрос: **зависит от настроек и от того, кэш это или нет.**

- Если Redis используется как **кэш** — потеря данных при рестарте обычно
  **нормальна**: кэш просто снова наполнится из БД при следующих запросах
  (cache-aside), правильность системы не страдает, только на время просядет
  скорость (и есть риск лавины — см. проблемы кэширования). Именно поэтому
  для чистого кэша персистентность часто и не включают.
- Если Redis — **источник данных** (сессии, счётчики, очереди), потеря
  недопустима, и нужна персистентность и репликация.

## Два механизма сохранения на диск

### RDB (snapshot)

Периодический **снимок** всего состояния на диск (раз в N минут / N
изменений). Компактный, быстро грузится при старте. Минус — теряются изменения
**между** снимками: упал через 4 минуты после снимка, сделанного раз в 5 минут
— потеряны последние минуты.

### AOF (append-only file)

**Журнал всех команд** записи. При старте Redis проигрывает его и
восстанавливает состояние. Надёжнее RDB (можно настроить сброс на диск на
каждую команду или раз в секунду — обычно `everysec`), но файл больше и
восстановление медленнее.

| | RDB | AOF |
|---|---|---|
| Что хранит | снимок состояния | журнал команд |
| Потери при сбое | до последнего снимка | до 1 секунды (`everysec`) |
| Старт | быстрый | медленнее |

На практике часто включают **оба**: AOF для минимальных потерь + RDB для
быстрого восстановления и бэкапов.

## Надёжность в бою

- **Репликация** — реплики Redis (как у БД: primary принимает записи, реплики
  копируют) для отказоустойчивости и чтения.
- **Redis Sentinel** — следит за primary и делает автоматический failover
  (повышает реплику при падении).
- **Redis Cluster** — шардирование данных по узлам, когда объём/нагрузка не
  влезают в один сервер (данные разбиты по 16384 слотам).

## Как ответить на интервью

Коротко: Redis in-memory, но умеет сохранять на диск. При перезапуске судьба
данных зависит от режима: для кэша потеря обычно нормальна — он наполнится из
БД заново (риск только просадки скорости и лавины), поэтому персистентность
часто не включают; для сессий/счётчиков она нужна. Два механизма: RDB —
периодический снимок (компактно, но теряет изменения между снимками), AOF —
журнал команд (потери до секунды, но старт медленнее); часто включают оба.
Для отказоустойчивости — репликация + Sentinel (failover), для масштаба —
Cluster (шардирование).
