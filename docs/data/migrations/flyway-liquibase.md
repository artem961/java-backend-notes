# Flyway и Liquibase

Два основных инструмента миграций в мире Java. Оба решают одну задачу
(версионировать и применять изменения схемы), но по-разному.

## Flyway

Миграции — обычные **SQL-файлы** с соглашением об именах:

```
V1__create_users.sql
V2__add_email_index.sql
V3__add_orders_table.sql
```

- `V<версия>__<описание>.sql` — версионные миграции, применяются по порядку
  один раз.
- Flyway хранит применённое в таблице `flyway_schema_history` с контрольной
  суммой каждого файла.
- Есть repeatable-миграции (`R__...`) — применяются заново при изменении
  (для вьюх, функций).

Плюс Flyway — **простота и прозрачность**: пишешь чистый SQL, видишь ровно то,
что уйдёт в базу. Минус — SQL специфичен для конкретной СУБД (но для проекта
на одной PostgreSQL это не проблема).

## Liquibase

Миграции — **changelog**, где изменения описаны абстрактно (XML/YAML/JSON или
тоже SQL). Каждое изменение — `changeSet` с id и автором:

```yaml
databaseChangeLog:
  - changeSet:
      id: 1
      author: artem
      changes:
        - createTable:
            tableName: users
            columns: [ ... ]
```

- Абстрактное описание Liquibase транслирует в SQL под конкретную СУБД —
  **переносимость** между разными базами.
- Умеет генерировать **rollback** (в т.ч. автоматически для простых
  изменений).
- Ценой многословности и лишнего слоя абстракции над SQL.

## Что выбирать

| | Flyway | Liquibase |
|---|---|---|
| Формат | чистый SQL | changelog (XML/YAML/SQL) |
| Переносимость СУБД | нет (SQL под конкретную) | да (абстракция) |
| Rollback | вручную (новой миграцией) | встроенный |
| Простота/прозрачность | выше | ниже (слой абстракции) |

Практический ориентир: если база одна (PostgreSQL) и хочется прозрачности —
**Flyway** с чистым SQL, чаще выбирают его. Если нужна переносимость между
разными СУБД и встроенный rollback — Liquibase.

## Интеграция со Spring Boot

Оба подключаются автоконфигурацией: добавил зависимость, положил скрипты в
`src/main/resources/db/migration` (Flyway) — Boot **сам применит миграции при
старте** приложения (до создания EntityManager). В CI/CD миграции часто
выносят в отдельный шаг перед деплоем, чтобы схема обновлялась контролируемо,
а не в момент старта каждого инстанса.

Важно: миграции управляют схемой, а Hibernate `ddl-auto` — **выключить**
(`validate` максимум). Автогенерация схемы Hibernate и миграции
несовместимы: источник правды о схеме — миграции.

## Как ответить на интервью

Коротко: Flyway — миграции обычным SQL с версионными именами
(`V1__...sql`), просто и прозрачно, хранит историю с контрольными суммами;
привязан к конкретной СУБД. Liquibase — абстрактный changelog (XML/YAML),
переносим между базами и умеет rollback, но многословнее. Для одной
PostgreSQL чаще берут Flyway ради прозрачности. Оба Spring Boot применяет при
старте автоконфигурацией; при этом Hibernate `ddl-auto` держим на `validate`
— правда о схеме в миграциях, а не в автогенерации.
