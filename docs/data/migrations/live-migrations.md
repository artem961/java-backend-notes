# Миграции на живой базе

Самая содержательная часть темы и частый вопрос. Проблема: во время выкатки
на проде **одновременно работают две версии кода** — старая ещё не погашена,
новая уже поднимается, — а база одна. Плюс некоторые операции берут тяжёлые
блокировки на большой таблице и кладут сервис. Отсюда два требования к
миграции: **обратная совместимость** и **отсутствие долгих блокировок**.

## Обратная совместимость: expand / contract

Ключевой принцип — схема на каждом шаге совместима **и со старым, и с новым**
кодом. Резкие изменения разбивают на этапы (expand-and-contract):

Нельзя одним шагом «переименовать столбец `name` → `full_name`»: пока
катится релиз, старый код пишет в `name`, новый — в `full_name`, кто-то
сломается. Правильно:

1. **Expand** — добавить новый столбец `full_name` (nullable), не трогая
   старый.
2. Выкатить код, который **пишет в оба** столбца (или читает новый, пишет в
   оба).
3. **Backfill** — заполнить `full_name` из `name` для старых строк (батчами).
4. Выкатить код, работающий только с `full_name`.
5. **Contract** — в следующем релизе удалить старый `name`.

Каждый шаг разворачивается и откатывается независимо, ни в один момент
работающий код не встречает несовместимую схему.

## Опасные операции и блокировки

На большой таблице некоторые DDL берут блокировку, останавливающую запись
(а иногда и чтение) на всё время операции:

- **Добавить столбец `NOT NULL` со значением** — в старых версиях PostgreSQL
  переписывало всю таблицу под блокировкой. В PostgreSQL 11+ добавление с
  `DEFAULT` не переписывает таблицу, но навешивание `NOT NULL` на
  существующие данные проверяет их под блокировкой.
- **Создание индекса** — обычный `CREATE INDEX` блокирует запись в таблицу.
  Спасение — `CREATE INDEX CONCURRENTLY`: строит индекс без блокировки записи
  (дольше, нельзя внутри транзакции).
- **Добавление внешнего ключа / `CHECK`** — проверяет все строки под
  блокировкой. Спасение — двухшаговое: `ADD CONSTRAINT ... NOT VALID` (быстро,
  без проверки старых данных), затем `VALIDATE CONSTRAINT` (проверяет, не
  блокируя запись).

Общий приём для `NOT NULL` на большой таблице без долгой блокировки:
добавить `CHECK (col IS NOT NULL) NOT VALID`, затем `VALIDATE CONSTRAINT`.

## Откат, когда миграция уже применилась

Отдельная ситуация: релиз надо откатить, а миграция уже прошла. Именно поэтому
миграции делают expand/contract:

- Если миграция была **аддитивной** (добавили столбец/таблицу, ничего не
  ломая) — откат кода безопасен: новый столбец просто перестанет
  использоваться, старый код его игнорирует. Схему назад не откатывают.
- **Разрушительные** шаги (`DROP`, `NOT NULL`) не делают в том же релизе, что
  и код, который на них рассчитывает, — иначе откатить код будет некуда.
  Их выносят в **следующий** релиз, когда новый код уже стабилен.
- Общее правило: **разрушительная миграция — только когда старого кода уже
  точно нет**. Тогда откат кода не встречает сломанной схемы.

## Как ответить на интервью

Коротко: на проде во время выкатки работают две версии кода при одной базе,
поэтому миграция должна быть обратно совместимой. Резкие изменения (переимен
столбца) разбивают по схеме expand-and-contract: добавить новое → писать в
оба → забэкфилить батчами → перейти на новое → в следующем релизе удалить
старое. Плюс избегаем долгих блокировок: индекс — `CREATE INDEX CONCURRENTLY`,
constraint — `NOT VALID` + `VALIDATE`, `NOT NULL` на большой таблице — через
`CHECK ... NOT VALID`. И разрушительные операции (`DROP`, `NOT NULL`) делаем
отдельным поздним релизом, когда старого кода уже нет, — тогда откат
безопасен.
