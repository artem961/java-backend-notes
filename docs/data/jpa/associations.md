# Связи между сущностями

Связи отображают внешние ключи на ссылки между объектами. Здесь сосредоточена
бо́льшая часть сложностей Hibernate, поэтому важно понимать не аннотации, а их
последствия.

## Виды связей

| Аннотация | Связь | Где внешний ключ |
|---|---|---|
| `@ManyToOne` | много-к-одному | на этой стороне (у `Order` → `user_id`) |
| `@OneToMany` | один-ко-многим | на противоположной стороне |
| `@OneToOne` | один-к-одному | на одной из сторон |
| `@ManyToMany` | многие-ко-многим | в промежуточной таблице |

Базовый и самый частый — **`@ManyToOne`**: у заказа один пользователь.

```java
@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "user_id")
    private User user;
}
```

## Владелец связи и `mappedBy`

У двунаправленной связи одна сторона — **владелец** (та, где реально лежит
внешний ключ), вторая — обратная, помеченная `mappedBy`:

```java
@Entity
public class User {
    @OneToMany(mappedBy = "user")   // "user" — поле в Order
    private List<Order> orders = new ArrayList<>();
}
```

Ключевое правило: **внешний ключ в БД меняет только сторона-владелец**.
Добавил заказ в `user.getOrders()`, но не выставил `order.setUser(user)` —
в базе `user_id` останется пустым. Отсюда практика — синхронизировать обе
стороны в хелпер-методе (`addOrder`, который делает и то, и другое).

## FetchType: LAZY или EAGER — критично

Как грузить связь при загрузке сущности:

- **`LAZY` (лениво)** — связь не грузится сразу; подтягивается при первом
  обращении к полю (отдельным запросом).
- **`EAGER` (жадно)** — связь грузится сразу вместе с сущностью.

**Правило: везде `LAZY`.** По умолчанию `@ManyToOne`/`@OneToOne` — `EAGER`,
и это плохо: каждая загрузка сущности тянет связи, которые часто не нужны, и
плодит запросы. `@OneToMany`/`@ManyToMany` по умолчанию `LAZY` — оставляем.
Правильный подход: всё `LAZY`, а где связь **нужна** — грузить явно через
`join fetch`/`@EntityGraph` в конкретном запросе.

`EAGER` — не «оптимизация», а источник N+1 и лишних данных (см. следующую
тему).

## Каскады и orphan removal

- **`cascade`** — распространять операции на связанные сущности:
  `CascadeType.PERSIST` (сохранил заказ — сохранились позиции),
  `CascadeType.ALL` включает и удаление. Осторожно: `REMOVE`/`ALL` на связи с
  общими сущностями удалит лишнее.
- **`orphanRemoval = true`** — убрал элемент из коллекции → он удаляется из
  БД. Уместно для «композиции» (позиции заказа не живут без заказа).

## `@ManyToMany` — оговорка

`@ManyToMany` кажется удобным, но на практике промежуточная таблица почти
всегда обзаводится своими полями (когда добавили, количество, роль). Как
только это случилось — `@ManyToMany` не хватает, и связь разбивают на две
`@ManyToOne` через явную сущность-связку. Поэтому многие сразу моделируют
связку явно.

## Как ответить на интервью

Коротко: связи — это отражение внешних ключей. Основной — `@ManyToOne` (там и
лежит FK). В двунаправленной связи FK меняет только владелец, обратная
сторона — `mappedBy`; поэтому обе стороны синхронизируют вручную. Главное
правило — все связи `LAZY`, а где нужны — грузить явно через `join fetch`,
потому что `EAGER` (дефолт у `@ManyToOne`) плодит лишние запросы и N+1.
Каскады и `orphanRemoval` — для композиции (позиции заказа), но с `REMOVE`
осторожно. `@ManyToMany` на практике часто перерастает в явную сущность-связку.
