# Блокировки и конкурентный доступ

JPA даёт средства не дать параллельным транзакциям затереть изменения друг
друга. Это тот же выбор «оптимистичная vs пессимистичная блокировка», что и на
уровне SQL (см. соответствующую тему в разделе БД), только выраженный
аннотациями JPA.

## Оптимистичная блокировка: `@Version`

Добавляем в сущность поле версии:

```java
@Entity
public class Product {
    @Id private Long id;
    private int stock;

    @Version
    private long version;   // Hibernate ведёт его сам
}
```

Дальше Hibernate делает всё автоматически. При `UPDATE` он добавляет версию в
условие и инкрементирует её:

```sql
UPDATE product SET stock = ?, version = 6 WHERE id = ? AND version = 5;
```

- Строку никто не трогал → версия совпала, обновилась 1 строка, всё хорошо.
- Кто-то опередил → версия уже не 5, обновилось **0 строк** → Hibernate
  бросает **`OptimisticLockException`** (в Spring —
  `ObjectOptimisticLockingFailureException`).

Приложение решает, что делать с конфликтом: показать пользователю «данные
изменились, обновите» или **повторить** операцию (перечитать — применить —
сохранить). Подходит, когда конфликты редки: веб-формы, где между чтением и
сохранением проходит время.

## Пессимистичная блокировка: `@Lock`

Когда за строку реально дерутся многие, версия с постоянными повторами
дороже, чем сразу заблокировать. `@Lock` транслируется в `SELECT ... FOR
UPDATE`:

```java
@Lock(LockModeType.PESSIMISTIC_WRITE)
@Query("SELECT p FROM Product p WHERE p.id = :id")
Optional<Product> findByIdForUpdate(@Param("id") Long id);
```

- `PESSIMISTIC_WRITE` → `FOR UPDATE`: блокирует строку на запись, остальные
  ждут коммита.
- `PESSIMISTIC_READ` → `FOR SHARE`: разделяемая блокировка.

Работает только **внутри транзакции** (блокировка держится до её конца),
поэтому метод вызывается из `@Transactional`.

## Как выбирать — коротко

| | `@Version` (оптимистичная) | `@Lock` (пессимистичная) |
|---|---|---|
| Когда | конфликты редки | конфликты часты, горячая строка |
| Цена | повтор при конфликте | ожидание, риск deadlock |
| Дефолт для | веб-форм, длинной паузы чтение→запись | списаний, остатков под нагрузкой |

По умолчанию для веба берут `@Version` — он дешёвый и не держит блокировок;
пессимистичную — точечно, где конкуренция за конкретную строку высокая.

## Важные нюансы

- `@Version` защищает и от **потери обновлений между HTTP-запросами**: версию
  кладут в форму/ETag, при сохранении сверяют — так ловят «двое редактировали
  одну запись», даже если между чтением и записью транзакции разные.
- При ошибке сериализации/оптимистичной блокировки правильная реакция —
  **retry** на уровне сервиса (например, Spring Retry), а не показ 500.
- Пессимистичная блокировка требует держать транзакцию открытой — не
  затягивать её внешними вызовами.

## Как ответить на интервью

Коротко: JPA даёт тот же выбор оптимистичной и пессимистичной блокировки.
Оптимистичная — поле `@Version`: Hibernate добавляет версию в `WHERE` и
инкрементирует, обновилось 0 строк → `OptimisticLockException`, операцию
повторяем; дефолт для веб-форм, где конфликты редки и блокировку держать
нельзя. Пессимистичная — `@Lock(PESSIMISTIC_WRITE)` → `SELECT ... FOR UPDATE`,
блокирует строку в рамках транзакции; для горячих строк с частой
конкуренцией. `@Version` заодно ловит потерю обновлений между разными
HTTP-запросами, а на конфликт правильно делать retry.
