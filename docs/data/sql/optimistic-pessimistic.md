# Оптимистичная и пессимистичная блокировка

Обе стратегии решают одну задачу — не дать параллельным изменениям затереть
друг друга (lost update) — но с противоположными допущениями о том, часты ли
конфликты.

## Две стратегии

- **Пессимистичная**: «конфликт вероятен — заблокирую заранее». Транзакция
  захватывает строку (`SELECT ... FOR UPDATE`), остальные ждут. Конфликтов по
  сути нет — они превращаются в ожидание.
- **Оптимистичная**: «конфликт маловероятен — не блокирую, но проверю при
  записи». Никто никого не ждёт; при сохранении проверяем, не изменил ли
  строку кто-то ещё, и если да — откатываемся и повторяем.

## Как работает оптимистичная

К строке добавляют столбец **версии** (`version`, число) или отметку времени.
Каждая запись сверяет версию и инкрементирует её:

```sql
-- прочитали строку с version = 5
UPDATE products
SET    stock = :new, version = version + 1
WHERE  id = :id AND version = 5;      -- условие на версию!
```

- Если строку никто не трогал — `version` всё ещё 5, `UPDATE` затрагивает
  1 строку, версия становится 6.
- Если кто-то успел записать раньше — версия уже 6, условие `version = 5`
  не совпадает, `UPDATE` затрагивает **0 строк**. Это и есть обнаруженный
  конфликт: приложение видит «0 обновлённых» → перечитывает и повторяет
  операцию.

Именно так работает `@Version` в JPA/Hibernate — только условие на версию и
проверку «сколько строк обновлено» ORM добавляет сам, кидая
`OptimisticLockException` при несовпадении (см. раздел про блокировки в JPA).

## Когда что выбирать

| | Пессимистичная | Оптимистичная |
|---|---|---|
| Допущение | конфликты часты | конфликты редки |
| Механизм | блокировка строки (`FOR UPDATE`) | версия + проверка при записи |
| Цена | ожидание, риск deadlock, дольше держит ресурсы | повтор операции при конфликте |
| Хорошо, когда | короткая горячая точка, высокая конкуренция за одну строку | веб-формы, редкие пересечения, длинная «дума» между чтением и записью |

Практический ориентир:

- **Оптимистичная** — по умолчанию для веб-сценариев «пользователь открыл
  форму, подумал, сохранил»: держать блокировку всё это время нельзя, а
  конфликты редки. Проиграл гонку — получил понятную ошибку «данные изменились,
  обновите страницу».
- **Пессимистичная** — когда за одну строку реально дерутся много транзакций
  одновременно (списание с одного счёта, остаток на складе под распродажу):
  постоянные повторы оптимистичной вышли бы дороже, чем короткое ожидание.

## Ещё вариант — атомарный UPDATE

Если вся логика — «прибавить/убавить», конфликт можно вообще не разрешать на
уровне приложения: `UPDATE accounts SET balance = balance + 100 WHERE id = 1`
атомарен внутри одного оператора, гонки нет, ни версия, ни `FOR UPDATE` не
нужны. Блокировки нужны, когда между чтением и записью есть **логика в
приложении**, которую нельзя выразить одним `UPDATE`.

## Как ответить на интервью

Коротко: обе борются с lost update. Пессимистичная — заблокировать строку
заранее (`SELECT ... FOR UPDATE`), остальные ждут; хороша при частых
конфликтах за одну строку. Оптимистичная — не блокировать, а хранить версию
строки и при записи проверять `WHERE version = :read`: обновилось 0 строк —
кто-то опередил, повторяем; так работает `@Version` в JPA. Оптимистичная —
дефолт для веб-форм с долгой паузой между чтением и сохранением, где
блокировку держать нельзя. А если операция — просто инкремент, лучше вообще
атомарный `UPDATE x = x + 1` без всяких блокировок.
