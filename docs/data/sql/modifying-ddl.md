# Изменение данных и DDL

Команды SQL делятся на группы. Для собеседования важно различать **DML**
(данные) и **DDL** (структура) — они по-разному ведут себя в транзакциях.

| Группа | Команды | Про что |
|---|---|---|
| DML | `INSERT`, `UPDATE`, `DELETE`, `SELECT` | строки данных |
| DDL | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` | структура (таблицы, индексы) |
| DCL | `GRANT`, `REVOKE` | права |
| TCL | `COMMIT`, `ROLLBACK`, `SAVEPOINT` | управление транзакцией |

## Изменение данных (DML)

```sql
INSERT INTO users (email) VALUES ('a@x.io') RETURNING id;
UPDATE orders SET status = 'paid' WHERE id = 42;
DELETE FROM orders WHERE created_at < '2024-01-01';
```

Главные правила безопасности:

- **`UPDATE`/`DELETE` почти всегда с `WHERE`.** Без него меняется/удаляется
  вся таблица. Хорошая привычка — сначала выполнить `SELECT` с тем же
  `WHERE`, убедиться, что затронуты нужные строки.
- `RETURNING` (PostgreSQL) — вернуть данные изменённых строк (например,
  сгенерированный `id`), без отдельного `SELECT`.
- **`INSERT ... ON CONFLICT`** (upsert) — вставить или, при нарушении
  уникальности, обновить: `ON CONFLICT (email) DO UPDATE SET ...` /
  `DO NOTHING`. Частый способ идемпотентной вставки.

## DDL и важная особенность

`CREATE TABLE`, `ALTER TABLE ADD COLUMN`, `DROP` меняют структуру. Две вещи,
о которых спрашивают:

- **`DELETE` vs `TRUNCATE`.** `DELETE` — DML, удаляет строки по условию,
  пишет каждую в журнал, работает в транзакции и её видят триггеры.
  `TRUNCATE` — DDL, мгновенно очищает всю таблицу целиком, не по строкам;
  быстрее на больших таблицах, но без `WHERE`.
- **DDL и транзакции.** В PostgreSQL DDL **транзакционен** — `CREATE`/`ALTER`
  можно откатить (`ROLLBACK`), они внутри `BEGIN...COMMIT`. Это большое
  удобство для миграций. В MySQL (до 8.0) и Oracle DDL вызывает неявный
  `COMMIT` — откатить нельзя. На собесе стоит знать, что это зависит от СУБД.

## Пакетные операции

Вставлять тысячу строк тысячей отдельных `INSERT` — медленно (round-trip на
каждую). Правильно — **batch**: один `INSERT` с множеством `VALUES` либо
батч на уровне драйвера/JPA. Аналогично `UPDATE ... WHERE id IN (...)` вместо
цикла. Меньше обращений к БД — меньше накладных расходов и нагрузки.

## Как ответить на интервью

Коротко: DML (`INSERT`/`UPDATE`/`DELETE`) меняет строки, DDL
(`CREATE`/`ALTER`/`DROP`/`TRUNCATE`) — структуру. `UPDATE`/`DELETE` — всегда
с `WHERE`, иначе затронется вся таблица; `RETURNING` и `ON CONFLICT` (upsert)
— полезные приёмы PostgreSQL. `TRUNCATE` быстро чистит таблицу целиком в
обход построчного удаления. Важная деталь: в PostgreSQL DDL транзакционен и
откатывается, а в MySQL/Oracle DDL делает неявный commit. Массовые изменения
— батчами, не по одной строке.
