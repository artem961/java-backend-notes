# Пессимистичные блокировки

MVCC снимает блокировки с чтения, но когда несколько транзакций **меняют**
одни данные, порядок всё равно нужно упорядочить. Пессимистичная блокировка —
это «сначала захвати, потом меняй»: транзакция явно блокирует строки, и
остальные ждут, пока она не завершится.

## Блокировки строк

- **`SELECT ... FOR UPDATE`** — читает строки и **блокирует их на запись**.
  Другая транзакция, попытавшаяся сделать `FOR UPDATE`/`UPDATE`/`DELETE` тех
  же строк, будет ждать `COMMIT`/`ROLLBACK` первой. Классический способ
  закрыть lost update:

  ```sql
  BEGIN;
  SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;  -- захватили строку
  -- посчитали новое значение в приложении
  UPDATE accounts SET balance = :new WHERE id = 1;
  COMMIT;                                                 -- отпустили
  ```
  Пока первая транзакция держит строку, вторая на `FOR UPDATE` той же строки
  ждёт — и прочитает уже обновлённое значение. Гонка исключена.

- **`SELECT ... FOR SHARE`** — разделяемая блокировка: читать под ней могут
  многие, но никто не может изменить строку, пока держится хоть одна `FOR
  SHARE`. Нужна, когда важно, чтобы строка «не уехала» под тобой, но менять
  её ты не собираешься.

Обычный `UPDATE`/`DELETE` тоже блокирует затронутые строки до конца
транзакции — `FOR UPDATE` нужен, когда между чтением и записью есть логика.

## Что делать при конфликте

Ждать не всегда приемлемо. Модификаторы:

- **`FOR UPDATE NOWAIT`** — не ждать, а сразу упасть с ошибкой, если строка
  занята.
- **`FOR UPDATE SKIP LOCKED`** — пропустить занятые строки. Основа паттерна
  «очередь задач в БД»: несколько воркеров разбирают строки, каждый берёт
  `SELECT ... FOR UPDATE SKIP LOCKED LIMIT 1` и не мешает другим.

## Deadlock

Взаимная блокировка: транзакция A держит строку 1 и ждёт строку 2, а B
держит строку 2 и ждёт строку 1 — обе застряли навсегда.

```mermaid
flowchart LR
    A["Транзакция A<br/>держит строку 1"] -->|ждёт строку 2| B["Транзакция B<br/>держит строку 2"]
    B -->|ждёт строку 1| A
```

PostgreSQL **сам обнаруживает** deadlock (по графу ожиданий) и убивает одну
из транзакций с ошибкой `deadlock detected` — приложение должно её повторить.
Профилактика: **захватывать блокировки в одном и том же порядке** во всех
местах кода (например, всегда по возрастанию `id`) — тогда цикл ожидания не
образуется.

## Уровни блокировок

Блокировки бывают не только на строки, но и на таблицу (например, `ALTER
TABLE` берёт тяжёлую блокировку таблицы). Для прикладной корректности данных
почти всегда речь о **строковых** блокировках; о табличных вспоминают в
контексте миграций на живой базе (см. соответствующий раздел).

## Как ответить на интервью

Коротко: пессимистичная блокировка — «захватить строку до изменения»:
`SELECT ... FOR UPDATE` блокирует строки на запись, остальные ждут коммита —
так закрывают lost update, когда между чтением и записью есть логика.
`FOR SHARE` — разделяемая, `NOWAIT`/`SKIP LOCKED` — не ждать или пропускать
занятые (очередь задач в БД). Deadlock (A ждёт B, B ждёт A) PostgreSQL сам
детектит и убивает одну транзакцию — её надо повторить, а лучше не допускать,
захватывая блокировки в едином порядке.
