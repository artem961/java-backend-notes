# Уровни изоляции и аномалии

Полная изоляция параллельных транзакций — дорогая: пришлось бы выполнять их
фактически по очереди. Поэтому стандарт SQL определяет **уровни изоляции** —
компромисс между строгостью и параллелизмом. Понимать их проще «от аномалий»:
сначала — что именно может сломаться, потом — какой уровень это запрещает.

## Аномалии параллельного выполнения

- **Dirty read (грязное чтение)** — транзакция читает чужие изменения, ещё
  **не** зафиксированные. Сосед может откатиться — и мы прочитали то, чего
  «никогда не было».
- **Non-repeatable read (неповторяемое чтение)** — читаем строку дважды в
  одной транзакции и получаем **разные** значения, потому что между чтениями
  другая транзакция изменила и закоммитила эту строку.
- **Phantom read (фантомное чтение)** — повторяем **выборку по условию**
  (`WHERE amount > 100`) и получаем **новые строки**, которых не было: сосед
  вставил подходящие под условие записи.
- **Lost update (потерянное обновление)** — две транзакции читают одно
  значение, обе меняют и записывают; второй `COMMIT` затирает первый.
  Классика: два запроса «прочитать баланс → прибавить → записать», один
  инкремент теряется. Особенно важная аномалия для корректности данных.

## Уровни изоляции стандарта

Каждый следующий уровень запрещает больше аномалий (и стоит дороже):

| Уровень | Dirty read | Non-repeatable | Phantom |
|---|---|---|---|
| Read Uncommitted | возможен | возможен | возможен |
| Read Committed | нет | возможен | возможен |
| Repeatable Read | нет | нет | возможен¹ |
| Serializable | нет | нет | нет |

¹ по стандарту фантомы на Repeatable Read возможны, но конкретная СУБД может
запрещать больше — см. ниже про PostgreSQL.

## Что реально делает PostgreSQL

Важная деталь, которую любят на собесе: PostgreSQL реализует изоляцию через
**MVCC** (снимки данных, см. отдельную тему), и его уровни строже стандарта:

- **Read Uncommitted** в PostgreSQL **не существует** как отдельный уровень —
  ведёт себя как Read Committed. Грязного чтения в PostgreSQL нет вообще
  никогда.
- **Read Committed** — уровень **по умолчанию**. Каждый оператор видит снимок
  данных на момент своего начала: не видит чужого незакоммиченного, но между
  двумя запросами данные могут измениться (возможны non-repeatable и phantom).
- **Repeatable Read** — вся транзакция работает с одним снимком, взятым в её
  начале. В PostgreSQL этот уровень запрещает и фантомы тоже (строже
  стандарта). Если при `COMMIT` окажется, что данные, которые ты менял,
  успел изменить другой, — PostgreSQL бросит ошибку сериализации
  (`could not serialize access`), транзакцию нужно повторить.
- **Serializable** — гарантирует, что результат параллельных транзакций
  эквивалентен какому-то их последовательному выполнению. Реализован через
  SSI (Serializable Snapshot Isolation): ловит и опасные аномалии вроде
  **write skew** (обе транзакции по отдельности корректны, но вместе нарушают
  инвариант). Тоже может отклонить транзакцию с ошибкой сериализации.

## Как выбирать

- **Read Committed** (по умолчанию) достаточно для большинства операций.
- Для lost update при инкрементах на Read Committed недостаточно —
  либо атомарный `UPDATE ... SET x = x + 1` (в одном операторе гонки нет),
  либо блокировки/версионирование (см. следующие темы).
- **Repeatable Read / Serializable** — когда логика читает несколько строк и
  принимает по ним решение (нужна консистентная картина). Плата — возможные
  ошибки сериализации, которые приложение должно **повторять** (retry).

## Как ответить на интервью

Коротко: уровни изоляции — компромисс строгость/параллелизм, их удобно
объяснять через аномалии: dirty read (чужое незакоммиченное), non-repeatable
(строка изменилась между чтениями), phantom (появились новые строки под
условие), lost update (одно из двух обновлений затёрто). Стандартных уровней
четыре; каждый следующий запрещает больше. В PostgreSQL грязного чтения нет
вовсе, по умолчанию Read Committed (снимок на каждый оператор), Repeatable
Read даёт один снимок на транзакцию и запрещает даже фантомы, Serializable
через SSI ловит write skew. Строгие уровни могут отклонить транзакцию ошибкой
сериализации — её надо повторять.
