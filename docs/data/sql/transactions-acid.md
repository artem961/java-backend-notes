# Транзакции и ACID

Транзакция — группа операций, которые для БД выполняются как **одно неделимое
действие**: либо применяются все, либо ни одна. Классический пример — перевод
денег: списать у одного и зачислить другому. Если между этими шагами что-то
упадёт, недопустимо оставить деньги списанными, но не зачисленными.

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT;   -- либо ROLLBACK — тогда обе операции отменяются
```

Гарантии транзакции описывают аббревиатурой **ACID**.

## A — Atomicity (атомарность)

«Всё или ничего». Транзакция применяется целиком или откатывается целиком;
частичного результата не бывает. При `ROLLBACK` или сбое все изменения
отменяются.

**Как достигается.** Прежде чем изменить данные, PostgreSQL записывает
намерение в журнал упреждающей записи — **WAL** (Write-Ahead Log): сначала
пишем в лог «строка X меняется так-то», и только потом — сами страницы
данных. Каждая транзакция имеет статус в специальной структуре (commit log).
`COMMIT` — это атомарная запись в WAL «транзакция N зафиксирована». Если
падение случилось до этой записи — при перезапуске изменения незавершённой
транзакции просто не применяются (их не видно), как будто их и не было.
То есть атомарность — следствие того, что «фиксация» это один атомарный факт
в журнале, а не множество отдельных записей на диск.

## C — Consistency (согласованность)

Транзакция переводит БД из одного корректного состояния в другое: все
ограничения (`FK`, `UNIQUE`, `CHECK`, `NOT NULL`) на границе `COMMIT`
соблюдены. Если транзакция нарушила бы ограничение — она откатывается.
Это гарантия совместной работы БД и приложения: БД держит свои инварианты,
приложение — бизнес-правила.

## I — Isolation (изоляция)

Параллельные транзакции не мешают друг другу и не видят промежуточных,
незафиксированных изменений соседа. Степень изоляции настраивается уровнями
(Read Committed, Repeatable Read, Serializable) — это большая отдельная тема
(см. «Уровни изоляции и аномалии»), потому что полная изоляция дорогая, и на
практике выбирают баланс между строгостью и параллелизмом.

## D — Durability (долговечность)

Зафиксированная (`COMMIT`) транзакция переживёт сбой/выключение питания.
Достигается тем же WAL: `COMMIT` возвращает управление только после того, как
запись о фиксации **физически** сброшена на диск (`fsync`). Данные могли ещё
не дойти до основных файлов таблиц, но по WAL их восстановят при перезапуске
(накат журнала). Именно поэтому WAL закрывает сразу и атомарность (что
применять), и долговечность (не потерять).

## Транзакции на практике

- В Spring транзакцией управляют не руками (`BEGIN/COMMIT`), а декларативно —
  `@Transactional` (см. раздел про транзакции в Spring). Под капотом — та же
  транзакция БД.
- **Транзакция должна быть короткой.** Открытая транзакция держит соединение
  и ресурсы; долгая (особенно с внешними вызовами внутри) копит блокировки и
  мешает остальным.
- Автокоммит: без явного `BEGIN` каждый одиночный statement — это отдельная
  транзакция.

## Как ответить на интервью

Коротко: транзакция — неделимая группа операций, «всё или ничего», гарантии —
ACID. Atomicity — целиком или откат; достигается через WAL: сначала пишем
намерение в журнал, `COMMIT` — атомарный факт фиксации, недокоммиченное при
сбое просто не применяется. Consistency — на границе коммита все ограничения
соблюдены. Isolation — параллельные транзакции не видят чужого незакоммиченного
(регулируется уровнями). Durability — после `COMMIT` данные переживут сбой,
тоже благодаря WAL (запись сброшена на диск до ответа клиенту). То есть WAL
даёт сразу и атомарность, и долговечность.
