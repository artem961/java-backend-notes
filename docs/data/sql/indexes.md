# Индексы

Индекс — отдельная структура, которая ускоряет поиск строк по значению
столбца ценой места на диске и замедления записи. Без индекса СУБД читает
таблицу целиком (**seq scan**); с индексом идёт сразу к нужным строкам.

## Как это работает

Аналогия — указатель в конце книги: вместо перелистывания всех страниц
находим термин в отсортированном списке и прыгаем на нужную. По умолчанию
в PostgreSQL индекс — **B-tree** (сбалансированное дерево): хранит значения
в отсортированном виде, поиск, диапазон и сортировка — за `O(log n)`.

B-tree хорош для: `=`, `<`, `>`, `BETWEEN`, `IN`, `ORDER BY`, `LIKE 'abc%'`
(префикс). Не помогает для `LIKE '%abc'` (поиск с середины) — тут нужны
другие типы.

## Что индексировать

- **Столбцы в `WHERE`, `JOIN`, `ORDER BY`** — по которым реально ищут.
- **Внешние ключи** — PostgreSQL **не создаёт** индекс на FK автоматически
  (в отличие от PK), а `JOIN` и проверки при удалении родителя по ним идут
  постоянно. Частая забытая оптимизация.
- **`UNIQUE`-ограничения** — под ними индекс уже есть (им и проверяется
  уникальность).

Не индексировать всё подряд: каждый индекс замедляет `INSERT`/`UPDATE`/`DELETE`
(его тоже надо обновлять) и занимает место.

## Составной индекс и правило левого префикса

Индекс по нескольким столбцам `(a, b, c)` работает как отсортированный по `a`,
внутри — по `b`, внутри — по `c`. Отсюда **правило левого префикса**: такой
индекс помогает запросам по `a`, по `a, b`, по `a, b, c`, но **не** по одному
`b` или `c` — порядок столбцов в индексе важен. Поэтому первым ставят
столбец с равенством, потом — с диапазоном/сортировкой.

## Типы индексов в PostgreSQL

| Тип | Для чего |
|---|---|
| B-tree | по умолчанию: равенство, диапазоны, сортировка |
| Hash | только `=` (редко нужен, B-tree обычно не хуже) |
| GIN | полнотекстовый поиск, `JSONB`, массивы |
| BRIN | огромные таблицы с естественным порядком (по времени) |

Отдельно полезны:

- **Частичный индекс** — `CREATE INDEX ... WHERE status = 'active'`: индексирует
  только нужное подмножество, меньше и быстрее.
- **Индекс по выражению** — `CREATE INDEX ... (lower(email))` для поиска
  `WHERE lower(email) = ...`.
- **Покрывающий индекс** — включает все нужные запросу столбцы (`INCLUDE`),
  тогда БД берёт данные из индекса, не заглядывая в таблицу (index-only scan).

## Почему индекс может не сработать

- Функция/приведение типа над столбцом: `WHERE lower(email) = ...` не
  использует индекс по `email` (нужен индекс по выражению).
- `LIKE '%abc'` (ведущий `%`), `!=`, `NOT IN`.
- Планировщик считает, что проще прочитать таблицу целиком — на маленькой
  таблице или когда условие возвращает большую часть строк, seq scan
  реально дешевле, и это нормально.

Проверять, используется ли индекс, — через `EXPLAIN` (см. «Производительность
запросов»).

## Как ответить на интервью

Коротко: индекс — отсортированная структура (обычно B-tree), которая
превращает полный перебор таблицы в поиск за `O(log n)`; платим местом и
скоростью записи. Индексируем столбцы из `WHERE`/`JOIN`/`ORDER BY` и не
забываем внешние ключи — PostgreSQL их сам не индексирует. Для составного
индекса действует правило левого префикса, поэтому порядок столбцов важен.
Индекс не сработает при функции над столбцом, ведущем `%` в `LIKE` или когда
планировщик решит, что seq scan дешевле. Проверяем через `EXPLAIN`.
