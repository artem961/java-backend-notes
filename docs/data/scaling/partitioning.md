# Партиционирование

Партиционирование — разбиение **одной большой таблицы** на несколько
физических частей (партиций) внутри **той же базы**. Логически это по-прежнему
одна таблица: приложение пишет `SELECT ... FROM orders`, а PostgreSQL сам
направляет запрос в нужную партицию. Это важно не путать с шардированием
(разные серверы) — здесь всё на одном сервере.

## Зачем

Когда таблица разрастается до сотен миллионов строк, проблемы не столько в
поиске (индекс помогает), сколько в обслуживании: `VACUUM` идёт по всей
таблице, индексы огромны, удаление старых данных — тяжёлый `DELETE`.
Партиционирование бьёт таблицу на управляемые куски.

## Виды (PostgreSQL declarative partitioning)

Партиция определяется по значению столбца (ключ партиционирования):

- **По диапазону (RANGE)** — самый частый: по дате. Заказы за январь — одна
  партиция, за февраль — другая. Идеально для данных с временем.
- **По списку (LIST)** — по дискретному значению: партиция на регион, на тип.
- **По хешу (HASH)** — равномерно раскидать по N партиций, когда естественного
  диапазона нет.

```sql
CREATE TABLE orders (id bigint, created_at date, ...)
    PARTITION BY RANGE (created_at);

CREATE TABLE orders_2025_01 PARTITION OF orders
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');
```

## Что это даёт

- **Partition pruning** — если в запросе есть условие по ключу
  (`WHERE created_at >= '2025-02-01'`), планировщик читает **только нужные
  партиции**, а не всю таблицу. Главный выигрыш.
- **Дешёвое удаление старых данных** — вместо `DELETE` на миллионы строк
  (долго, bloat) просто `DROP TABLE orders_2025_01` — мгновенно отцепить
  и удалить целую партицию. Классика для хранения «последние N месяцев».
- **VACUUM и индексы по партиции** — обслуживание идёт по кускам, не по
  гигантской таблице.

## Ограничения

- Ключ партиционирования желательно в условиях запросов — иначе pruning не
  работает и читаются все партиции (хуже обычной таблицы).
- `UNIQUE`-ограничение должно включать ключ партиционирования.
- Усложняется схема; менять ключ задним числом больно.

## Партиционирование vs шардирование

| | Партиционирование | Шардирование |
|---|---|---|
| Где части | один сервер, одна БД | разные серверы |
| Прозрачность | СУБД сама, приложение не знает | обычно знает приложение/прокси |
| Решает | размер таблицы, обслуживание | размер, превышающий один сервер |

Партиционирование — «навести порядок внутри одного сервера». Когда данные не
влезают и в один сервер — тогда шардирование.

## Как ответить на интервью

Коротко: партиционирование — разбиение одной таблицы на части внутри той же
базы; логически таблица одна, физически — куски по ключу (обычно RANGE по
дате). Выигрыш: partition pruning (запрос с условием по ключу читает только
нужные партиции) и дешёвое удаление старых данных через `DROP` партиции
вместо тяжёлого `DELETE`, плюс обслуживание по кускам. Отличие от
шардирования — всё на одном сервере и прозрачно для приложения; шардирование
— это уже разные серверы.
