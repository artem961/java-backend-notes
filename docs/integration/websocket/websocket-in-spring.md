# WebSocket в Spring

Spring поддерживает WebSocket двумя способами: «сырой» низкоуровневый API и
более удобный **STOMP поверх WebSocket** с брокером сообщений. Для чатов и
рассылок обычно берут второй.

## Два уровня

- **Голый WebSocket** — реализуешь `WebSocketHandler`, работаешь с текстовыми/
  бинарными сообщениями руками. Максимум контроля, но всё вручную.
- **STOMP поверх WebSocket** — протокол обмена сообщениями с адресами
  (destination), подписками и брокером. Даёт паб/саб-модель «из коробки».

## STOMP-модель

STOMP добавляет поверх WebSocket понятную схему:

- Клиент **подписывается** на топик (`/topic/prices`).
- Сервер **публикует** сообщения в топик — они уходят всем подписчикам.
- Клиент **шлёт** сообщения на `@MessageMapping`-эндпоинты (аналог
  контроллеров).

```java
@Controller
class ChatController {
    @MessageMapping("/chat")      // клиент шлёт сюда
    @SendTo("/topic/messages")    // рассылается подписчикам
    public Message handle(Message in) {
        return in;
    }
}
```

Настройка — `@EnableWebSocketMessageBroker`: регистрируешь STOMP-эндпоинт для
подключения и включаешь брокер (простой in-memory или внешний, например
RabbitMQ/ActiveMQ для масштаба).

## Масштабирование

Простой встроенный брокер держит подписки **в памяти одного инстанса** — при
нескольких инстансах подписчики на разных нодах не увидят сообщения друг
друга. Для горизонтального масштаба подключают **внешний брокер** (или
Redis-релей), через который сообщения расходятся между инстансами.

## Как ответить на интервью

Коротко: в Spring WebSocket можно использовать голым `WebSocketHandler` (всё
руками) или через STOMP поверх WebSocket с брокером — второй удобнее для
чатов и рассылок. STOMP даёт паб/саб: клиент подписывается на топик, сервер
публикует туда сообщения, а входящие обрабатывают методы с
`@MessageMapping`. Включается через `@EnableWebSocketMessageBroker`. Важный
нюанс масштабирования: встроенный брокер хранит подписки в памяти одного
инстанса, поэтому для нескольких инстансов нужен внешний брокер.
