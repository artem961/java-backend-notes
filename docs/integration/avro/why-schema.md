# Зачем нужна схема

Когда сервисы обмениваются сообщениями (например через Kafka), встаёт вопрос:
как продюсер и консьюмер **договорятся о формате** данных и не сломают друг
друга при изменениях. Это решает схема.

## Проблема «просто JSON»

JSON удобен, но у него нет строгого контракта:

- Продюсер переименовал поле или сменил тип — консьюмер молча падает или читает
  мусор, узнаём об этом уже в проде.
- Нет проверки совместимости заранее — «договорённость» держится на честном
  слове и документации.
- Многословность: имена полей повторяются в каждом сообщении → больше байт.

## Что даёт схема

**Схема** — формальное описание структуры сообщения (поля, типы,
обязательность). С ней:

- **Контракт** явный и проверяемый — обе стороны знают формат.
- **Совместимость** можно проверять автоматически: новое изменение схемы либо
  совместимо со старыми потребителями, либо система его не пропустит.
- **Компактность** — при бинарной сериализации (Avro) передаются значения без
  повторения имён полей.
- **Кодогенерация** — из схемы можно сгенерировать классы.

## Где применяется

Типично — сообщения в Kafka: продюсер сериализует по схеме, консьюмер
десериализует по (совместимой) схеме. Схемы хранят централизованно в **Schema
Registry**, а формат данных — часто **Avro**.

!!! note "Честно про опыт"
    В проде со схемами сообщений не работал — разбирал теорию и пробовал на
    пет-проекте. Говорю про идею: явный контракт + проверка совместимости.

## Как ответить на интервью

Коротко: когда сервисы шлют друг другу сообщения, нужен явный контракт формата,
иначе изменение у продюсера (переименовал поле, сменил тип) молча ломает
консьюмеров уже в проде. Схема формально описывает структуру сообщения, даёт
проверяемую совместимость (можно заранее убедиться, что новое изменение не
сломает старых читателей), компактность при бинарной сериализации и
кодогенерацию. На практике это Avro для формата и Schema Registry для
централизованного хранения схем — типично в связке с Kafka.
