# Эволюция схем

Схемы меняются: добавляются поля, что-то устаревает. Эволюция схем — про то,
как менять формат сообщений, **не ломая** уже работающих продюсеров и
консьюмеров, которые обновляются не одновременно.

## Почему это важно

Продюсер и консьюмеры деплоятся в разное время. В любой момент в системе живут
**и старая, и новая** версия схемы одновременно. Значит, изменение должно быть
совместимым хотя бы в одну сторону, иначе часть сервисов сломается.

## Виды совместимости

- **Backward (обратная)** — код с **новой** схемой читает данные, записанные
  **старой**. Обычно обновляют сначала консьюмеров. Пример совместимого
  изменения: добавить поле **со значением по умолчанию**.
- **Forward (прямая)** — код со **старой** схемой читает данные, записанные
  **новой**. Обычно обновляют сначала продюсеров. Пример: удалить поле, у
  которого есть default.
- **Full** — совместимость в обе стороны.

## Правила безопасных изменений (Avro)

- **Добавить поле** — только с **default**, тогда старые данные читаются (поле
  получает дефолт), а старый код игнорирует новое.
- **Удалить поле** — безопасно, если у него есть default.
- **Переименовать/сменить тип** — **ломающее** изменение; делают через новое
  поле + переходный период, а не правкой на месте.
- Значения по умолчанию — главный инструмент совместимости.

## Роль Schema Registry

Registry **автоматически проверяет** новую версию схемы по выбранной политике
совместимости и отклоняет несовместимую. Это превращает договорённости в
принудительное правило: сломать потребителей несовместимым изменением система
просто не даст.

!!! note "Честно про опыт"
    В проде эволюцию схем не вёл — разбирал теорию и правила совместимости на
    пет-проекте.

## Как ответить на интервью

Коротко: продюсер и консьюмеры обновляются не одновременно, поэтому в системе
одновременно живут старая и новая схема, и изменение должно быть совместимым.
Backward — новый код читает старые данные (обновляем сначала консьюмеров),
forward — старый код читает новые данные (сначала продюсеров), full — оба.
Главный инструмент — значения по умолчанию: добавлять поле только с default,
удалять — если есть default; смена типа или переименование ломающие, их делают
через новое поле с переходным периодом. Schema Registry автоматически проверяет
совместимость и не пропускает ломающее изменение.
