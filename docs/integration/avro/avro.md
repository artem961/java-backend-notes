# Avro

Avro — формат сериализации данных со **схемой**. Данные кодируются бинарно и
компактно, а структуру описывает схема в JSON. Часто используется для сообщений
в Kafka.

## Как устроен

- **Схема** пишется в JSON: имя, поля, типы, значения по умолчанию.
- **Данные** сериализуются **бинарно** по этой схеме — в самих данных нет имён
  полей, только значения в порядке схемы. Отсюда компактность.
- Для чтения нужна схема — Avro использует **схему записи** (writer) и **схему
  чтения** (reader), сопоставляя их.

```json
{
  "type": "record",
  "name": "OrderCreated",
  "fields": [
    {"name": "id", "type": "long"},
    {"name": "accountId", "type": "string"},
    {"name": "amount", "type": "double"},
    {"name": "comment", "type": ["null", "string"], "default": null}
  ]
}
```

## Writer schema и reader schema

Ключевая идея Avro: данные можно прочитать схемой, **отличной** от той, которой
их записали, если они совместимы. Avro сопоставляет поля по имени и применяет
значения по умолчанию для отсутствующих — это и делает возможной эволюцию схем.

## Avro vs JSON vs Protobuf

- **JSON** — текст, человекочитаем, но многословен и без строгой схемы.
- **Avro** — бинарь + схема в JSON; схема нужна при чтении; силён в
  Kafka-экосистеме и эволюции.
- **Protobuf** — бинарь + `.proto`; поля по номерам; силён в gRPC.

Все бинарные форматы компактнее и быстрее JSON; выбор чаще диктует экосистема
(Avro — Kafka/Schema Registry, Protobuf — gRPC).

## Как ответить на интервью

Коротко: Avro — формат сериализации, где данные кодируются бинарно и компактно,
а структура описана схемой в JSON. В самих данных нет имён полей — только
значения по порядку схемы, поэтому компактно. Важная особенность — writer
schema и reader schema: данные можно читать схемой, отличной от записанной,
если они совместимы (поля по имени, значения по умолчанию), — на этом стоит
эволюция схем. По сравнению с Protobuf идея та же (бинарь + контракт), но Avro
живёт в экосистеме Kafka и Schema Registry.
