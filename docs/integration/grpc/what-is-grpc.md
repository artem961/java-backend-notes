# Что такое gRPC и зачем

gRPC — фреймворк для вызова удалённых процедур (**RPC**): вместо ресурсов и
URL, как в REST, ты вызываешь **метод сервиса** так, будто он локальный.
Контракт описывается в `.proto`, данные передаются бинарным **Protobuf**
поверх **HTTP/2**.

## Идея RPC

Определяешь сервис и его методы в контракте, генератор создаёт клиентский и
серверный код. Вызов `stub.getUser(request)` под капотом уходит по сети —
сериализация, транспорт и десериализация скрыты.

## Из чего состоит

- **Контракт `.proto`** — описание сервисов, методов и сообщений (единый
  источник правды для обеих сторон).
- **Protobuf** — бинарная сериализация: компактнее и быстрее JSON.
- **HTTP/2** — транспорт: мультиплексирование, стриминг, сжатие заголовков.
- **Кодогенерация** — по `.proto` генерятся классы и стабы для многих языков.

## Чем отличается от REST

| | REST | gRPC |
|---|---|---|
| Модель | ресурсы + методы HTTP | вызов процедур |
| Формат | обычно JSON (текст) | Protobuf (бинарь) |
| Контракт | опционально (OpenAPI) | обязателен (`.proto`) |
| Транспорт | HTTP/1.1 и выше | HTTP/2 |
| Стриминг | ограниченно | нативно, в обе стороны |
| Браузер | напрямую | только через прокси (grpc-web) |

## Где уместен

- **Межсервисное общение** внутри системы — быстро, строгий контракт, стриминг.
- Низколатентные и высоконагруженные внутренние API.
- **Не** идеален для публичного API «наружу» и прямых вызовов из браузера — там
  чаще REST.

## Как ответить на интервью

Коротко: gRPC — это RPC-фреймворк, где ты вызываешь метод удалённого сервиса
как локальный. Контракт описывается в `.proto`, данные летят бинарным Protobuf
поверх HTTP/2, а клиент и сервер генерируются из контракта. По сравнению с REST
он компактнее и быстрее, обязателен строгий контракт и нативно поддержан
двусторонний стриминг — поэтому его берут для межсервисного общения внутри
системы. Наружу и в браузер чаще оставляют REST (gRPC в браузере — только через
grpc-web).
