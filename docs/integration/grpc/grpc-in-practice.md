# gRPC на практике

Как gRPC выглядит в реальной разработке: генерация кода, ошибки, дедлайны,
эксплуатация — и где его границы.

## Рабочий цикл

1. Пишешь `.proto` — контракт.
2. Плагин сборки (Maven/Gradle) по нему **генерирует** сообщения и стабы.
3. На сервере реализуешь сгенерированный интерфейс сервиса.
4. Клиент вызывает методы через сгенерированный стаб.

Контракт — единый источник правды; при изменении `.proto` перегенерируются обе
стороны.

## Ошибки и статусы

Вместо HTTP-кодов — свой набор **status codes** gRPC: `OK`, `NOT_FOUND`,
`INVALID_ARGUMENT`, `PERMISSION_DENIED`, `UNAVAILABLE`, `DEADLINE_EXCEEDED` и
др. Детали ошибки можно прикладывать структурировано.

## Дедлайны и отмена

- **Deadline/timeout** — клиент задаёт крайний срок; если сервер не успел,
  вызов обрывается с `DEADLINE_EXCEEDED`. Дедлайн **пробрасывается** по цепочке
  вызовов — важно для распределённых систем, чтобы не копить зависшие запросы.
- **Cancellation** — клиент может отменить вызов, сервер узнаёт об этом.

## Эксплуатация

- **Метаданные** (аналог заголовков) — прокидывают токены, trace-id.
- **Interceptors** — аналог фильтров/middleware: логирование, аутентификация,
  метрики.
- **Балансировка** — сложнее, чем в HTTP/1.1: HTTP/2 держит долгие соединения,
  поэтому нужен client-side LB или прокси, умеющий gRPC.
- **Наблюдаемость** — метрики и трейсинг через interceptors.

## Границы применения

- В **браузер** напрямую нельзя — нужен **grpc-web** через прокси.
- Для публичного API наружу чаще REST (проще, отлаживается curl'ом, знаком
  всем).
- Отладка бинарного трафика менее наглядна, чем текстового JSON.

    В проде не применял — пробовал на пет-проекте. Уверенно говорю про модель
    (контракт, кодогенерация, дедлайны, статусы), а не про тонкости

## Как ответить на интервью

Коротко: цикл — пишешь `.proto`, сборка генерит стабы и сообщения, реализуешь
сервис, клиент зовёт через стаб. Ошибки — свои gRPC-статусы (`NOT_FOUND`,
`UNAVAILABLE`, `DEADLINE_EXCEEDED`), а не HTTP-коды. Важная штука — дедлайны:
клиент задаёт срок, он пробрасывается по цепочке, чтобы не копились зависшие
вызовы. Сквозные вещи (токены, логирование, метрики) вешают через метаданные и
interceptors. Из ограничений: в браузер только через grpc-web, наружу обычно
всё равно REST, а бинарный трафик труднее отлаживать.
