# Protobuf и контракт

Protobuf (Protocol Buffers) — язык описания данных и бинарный формат
сериализации. В gRPC файл `.proto` — единый **контракт**: по нему генерят код
для клиента и сервера, и обе стороны гарантированно понимают друг друга.

## Как выглядит контракт

```proto
syntax = "proto3";

message GetUserRequest {
  int64 id = 1;
}

message User {
  int64 id = 1;
  string name = 2;
  string email = 3;
}

service UserService {
  rpc GetUser(GetUserRequest) returns (User);
}
```

- **`message`** — структура данных с типизированными полями.
- **`service` / `rpc`** — сервис и его методы (вход → выход).
- **Номера полей** (`= 1`, `= 2`) — вот что важно: в бинарном формате
  передаётся **номер**, а не имя поля.

## Почему бинарь компактнее JSON

- Передаются номера полей и упакованные значения, а не текстовые имена и
  кавычки.
- Числа кодируются переменной длиной (varint), нет накладных расходов на
  разбор текста.
- Меньше байт по сети + быстрее (де)сериализация — отсюда выигрыш на нагрузке.

## Эволюция схемы (совместимость)

Правила, чтобы не сломать старых клиентов:

- **Не менять номер** у существующего поля и не переиспользовать номера
  удалённых полей (помечать `reserved`).
- Добавлять новые поля — с **новыми номерами**; старый код их просто
  проигнорирует.
- В proto3 у полей нет «обязательности» в старом смысле — отсутствующие поля
  получают значения по умолчанию, что помогает совместимости.

## Как ответить на интервью

Коротко: Protobuf — это и язык описания данных, и бинарный формат. Контракт в
`.proto` описывает сообщения и сервисы, по нему генерится код для обеих сторон.
Ключ к формату — у каждого поля есть числовой номер, и по сети идёт именно
номер, а не имя, поэтому получается компактнее и быстрее JSON. Совместимость
держится на правилах: номера полей не меняют и не переиспользуют, новые поля
добавляют с новыми номерами, а старый код неизвестные поля просто игнорирует.
