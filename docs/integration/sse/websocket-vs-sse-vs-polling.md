# Что выбрать: WebSocket, SSE или Long Polling

Все три решают задачу «получать обновления от сервера», но по-разному. Выбор
зависит от направления данных, частоты обновлений и сложности, которую готов
нести.

## Сравнение

| | Направление | Транспорт | Сложность | Когда |
|---|---|---|---|---|
| **Polling** | клиент опрашивает | обычные HTTP-запросы | минимальная | редкие/некритичные обновления |
| **Long polling** | сервер→клиент (эмуляция) | висящий HTTP-запрос | низкая | пуш без WebSocket, простые случаи |
| **SSE** | сервер→клиент | один открытый HTTP | низкая | поток событий в одну сторону |
| **WebSocket** | обе стороны | отдельный дуплекс-канал | высокая | интерактив в обе стороны |

## Как выбирать

- **Нужен обмен в обе стороны** (чат, игра, совместное редактирование) →
  **WebSocket**.
- **Только сервер шлёт события** (уведомления, котировки, прогресс задачи) →
  **SSE**: проще, обычный HTTP, авто-reconnect из коробки.
- **Обновления редкие**, реалтайм не критичен → **обычный polling**: проще
  всего, никаких долгих соединений.
- **Нужен пуш, но нельзя WebSocket/SSE** (старое окружение, ограничения прокси)
  → **long polling** как запасной вариант.

## Правило по умолчанию

Не бери WebSocket «на всякий случай» — он дороже в эксплуатации (stateful
соединения, sticky sessions, брокер для рассылки). Если поток нужен в одну
сторону, начинай с SSE; если данные редкие — с polling. WebSocket — когда
реально нужен двусторонний интерактив.

## Как ответить на интервью

Коротко: выбираю по направлению данных. Обе стороны и интерактив — WebSocket,
но он дороже (stateful, sticky sessions, брокер для рассылки). Только
сервер→клиент — SSE: обычный HTTP, авто-переподключение, проще. Редкие
обновления — обычный polling, без долгих соединений. Long polling — запасной
вариант для пуша там, где WebSocket/SSE недоступны. Общее правило: не тащить
WebSocket по умолчанию, начинать с самого простого, что закрывает задачу.
