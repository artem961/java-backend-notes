# SSE в Spring

В Spring SSE поддержан прямо в MVC — не нужен отдельный протокол или брокер.
Достаточно вернуть из контроллера специальный тип, и Spring будет держать
соединение открытым, отдавая события по мере их появления.

## Простой способ — SseEmitter

Контроллер возвращает `SseEmitter`; в него из другого потока пушат события,
пока не закроют.

```java
@GetMapping("/stream")
public SseEmitter stream() {
    SseEmitter emitter = new SseEmitter();
    // из фонового потока:
    emitter.send(SseEmitter.event().name("price").data(100));
    // ...по завершении:
    emitter.complete();
    return emitter;
}
```

- Соединение живёт, пока не вызовут `complete()` или не случится таймаут/
  ошибка.
- Важно обрабатывать `onCompletion`/`onError`/`onTimeout` и убирать эмиттер из
  своих структур, иначе — утечка.

## Реактивный способ — Flux

В WebFlux (реактивный стек) поток событий — это просто `Flux`, отдаваемый как
`text/event-stream`:

```java
@GetMapping(value = "/stream", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
public Flux<Price> stream() {
    return priceService.priceUpdates(); // поток обновлений
}
```

Реактивный стек лучше держит **много одновременных** долгих соединений
(неблокирующий I/O), тогда как в классическом MVC каждое висящее соединение
занимает поток из пула.

## Эксплуатация

- **Таймауты** — задавать осознанно; долгое соединение не должно висеть вечно.
- **Много соединений** — в блокирующем MVC упирается в пул потоков; для
  масштаба лучше WebFlux.
- **Прокси** — убедиться, что не буферизуют и не рубят стрим (может понадобиться
  отключить буферизацию).

## Как ответить на интервью

Коротко: в Spring MVC SSE делают через `SseEmitter` — контроллер возвращает
эмиттер, в него из фонового потока шлют события, а по завершении вызывают
`complete()`; обязательно чистить эмиттеры по `onError`/`onTimeout`, иначе
утечка. В реактивном WebFlux поток — это `Flux`, отдаваемый как
`text/event-stream`, и реактивный стек лучше держит много долгих соединений,
потому что в блокирующем MVC каждое висящее соединение занимает поток из пула.
Ещё следят за таймаутами и за тем, чтобы прокси не буферизовал стрим.
