# Ошибки и статусы

Ошибки в REST — часть контракта. Клиент должен по коду статуса понять, что
случилось и что делать (повторить, поправить запрос, залогиниться), а тело —
дать детали для человека и программы.

## Код статуса несёт смысл

- **4xx** — виноват клиент, повторять бессмысленно без изменений:
  `400` (некорректный запрос), `401` (не аутентифицирован), `403` (нет прав),
  `404` (нет ресурса), `409` (конфликт состояния), `422` (не прошло валидацию),
  `429` (превышен лимит).
- **5xx** — виноват сервер, повтор может помочь: `500` (внутренняя),
  `503` (временно недоступен).

!!! warning "Антипаттерн"
    Отдавать `200 OK` с `{"error": ...}` в теле. Клиент и промежуточные
    прокси/мониторинг судят по коду — ошибку нельзя прятать под успех.

## Тело ошибки — единый формат

Один формат на всё API, чтобы клиент парсил ошибки одинаково. Хороший
ориентир — **RFC 7807 (Problem Details)**:

```json
{
  "type": "https://api.example.com/errors/insufficient-funds",
  "title": "Недостаточно средств",
  "status": 409,
  "detail": "На счёте 42 не хватает 500 руб.",
  "instance": "/accounts/42/withdraw",
  "traceId": "abc-123"
}
```

- **Машиночитаемый код** ошибки (`type`/свой `code`) — чтобы клиент реагировал
  логикой, а не парсил текст.
- **Человекочитаемое сообщение** — для логов/поддержки.
- **`traceId`** — для связки с логами и трассировкой.
- Для валидации — список полей с ошибками.

## Безопасность в ошибках

Не отдавать наружу стектрейсы, SQL, внутренние пути — это утечка информации.
Детали в лог (по `traceId`), клиенту — аккуратное сообщение.

## В Spring

Централизованно через `@RestControllerAdvice` + `@ExceptionHandler`: маппить
исключения в нужный статус и единый формат тела. Есть готовый
`ProblemDetail`/`ResponseEntity`.

## Как ответить на интервью

Коротко: код статуса — часть контракта: 4xx значит «поправь запрос» (400/401/
403/404/409/422/429), 5xx — «проблема на сервере, можно повторить». Прятать
ошибку под `200 OK` нельзя — по коду судят клиенты и мониторинг. Тело ошибки —
единый формат (удобно по RFC 7807 Problem Details): машиночитаемый код +
человеческое сообщение + `traceId` для связки с логами. Наружу не отдаём
стектрейсы и SQL. В Spring это делают централизованно через
`@RestControllerAdvice`.
