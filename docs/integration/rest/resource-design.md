# Проектирование ресурсов и данных

Хорошее REST API читается как понятная модель предметной области: ресурсы,
связи, предсказуемые URL и аккуратные тела запросов/ответов.

## Именование ресурсов

- Существительные во **множественном числе**: `/users`, `/orders`.
- Идентификатор — в пути: `/users/42`.
- Вложенность для связей: `/users/42/orders` — заказы пользователя 42.
- Глаголы в URL — плохо; действие несёт метод. Исключение — операции, не
  ложащиеся в CRUD (`POST /orders/42/cancel`): допустимо, не надо
  фанатично прятать под ресурс.

## Коллекция и элемент

- `GET /users` — список (с пагинацией/фильтрами).
- `POST /users` — создать, вернуть `201 Created` + `Location` на новый ресурс.
- `GET /users/42` — один элемент.
- `PUT /users/42` — заменить целиком; `PATCH /users/42` — частично.
- `DELETE /users/42` — удалить.

## Параметры запроса

- **Фильтрация**: `GET /orders?status=paid`.
- **Пагинация**: `?page=2&size=20` или курсорная `?cursor=...` (стабильнее на
  меняющихся данных).
- **Сортировка**: `?sort=createdAt,desc`.
- **Проекция полей**: `?fields=id,name` (отдавать не всё).

## Форма данных (DTO)

- Отдавать **DTO**, а не сущности БД напрямую — иначе наружу утекает внутренняя
  модель и её нельзя менять, не ломая клиентов.
- Не отдавать лишнее (пароли, внутренние флаги).
- Единый стиль полей (`camelCase`), явные типы дат (ISO-8601), деньги — строкой/
  минимальными единицами, не `double`.
- Связи — либо вложенным объектом, либо ссылкой/идентификатором; выбирать
  осознанно, чтобы не тянуть лишнее.

## Как ответить на интервью

Коротко: ресурсы — существительные во множественном числе (`/users/42`),
связи — вложенностью (`/users/42/orders`), действия несёт метод, а не URL
(глаголы в пути — плохо, кроме нестандартных операций вроде `/cancel`). Список
поддерживает фильтры, пагинацию и сортировку через query-параметры. Наружу
отдаём DTO, а не JPA-сущности, чтобы не завязывать клиентов на внутреннюю
модель и не светить лишние поля; даты в ISO-8601, деньги не во `float`.
