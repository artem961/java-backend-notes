# Заголовки и тело

Заголовки — метаданные запроса/ответа (кто, что, в каком формате). Тело —
сами данные. Понимание ключевых заголовков нужно, чтобы объяснять контент,
кэш, авторизацию и content negotiation.

## Тело и его формат

- Формат тела задаёт заголовок **`Content-Type`** (`application/json`,
  `text/html`, `multipart/form-data` для файлов).
- Размер — **`Content-Length`**, либо потоковая передача через
  **`Transfer-Encoding: chunked`**, когда длина заранее неизвестна.
- У `GET`/`DELETE` тела обычно нет, данные идут в URL (path + query).

## Content negotiation

Клиент говорит, что готов принять, сервер выбирает представление:

- **`Accept`** — желаемый формат ответа (`application/json`);
- **`Accept-Language`** — язык;
- **`Accept-Encoding`** — сжатие (`gzip`, `br`), сервер отвечает
  `Content-Encoding`.

## Важные заголовки

- **`Host`** — какой домен запрашивается (обязателен в HTTP/1.1, позволяет
  одному IP обслуживать много сайтов).
- **`Authorization`** — креды/токен (`Bearer <jwt>`).
- **`Cookie` / `Set-Cookie`** — куки (см. отдельную тему).
- **`Cache-Control`, `ETag`** — управление кэшем (см. тему про кэширование).
- **`Content-Type`, `Accept`** — формат данных.
- **`User-Agent`** — что за клиент.
- **`X-Request-Id` / трейсинг-заголовки** — сквозной идентификатор запроса
  через сервисы (для логов и трассировки).

## Кастомные заголовки

Раньше префиксовали `X-` (`X-Request-Id`), сейчас префикс не обязателен, но по
традиции часто оставляют. Через заголовки удобно прокидывать сквозные вещи:
идентификатор запроса, версию клиента, флаги.

## Как ответить на интервью

Коротко: заголовки — это метаданные, тело — данные. Формат тела задаёт
`Content-Type`, размер — `Content-Length` или chunked-передача. Клиент через
`Accept`/`Accept-Language`/`Accept-Encoding` договаривается с сервером о
формате (content negotiation). Ключевые заголовки: `Host` (какой сайт),
`Authorization` (токен), `Cookie`/`Set-Cookie`, `Cache-Control`/`ETag` для
кэша. Сквозные вещи вроде `X-Request-Id` прокидывают через заголовки для
логирования и трассировки между сервисами.
