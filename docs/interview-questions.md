# Вопросы с собеседований

Банк вопросов для **закрепления**: прочитал тему в конспекте — проверь, что
понимаешь её, а не просто узнаёшь слова. Вопросы разбиты по блокам и темам,
внутри каждой — от простого к глубокому. Формат флеш-карт: читаешь вопрос,
формулируешь ответ вслух своими словами, потом раскрываешь спойлер и
сверяешься.

Упор не на зубрёжку («перечисли модификаторы»), а на понимание того, **как оно
работает внутри и почему так**, — потому что именно это проверяют уточняющими
вопросами. Особенно глубоко проработаны Java Core, многопоточность, JVM,
внутренности Spring, согласованность данных и устройство реального прода.

## Java

### Java Core: классы, объекты, полиморфизм

??? question "Что происходит в памяти, когда ты пишешь `new Order()`?"
    JVM выделяет в куче память под объект: **заголовок** (header) плюс поля.
    Заголовок содержит mark word (хеш, биты блокировки, возраст для GC)
    и указатель на класс (klass pointer) — по нему JVM знает тип и таблицу
    методов. Поля сначала зануляются (0/false/null), затем выполняется цепочка
    конструкторов сверху вниз от `Object`.

    В стеке текущего потока лежит только **ссылка** (4 или 8 байт) на объект,
    сам объект — в куче. Отсюда важные следствия: две ссылки могут указывать
    на один объект (aliasing), а `==` сравнивает ссылки, не содержимое.

??? question "В каком порядке инициализируется объект наследника?"
    Строгий порядок:

    1. **Статика** родителя, потом наследника (статические поля и блоки) —
       один раз при загрузке класса.
    2. При `new`: поля и блоки инициализации экземпляра **родителя**, затем
       тело конструктора родителя.
    3. Поля и блоки экземпляра **наследника**, затем тело его конструктора.

    Ловушка, которую любят: если конструктор родителя вызывает
    переопределённый метод, тот выполнится в наследнике **раньше**, чем
    инициализируются поля наследника, — увидишь `null`/`0`. Поэтому
    из конструктора не вызывают переопределяемые методы.

??? question "Можно ли переопределить `static`-метод? А `private`?"
    Нет и нет — оба не участвуют в полиморфизме.

    - **static** привязан к классу. Одноимённый метод в наследнике — это
      сокрытие (hiding), а не переопределение: какой вызовется, решает
      **статический тип ссылки** на этапе компиляции, а не фактический объект.
    - **private** вообще не виден наследнику — «переопределять» нечего,
      в наследнике будет независимый метод.

    Полиморфно (по фактическому типу объекта) вызываются только обычные
    методы экземпляра. `final` и конструкторы тоже разрешаются статически.

??? question "Как JVM решает, какой метод вызвать при `a.doWork()`, если тип ссылки родительский, а объект — наследник?"
    **Динамическая диспетчеризация.** У каждого класса есть таблица
    виртуальных методов (vtable), а ссылка на объект несёт klass pointer.
    В рантайме JVM по фактическому классу объекта находит в vtable нужную
    реализацию — поэтому вызовется метод наследника, хотя тип ссылки
    родительский. Компилятор лишь проверяет, что метод в типе ссылки есть.

    Не виртуальны (разрешаются на компиляции): `static`, `private`, `final`
    и конструкторы — их JIT может смело инлайнить.

??? question "Интерфейсы теперь имеют `default`-методы. Зачем тогда абстрактные классы?"
    Разница — в **состоянии и модели наследования**.

    - Абстрактный класс может иметь поля (изменяемое состояние), конструкторы,
      любые модификаторы доступа — но наследоваться можно только от одного.
    - Интерфейс — контракт без состояния (только константы); `default`-методы
      дают поведение по умолчанию, но данных не хранят, зато реализовать можно
      сколько угодно интерфейсов.

    Выбор: «is-a с общим состоянием и базовой реализацией» → абстрактный
    класс; «способность/роль, примешиваемая к разным иерархиям» → интерфейс.
    `default`-методы придумали в основном чтобы **расширять** существующие
    интерфейсы (добавить `stream()` в `Collection`), не ломая все реализации.

??? question "Что означает `final` для класса, метода, переменной и поля?"
    Четыре разных смысла:

    - **переменная/поле** — присвоить один раз (для ссылки фиксируется ссылка,
      не содержимое объекта);
    - **метод** — нельзя переопределить (сохраняет контракт, помогает JIT);
    - **класс** — нельзя наследовать (`String`, классы-значения).

    Отдельная важная деталь: **`final`-поля дают безопасную публикацию** —
    после завершения конструктора их значение корректно видно другим потокам
    без синхронизации. На этом стоит потокобезопасность неизменяемых объектов.

### equals, hashCode и идентичность

??? question "Зачем нужен hashCode, если есть equals?"
    Чтобы хеш-структуры работали за константное время. `HashMap` не сравнивает
    ключ со всеми существующими — он вычисляет `hashCode`, находит бакет
    и сравнивает через `equals` только внутри него.

    Отсюда контракт: **равные объекты обязаны иметь равный хеш**. Если
    переопределить только `equals`, два логически равных объекта попадут
    в разные бакеты, и `map.get(key)` вернёт `null` для ключа, который
    в карте есть. Ошибка не проявится ни при компиляции, ни в простом тесте
    на равенство — только на коллекциях.

    Обратное неверно: равные хеши не означают равные объекты. Это коллизия,
    нормальная ситуация, она разрешается через `equals`.

    Практическая ловушка: **мутабельные поля в ключе**. Положили объект
    в `HashSet`, изменили поле, участвующее в хеше, — объект остался
    в старом бакете, и найти его больше нельзя. Поэтому ключи делают
    неизменяемыми.

??? question "Почему equals и hashCode переопределяют всегда вместе?"
    Потому что хеш-структуры полагаются на их согласованность: сначала ищут
    бакет по `hashCode`, потом внутри сравнивают через `equals`.

    - Переопределил только `equals` — «равные» объекты имеют разный хеш,
      попадают в разные бакеты, и `HashMap` их равными не увидит.
    - Переопределил только `hashCode` — объекты лежат в одном бакете, но
      `equals` сравнивает по ссылке, и снова не находятся.

    Корректно работает только пара. `record` и генерация в IDE делают их
    согласованными автоматически — руками легко забыть половину.

??? question "Что будет, если hashCode() всегда возвращает одну и ту же константу?"
    Формально контракт не нарушен (равные объекты дают равный хеш), но все
    ключи падают в один бакет, и `HashMap` вырождается в список: `get`/`put`
    становятся O(n) вместо O(1). С Java 8 переполненный бакет (≥8 элементов
    при таблице ≥64) перестраивается в красно-чёрное дерево — тогда O(log n),
    но это всё равно не то, ради чего берут хеш-таблицу.

    Случайный хеш ещё хуже — ломает сам контракт: у объекта хеш меняется между
    вызовами, и `map.get(key)` не найдёт то, что только что положил. Правило:
    `hashCode` согласован с `equals`, стабилен, пока объект в коллекции,
    и **равномерно** распределяет — иначе коллизии съедают всю выгоду.

??? question "В equals сравнивать через getClass() или instanceof — есть разница?"
    Есть, и проявляется она на наследниках.

    - **`getClass() != o.getClass()`** — строгое сравнение: равны только объекты
      ровно одного класса. Симметричность гарантирована автоматически, но
      наследник никогда не равен родителю — даже если не добавил ни одного поля.
    - **`o instanceof Point`** — мягкое: объект наследника может быть равен
      родителю. Но если наследник переопределит `equals` с учётом своих полей,
      ломается симметричность: `parent.equals(child)` даст `true`,
      а `child.equals(parent)` — `false`. Коллекции при этом начинают вести
      себя непредсказуемо — `contains` зависит от того, кто с кем сравнивается.

    Честный вывод: корректно совместить наследование с `equals` по значению
    нельзя. Поэтому классы-значения делают `final` (или `record`) — и тогда
    оба варианта эквивалентны.

    Практическая деталь, за которую цепляются на собеседованиях по JPA:
    Hibernate подсовывает **прокси-подклассы** сущностей, у прокси `getClass()`
    отличается от класса сущности. `equals` через `getClass()` перестаёт
    находить равенство сущности и её прокси, поэтому для JPA-сущностей
    рекомендуют `instanceof`.

### Строки, примитивы и обёртки

??? question "Почему String неизменяемый и что это даёт?"
    Осознанное решение ради нескольких вещей сразу:

    - **безопасность** — строку с путём/хостом/SQL нельзя подменить после
      проверки, её можно спокойно передавать;
    - **пул строк** — одинаковые литералы переиспользуются, потому что их
      точно никто не изменит;
    - **кеш хеша** — `hashCode` считается один раз и хранится в поле, поэтому
      `String` — идеальный ключ `HashMap`;
    - **потокобезопасность даром** — неизменяемый объект шарится между потоками
      без синхронизации.

    Цена — конкатенация в цикле плодит объекты-мусор; для этого `StringBuilder`.

??? question "`Integer a = 127, b = 127; a == b` → true, а при 128 → false. Почему?"
    Автоупаковка идёт через `Integer.valueOf(int)`, а он **кеширует** объекты
    для диапазона −128..127 (Integer cache). Для 127 обе ссылки указывают на
    один закешированный объект, `==` (сравнение ссылок) даёт `true`. Для 128
    создаются два разных объекта, ссылки разные — `false`. Значения при этом
    равны, `equals` вернул бы `true` в обоих случаях.

    Мораль: обёртки всегда сравнивать через `equals`; `==` на них — источник
    плавающих багов, которые «на маленьких числах работают».

??? question "`int x = someInteger;` — как это может кинуть NPE?"
    Через **автораспаковку**. Если `someInteger` типа `Integer` равен `null`
    (например, `map.get(key)` не нашёл ключ), присваивание примитиву вызывает
    `someInteger.intValue()` на `null` → NPE. Коварство в том, что в строке
    нет ни одного явного вызова метода — NPE будто «из ниоткуда». Особенно
    любят на тернарнике: `Integer x = cond ? 1 : null;` и последующая
    распаковка. Вывод: смешивать `int` и `Integer` в арифметике опасно.

??? question "Чем `new String("abc")` отличается от литерала `"abc"`?"
    Литерал берётся из **пула строк** — один объект на весь JVM для одинаковых
    литералов. `new String("abc")` форсирует новый объект в куче, даже если
    такой уже есть в пуле. Поэтому `"abc" == "abc"` → `true` (один пул-объект),
    а `new String("abc") == "abc"` → `false` (разные объекты), при этом
    `equals` в обоих `true`. `new String` практически никогда не нужен;
    `intern()` умеет вернуть объект из пула, но злоупотреблять им не стоит.

### Исключения

??? question "Checked или unchecked — как выбирать и в чём смысл разделения?"
    Идея **checked**: «вызывающий обязан осознанно решить, что делать
    с восстановимой ошибкой» — компилятор заставляет обработать или пробросить
    (`IOException`, `SQLException`). **Unchecked** (`RuntimeException`) — ошибки
    программиста или невосстановимые ситуации (`NPE`, `IllegalArgument`,
    `IllegalState`), их объявлять не обязательно.

    На практике в современных приложениях и в Spring массово используют
    unchecked: checked замусоривают сигнатуры и всё равно часто просто
    оборачиваются. Ключевой навык — не «какое кинуть», а **не проглатывать**:
    пустой `catch` — худшее, что можно сделать, ошибка исчезает бесследно.

??? question "Почему обычно нельзя ловить Exception или Throwable общим catch?"
    `catch (Exception)` заодно ловит то, что ты не собирался обрабатывать,
    и маскирует реальные баги: `NPE`, который надо чинить, тихо превращается
    в «что-то пошло не так». `catch (Throwable)` ещё опаснее — ловит `Error`
    (`OutOfMemoryError`, `StackOverflowError`), после которых приложение обычно
    уже нельзя нормально продолжать, а ты делаешь вид, что справился.

    Ловить надо максимально конкретный тип и только там, где реально знаешь,
    что с ним делать. Глобальный «поймать всё и залогировать» уместен ровно
    на верхней границе — обработчик запроса, `@ControllerAdvice`.

??? question "Что произойдёт, если исключение вылетит из блока finally?"
    Оно **вытеснит** исключение из `try`: наружу улетит только исключение
    из `finally`, а исходное — настоящая причина падения — молча исчезнет.
    В логах будет какая-нибудь ошибка закрытия ресурса, а то, что до этого
    упал сам запрос к базе, не останется нигде. То же самое делает `return`
    в `finally` — он «проглатывает» любое исключение из `try`.

    Поэтому правило: в `finally` не бросать исключений и не писать `return`.

    Правильное решение для ресурсов — **try-with-resources**: если упали
    и тело, и `close()`, наружу летит основное исключение из тела, а ошибка
    закрытия прикрепляется к нему как suppressed — видна в стектрейсе
    (`Suppressed: ...`) и доступна через `getSuppressed()`. То есть
    try-with-resources не просто короче ручного `finally` — он ещё
    и не теряет информацию.

### Дженерики и стирание типов

??? question "Что такое стирание типов (type erasure) и что из-за него нельзя?"
    Дженерики существуют только на компиляции: компилятор проверяет типы
    и вставляет приведения, а в байткоде параметр типа **стирается** (до
    `Object` или до верхней границы). Поэтому в рантайме `List<String>`
    и `List<Integer>` — один и тот же класс `List`, а `list.getClass()`
    у обоих одинаков.

    Отсюда запреты: нельзя `new T()` (тип неизвестен в рантайме), нельзя
    `new T[]`, нельзя `obj instanceof List<String>`, нельзя две перегрузки,
    отличающиеся только параметром дженерика. Обходят это, передавая
    `Class<T>` явным аргументом.

??? question "Почему List<String> нельзя присвоить в List<Object>, хотя String — это Object?"
    Дженерики **инвариантны** — специально, ради безопасности типов. Если бы
    можно было `List<Object> l = stringList;`, то через `l.add(42)` в список
    строк попал бы `Integer`, и падение случилось бы позже и не там.

    Массивы, наоборот, ковариантны (`Object[] = String[]` разрешено) —
    историческое решение, и за него платят: `arr[0] = 42` компилируется,
    но кидает `ArrayStoreException` в рантайме. Для гибкости у дженериков есть
    wildcards: из `List<? extends Object>` читать можно, добавлять — нет.

??? question "Что такое PECS (? extends vs ? super)?"
    Producer Extends, Consumer Super.

    - Структура — **источник**, из которого только читаешь → `? extends T`
      (берёшь как `T`, но положить не можешь, кроме `null`).
    - Структура — **приёмник**, куда только кладёшь → `? super T` (кладёшь `T`,
      но при чтении получаешь `Object`).

    Классика — `Collections.copy(dest, src)`: `src` это `? extends T`
    (producer), `dest` это `? super T` (consumer). Правило даёт API гибкость
    по типам, не теряя безопасность.

### Неизменяемость, null и Optional

??? question "Ты сделал класс неизменяемым, но его состояние всё равно меняется. Как?"
    Почти наверняка через **мутабельный объект внутри**. `final` фиксирует
    только ссылку: поле `final List<String> tags` не мешает делать
    `tags.add(...)`. Утечка происходит двумя путями:

    - **Через конструктор**: класс сохранил переданную ссылку на список,
      а вызывающий код продолжает этот список менять — «неизменяемый» объект
      меняется снаружи.
    - **Через геттер**: `getTags()` вернул ссылку на внутренний список,
      и получатель дописал в него элементы.

    Решение — защитные копии: `List.copyOf(tags)` на входе в конструктор
    (заодно результат сам неизменяем, так что геттер может отдавать его
    спокойно), для массивов и изменяемых объектов — копия и на выходе.

    Второй, более редкий путь: класс не `final`, и кто-то унаследовался,
    добавив изменяемое состояние или переопределив методы. Поэтому канон
    неизменяемого класса: `final` класс, `final` поля, никаких сеттеров,
    защитные копии всего мутабельного. Или просто `record` — но копии
    мутабельных компонентов он за тебя не сделает.

??? question "Метод возвращает Optional, а код всё равно падает с NPE. Каким образом?"
    Основных сценария три:

    - **Метод вернул `null` вместо `Optional`.** Тип в сигнатуре — не гарантия:
      если внутри написано `return null;`, вызывающий код упадёт с NPE на самом
      Optional (`opt.map(...)` по null-ссылке). Метод, возвращающий `Optional`,
      не должен возвращать `null` никогда — это нарушение контракта.
    - **`orElse(null)`** — Optional честно развернули обратно в `null`,
      и дальше по коду он взорвался как обычно. Optional спасает, только пока
      цепочка не заканчивается возвратом null.
    - **Значение внутри есть, но его поле — `null`**: `opt.get().getName().length()`
      падает не на Optional, а на `getName()`. Optional защищает ровно один
      уровень — само значение, не его внутренности.

    Бонусная деталь: `Optional.of(null)` кидает NPE сразу при создании —
    для потенциально пустых значений нужен `Optional.ofNullable`. А вот
    `opt.get()` на пустом Optional кидает `NoSuchElementException`, не NPE, —
    хороший маркер, что человек реально с этим работал.

??? question "Как правильно и неправильно использовать Optional?"
    Optional придуман для одной задачи — **возвращаемое значение метода**,
    которого может не быть (`findById`). Правильно: вернуть `Optional<User>`
    и заставить вызывающего осознанно обработать пустоту через
    `map`/`orElse`/`orElseThrow`.

    Неправильно: поля-Optional в сущностях (лишняя обёртка, не сериализуется,
    ломает JPA), параметры-Optional (проще перегрузка или `@Nullable`),
    `Optional` в коллекциях. И антипаттерн `opt.isPresent()` + `opt.get()` —
    это тот же null-чек, только многословнее; смысл Optional в цепочке
    `map().filter().orElse()`.

??? question "Откуда на самом деле берётся NPE и как защищаться системно?"
    NPE — это вызов метода или обращение к полю по `null`-ссылке. Системная
    защита не в том, чтобы завешать код `if (x != null)`, а в дисциплине:

    - **не возвращать `null`** из методов — пустая коллекция, `Optional`,
      либо исключение;
    - **проверять аргументы на границе**: `Objects.requireNonNull(arg, "msg")`
      падает сразу и с внятным сообщением, а не через три вызова дальше;
    - использовать неизменяемые объекты и аннотации `@Nullable`/`@NonNull`.

    Хуже всего — «защитно» проглотить `null` и вернуть его дальше: тогда NPE
    вылезет далеко от причины, и искать будешь долго.

### Коллекции

??? question "Как HashMap устроен внутри и что происходит при коллизии?"
    Массив бакетов (по умолчанию 16). При `put`: берётся `hashCode` ключа,
    дополнительно перемешивается (чтобы старшие биты влияли на индекс), индекс
    считается как `hash & (n-1)`. Если бакет пуст — кладётся запись; если занят
    (коллизия) — элементы висят списком, новый ключ сравнивается с имеющимися
    через `equals`: равный — перезапись значения, иначе добавление в цепочку.

    С Java 8 длинная цепочка (≥8 элементов при таблице ≥64) перестраивается
    в **красно-чёрное дерево** → поиск O(log n) вместо O(n). При заполнении
    сверх load factor таблица удваивается, и все записи перераспределяются
    (resize) — дорогая операция.

??? question "Почему load factor по умолчанию 0.75?"
    Компромисс память/скорость. Плотнее таблица → больше коллизий и длиннее
    цепочки (медленнее); разреженнее → больше пустой памяти и чаще resize.
    0.75 эмпирически балансирует: при равномерном хеше коллизий ещё мало,
    а память используется прилично. Если заранее знаешь размер — задай
    начальную ёмкость, чтобы избежать многократных resize при наполнении.

??? question "ArrayList или LinkedList — что когда, и почему LinkedList почти всегда проигрывает?"
    `ArrayList` — массив: доступ по индексу O(1), добавление в конец
    амортизированно O(1), вставка/удаление в середину O(n) со сдвигом.
    `LinkedList` — двусвязный список: вставка/удаление O(1), **если у тебя уже
    есть итератор на месте**, но доступ по индексу O(n).

    На практике `ArrayList` почти всегда быстрее даже там, где теория обещает
    преимущество списку: массив лежит в памяти подряд (cache locality, префетч
    процессора), а список — разбросанные узлы с указателями, каждый переход —
    промах кеша плюс накладные на объект-узел. `LinkedList` оправдан редко,
    в основном как `Deque`/очередь.

??? question "Что сломается, если использовать HashMap из нескольких потоков?"
    Всё что угодно, и это худший вариант — не падение, а тихая порча данных.

    Конкретно: записи теряются, потому что два потока одновременно обновляют
    один бакет; счётчик `size` перестаёт соответствовать реальности; при
    одновременном расширении структура повреждается, и последующие чтения
    возвращают неверные результаты. В Java 7 такая гонка при расширении могла
    привести к зацикливанию на чтении; в Java 8 перестройку переписали,
    и этот конкретный сценарий уже не воспроизводится, но повреждение
    данных никуда не делось.

    Что использовать: `ConcurrentHashMap`. Он не блокирует всю карту целиком,
    поэтому параллельное чтение и запись в разные части работают быстро.
    `Collections.synchronizedMap()` тоже даёт потокобезопасность, но через
    один общий монитор, и на нагрузке это узкое место. Плюс важная деталь:
    у обёртки итерирование всё равно нужно синхронизировать вручную,
    а составные операции («проверил и положил») не атомарны — для этого
    у `ConcurrentHashMap` есть `putIfAbsent` и `compute`.

??? question "Положил объект в HashSet и изменил его поле. Что теперь с этим объектом?"
    Если поле участвует в `hashCode` — объект «потерялся». `HashSet` положил
    его в бакет по **старому** хешу, а искать теперь будет по **новому**:

    - `contains(obj)` возвращает `false`, хотя объект в коллекции есть;
    - `remove(obj)` не удаляет его — объект навсегда завис в сете
      (готовая утечка памяти, если так делать систематически);
    - повторный `add(obj)` добавит **дубликат** — тот же объект окажется
      в сете дважды, в разных бакетах;
    - при итерировании объект по-прежнему виден — что и сбивает с толку
      при отладке: «вот же он, но contains говорит false».

    Причина: хеш-структуры вычисляют позицию один раз, при вставке, и не
    узнают об изменении объекта. Вывод, который отсюда ждут: ключи `HashMap`
    и элементы `HashSet` должны быть неизменяемыми — или как минимум
    не менять поля, входящие в `equals`/`hashCode`, пока объект лежит
    в коллекции.

??? question "Почему `for (X x : list) list.remove(x)` кидает ConcurrentModificationException без всяких потоков?"
    Это **fail-fast**, а не про многопоточность. Коллекция хранит счётчик
    модификаций `modCount`; итератор (за for-each стоит именно он) запоминает
    его в начале и на каждом шаге сверяет. Прямой `list.remove()` меняет
    `modCount` мимо итератора — на следующем `next()` числа расходятся,
    и итератор бросает CME, чтобы не работать с испорченным состоянием.

    Правильно удалять через `iterator.remove()` или `list.removeIf(...)`.
    Название «Concurrent» вводит в заблуждение — чаще всего это однопоточная
    ошибка.

### Stream API

??? question "Стрим отработал без ошибок, а результата нет. Что не так?"
    Скорее всего, стрим **не отработал вовсе**. Промежуточные операции
    (`map`, `filter`, `peek`) ленивы — они только строят конвейер. Выполняет
    его **терминальная** операция (`toList`, `count`, `forEach`), и если её
    нет, не выполнится ни одна лямбда:

    ```java
    orders.stream().map(o -> { audit(o); return o; }); // audit не вызовется
    ```

    Родственные варианты той же проблемы:

    - Побочный эффект спрятан в `peek`, а терминальная операция сумела
      обойтись без обхода элементов (например, `count()` у стрима без
      фильтров) — `peek` пропущен.
    - Результат терминальной операции **не присвоен**: `stream.toList()`
      создаёт новый список, исходная коллекция не меняется — ждали мутацию
      исходника, а её у стримов не бывает.
    - Стрим уже был использован: вторая терминальная операция кидает
      `IllegalStateException` — но это как раз «с ошибкой», легче заметить.

    Вывод, который проверяет вопрос: стрим — ленивое описание конвейера,
    а не цикл; побочные эффекты в промежуточных операциях — ошибка дизайна.

??? question "Стрим обходит элементы «слоями» или каждый целиком через весь конвейер?"
    Каждый элемент проходит **весь конвейер до конца**, потом берётся
    следующий (вертикальная обработка), а не «сначала все `map`, потом все
    `filter`». Это следствие ленивости и даёт короткое замыкание: `findFirst`,
    `anyMatch`, `limit` останавливают обход, как только результат найден, —
    остальные элементы не читаются вовсе.

    Поэтому работает бесконечный `Stream.iterate(...).filter(...).findFirst()`.
    Практический вывод: тяжёлый `map` не выполнится для элементов, отсеянных
    ранним `limit` или найденным `findFirst`.

??? question "Когда parallelStream() навредит?"
    Часто. Он раскидывает работу по общему `ForkJoinPool.commonPool` (один на
    JVM) — если несколько мест разом используют `parallelStream`, они дерутся
    за те же потоки, а одна долгая блокирующая задача (I/O, поход в БД) забивает
    пул для всех. Выгода есть только при большом объёме, дешёвом разбиении
    (`ArrayList` — ок, `LinkedList` — нет) и CPU-bound операции.

    Плюс порядок: `parallel` + `forEach` не гарантирует порядок (нужен
    `forEachOrdered`, что съедает выигрыш), а мутация общего состояния из лямбды
    даёт гонки. Правило: по умолчанию последовательный стрим; parallel — только
    измерив, на CPU-bound и без общего состояния.

### Многопоточность

??? question "Два потока увеличивают счётчик по 1000 раз каждый. Почему в конце не 2000?"
    Потому что `count++` — не одна операция, а три: прочитать значение,
    прибавить единицу, записать обратно. Потоки перемешиваются между этими
    шагами: оба прочитали 5, оба записали 6 — один инкремент потерян.
    Это классическое состояние гонки (read-modify-write), и результат каждый
    запуск разный — примерно от 1000 до 2000.

    Варианты починки, от лучшего к худшему для этого случая:

    - **`AtomicInteger.incrementAndGet()`** — атомарный инкремент через CAS,
      без блокировок; правильный ответ для счётчика.
    - `synchronized` вокруг инкремента — тоже корректно, но потоки будут
      ждать друг друга.
    - `LongAdder` — если счётчик под очень высокой конкуренцией.

    Уточнение, которое любят проверять следом: `volatile int count` проблему
    **не решит** — volatile даёт только видимость, а `count++` остаётся тремя
    операциями, и гонка никуда не денется.

??? question "Ты поставил synchronized, а гонка осталась. Как это возможно?"
    `synchronized` защищает только тех, кто входит через **тот же монитор**
    и только **то, что внутри блока**. Отсюда все сценарии:

    - **Разные мониторы.** Потоки синхронизируются на разных объектах —
      например, метод synchronized (монитор `this`), а объектов несколько.
      Или смешаны статический и обычный synchronized-методы: у них разные
      мониторы (`Класс.class` и `this`), друг другу они не мешают.
    - **Защищены не все точки доступа.** Запись под замком, а чтение — без:
      читающий поток спокойно влезает в середину «защищённой» операции.
      Синхронизировать нужно **все** обращения к общему состоянию.
    - **Составная операция шире блока.** Атомарны два вызова по отдельности,
      а сценарий «проверил — сделал» — нет: между `synchronized`-проверкой
      и `synchronized`-записью вклинивается другой поток. Замок должен
      охватывать всю логическую операцию.
    - Экзотика: синхронизация на изменяемом поле (`synchronized (lock)`,
      а `lock` кто-то переприсвоил) — потоки оказываются на разных мониторах.

    Общий вывод: synchronized — не свойство данных, а соглашение между
    потоками; работает, только если все стороны его соблюдают на одном
    и том же мониторе.

??? question "Один поток изменил поле, другой этого не увидел. Почему?"
    Это проблема **видимости**. Без синхронизации JVM не обязана доставлять
    изменения из одного потока в другой: значение может жить в регистре или
    кэше ядра процессора, а JIT вправе вынести чтение поля за пределы цикла —
    «в этом потоке его же никто не меняет». Классика — цикл
    `while (!stopped) { ... }`, который не останавливается, хотя другой поток
    давно сделал `stopped = true`.

    Гарантию видимости даёт только отношение **happens-before** между записью
    и чтением. Способы его получить:

    - `volatile` на поле — запись немедленно видна всем читателям;
      идеален для флагов и статусов;
    - `synchronized`: выход из блока happens-before входа в блок
      по тому же монитору;
    - механизмы `java.util.concurrent` — отправка в `ExecutorService`,
      `BlockingQueue`, `CountDownLatch` уже несут happens-before;
    - `start()`/`join()` потока.

    Важная граница: volatile решает видимость, но **не атомарность** —
    для `count++` нужен атомик или замок. И наоборот: раз гонки «не
    воспроизводятся на моей машине», это не значит, что их нет, — видимость
    ломается недетерминированно, под нагрузкой и после JIT-компиляции.

??? question "Что именно гарантирует volatile, а что нет?"
    Гарантирует две вещи:

    - **видимость** — запись в volatile-поле немедленно видна всем читающим
      потокам, чтения не кешируются;
    - **запрет переупорядочивания** вокруг обращения к нему (happens-before),
      в частности безопасную публикацию объекта, ссылку на который положили
      в volatile-поле.

    Не гарантирует **атомарности** составных операций: `volatile count++` всё
    равно теряет инкременты, потому что это read-modify-write. Вывод: volatile —
    для флагов и одиночной публикации ссылок; для счётчиков — атомики,
    для составной логики — блокировки.

??? question "synchronized или ReentrantLock — в чём разница и когда второе?"
    `synchronized` проще и достаточен в большинстве случаев: JVM сама
    освобождает монитор на выходе, в том числе при исключении. `ReentrantLock`
    даёт то, чего у `synchronized` нет:

    - `tryLock()`, в том числе с таймаутом — не виснуть навсегда;
    - прерываемое ожидание (`lockInterruptibly`);
    - справедливость (fairness — FIFO вместо гонки за монитор);
    - несколько `Condition` вместо одного wait-set.

    Цена — надо руками `unlock()` в `finally`, иначе вечная блокировка. Берут
    Lock, когда нужен таймаут/прерывание/tryLock; иначе `synchronized`.

??? question "Как AtomicInteger обходится без блокировок? Что такое CAS и проблема ABA?"
    Через **CAS** (compare-and-swap) — атомарную инструкцию процессора «сравни
    значение по адресу с ожидаемым и, если совпало, запиши новое».
    `incrementAndGet` крутит цикл: прочитал текущее `v`, посчитал `v+1`, сделал
    `CAS(ожидаю v, ставлю v+1)`; если другой поток успел изменить — CAS не
    прошёл, читаем заново и повторяем. Это оптимистичный подход: никто не ждёт
    на мониторе, потоки соревнуются и проигравший повторяет.

    Проблема **ABA**: значение было A, стало B и снова A — CAS думает, что
    ничего не менялось, хотя между этим что-то произошло; для ссылок лечится
    версией/меткой (`AtomicStampedReference`). Под очень высокой конкуренцией
    CAS-цикл крутится вхолостую — тогда берут `LongAdder`.

??? question "Что такое deadlock, какие условия нужны и как избегать?"
    Взаимная блокировка: поток A держит замок 1 и ждёт замок 2, поток B держит
    2 и ждёт 1 — оба стоят вечно. Нужны одновременно четыре условия (Коффмана):
    взаимное исключение, удержание с ожиданием, отсутствие вытеснения, круговое
    ожидание.

    Ломают обычно последнее — **глобальный порядок захвата** замков (все берут
    сначала 1, потом 2). Другие приёмы: `tryLock` с таймаутом (отступить
    и повторить), не держать замок при вызове чужого кода, уменьшать число
    замков. Диагностика в проде — thread dump (`jstack`): JVM прямо пишет
    «Found deadlock» и какие потоки кого ждут.

??? question "Что такое ThreadLocal, как устроен и чем опасен в пуле потоков?"
    Хранилище «своё значение на каждый поток»: у каждого `Thread` есть карта
    `ThreadLocalMap`, ключ — сам объект `ThreadLocal`, значение — данные этого
    потока. Используют для передачи контекста без параметров: `SecurityContext`
    в Spring Security, транзакция/`EntityManager`, `MDC` для логов.

    Опасность в пуле: поток не умирает, а переиспользуется, поэтому значение
    от прошлого запроса останется и «протечёт» в следующий — в лучшем случае
    мусор, в худшем чужие данные (дыра в безопасности). Плюс утечка памяти,
    если значение крупное. Правило: положил в `ThreadLocal` на входе —
    обязательно `remove()` в `finally` на выходе.

??? question "Почему нельзя создавать по потоку на каждый запрос и как работает пул потоков?"
    Поток ОС дорогой: ~1 МБ стека, накладные на создание и переключение
    контекста; тысячи потоков — и память кончилась, а планировщик захлёбывается.
    Пул держит фиксированное число рабочих потоков и очередь задач: задача
    попадает в очередь, свободный поток её забирает.

    У `ThreadPoolExecutor`: `corePoolSize` (базовые потоки), `maxPoolSize`
    (до скольких растёт), очередь и политика при переполнении
    (`RejectedExecutionHandler`). Ключевое для стабильности прода: **очередь
    должна быть ограниченной** — иначе при перегрузке она растёт до OOM. Лучше
    явно отказывать (backpressure), чем тихо копить необработанные задачи.

??? question "Чем CompletableFuture отличается от Future и зачем он?"
    `Future` умеет только блокирующе ждать результат (`get()`) и проверять
    готовность — композировать асинхронные шаги нельзя. `CompletableFuture` —
    это `Future` плюс декларативная цепочка без блокировки: `thenApply`
    (преобразовать результат), `thenCompose` (связать с другим async-шагом),
    `thenCombine` (объединить два), `exceptionally` (обработать ошибку).

    Позволяет строить конвейеры и запускать вызовы параллельно, собирая
    результаты. Важно: по умолчанию использует `commonPool` — для блокирующих
    операций (походы в БД/сеть) передавай свой `Executor`, иначе забьёшь общий
    пул и затормозишь всё приложение.

### JVM, память и сборка мусора

??? question "Как устроена память JVM — что где лежит?"
    Основное — **куча** (heap): все объекты и массивы, общая для всех потоков,
    её и убирает GC; делится на молодое поколение (Eden + два Survivor)
    и старое. У каждого потока свой **стек**: кадры вызовов, локальные
    переменные и ссылки (сами объекты — в куче, в стеке только ссылки).

    **Metaspace** (вне кучи, в нативной памяти) — метаданные классов, код
    методов; заменил PermGen в Java 8. Плюс off-heap буферы
    (`DirectByteBuffer`). Ключевое: объект всегда в куче; локальная
    переменная-примитив и ссылка — в стеке; `static`-поля держат объекты
    живыми через класс.

??? question "Как работает сборка мусора и зачем поколения?"
    GC удаляет объекты, до которых нет пути от **GC roots** (стеки потоков,
    `static`-поля, JNI). Базовый алгоритм — mark-sweep(-compact): пометить
    достижимые, освободить остальное, при необходимости уплотнить.

    Поколения — оптимизация под наблюдение «большинство объектов умирает
    молодыми»: новые создаются в Eden, частый и быстрый **minor GC** собирает
    молодое поколение (копируя выживших в Survivor, затем в старое), а дорогой
    **major/full GC** по старому поколению случается редко. Так основную работу
    делает дешёвый minor GC. Циклические ссылки сборке не мешают — GC идёт
    от корней, а не считает ссылки.

??? question "Что такое stop-the-world и почему это боль в проде?"
    Момент, когда GC (или другая VM-операция) останавливает **все** потоки
    приложения, чтобы согласованно пройтись по объектам. Короткие паузы minor
    GC незаметны, но full GC на большой куче может встать на сотни мс — секунды,
    и всё это время сервис не отвечает: растут задержки, вышестоящие вызовы
    отваливаются по таймауту, health-check может решить, что под мёртв,
    и перезапустить его.

    Поэтому современные сборщики (G1, ZGC, Shenandoah) минимизируют паузы,
    работая конкурентно с приложением. Диагностика — GC-логи: если после
    каждого full GC занятая память не падает — утечка; если паузы длинные
    и частые — не хватает памяти либо неудачный сборщик/настройки.

??? question "Ты обнулил ссылку, а объект не собрался сборщиком мусора. Где он держится?"
    GC удаляет объект, только когда до него **нет ни одного пути от GC roots**
    (стеки потоков, статические поля). Обнуление одной ссылки ничего не значит,
    если жива хоть одна другая. Где искать держателя:

    - **Статическая коллекция или кэш** — объект положили в `static Map`
      и не удалили; самый частый случай.
    - **Коллекция вообще** — объект всё ещё лежит в списке/сете, о котором
      забыли (включая «потерянный» элемент HashSet с изменённым хешем —
      его даже `remove` не достаёт).
    - **`ThreadLocal` в пуле потоков** — поток живёт вечно, его
      ThreadLocal-значения тоже, пока не вызван `remove()`.
    - **Слушатель/подписка** — объект зарегистрировали как колбэк
      и не отписали.
    - **Внутренний класс** — не-static вложенный класс (и анонимный) держит
      скрытую ссылку на внешний объект.

    Как найти на практике: heap dump (`jcmd GC.heap_dump` или
    `-XX:+HeapDumpOnOutOfMemoryError`) и анализатор (MAT) — он показывает
    путь от объекта до GC roots, то есть буквально «кто держит».

    Дополнительный балл: циклические ссылки объект живым **не делают** —
    GC работает от корней, а не подсчётом ссылок.

??? question "Приложение падает по OutOfMemoryError, но явной утечки в коде нет. Где искать?"
    Сначала — факты, потом гипотезы. Порядок разбора:

    1. **Какой именно OOM?** `Java heap space` — куча; `Metaspace` —
       генерация классов (прокси, скрипты); `unable to create native thread` —
       не память вовсе, а лимит потоков ОС. Лечатся они по-разному.
    2. **Динамика кучи по метрикам**: если занятая память после каждой сборки
       монотонно растёт — всё-таки утечка, какой бы «неявной» она ни была.
       Если пилообразный график упирается в потолок на пиках — утечки нет,
       приложению честно не хватает памяти.
    3. **Heap dump** (`-XX:+HeapDumpOnOutOfMemoryError` — чтобы снялся сам
       в момент падения) и анализатор: что занимает кучу и кто держит.

    Частые «не-утечки», которые находят на шаге 3:

    - **Большие батчи**: вычитали из БД миллион строк разом
      (`findAll()` без пагинации), собрали гигантский список/отчёт в памяти.
    - **Кэши без ограничений** — работают как задумано, но растут с данными.
    - **Убавили память контейнеру**, а `-Xmx` остался прежним, или наоборот —
      куче отдали весь лимит контейнера, не оставив на стеки и off-heap.
    - Рост нагрузки: больше потоков × больше данных на запрос — пиковое
      потребление выросло легально.

    Решения соответственно: пагинация и стриминг вместо «всё в память»,
    лимиты кэшам, честный пересчёт `-Xmx` под лимит контейнера.

??? question "Что такое JIT и почему приложение «разогревается»?"
    JVM сначала **интерпретирует** байткод (медленно, но сразу), а горячие
    методы (часто вызываемые) на лету компилирует в машинный код — это **JIT**
    (C1 быстро и грубо, C2 медленно и оптимально), с агрессивными оптимизациями:
    инлайнинг, устранение проверок, спекуляции.

    Поэтому первые запросы после старта медленнее, а через какое-то время под
    нагрузкой сервис ускоряется — это warmup. Следствия: бенчмаркать без
    прогрева бессмысленно (нужен JMH); в проде холодный старт после деплоя даёт
    временный всплеск задержек — это учитывают в canary-выкатке и прогреве
    трафиком.

??? question "Приложение стартует 40 секунд вместо пяти. С чего начнёшь разбираться?"
    Главный принцип — **измерить, а не угадывать**: понять, на каком этапе
    теряется время. Смотрим логи старта с таймстампами: где между строками
    дыра в десятки секунд — там и проблема.

    Что чаще всего оказывается причиной (по убыванию вероятности):

    - **Ожидание внешних систем**: приложение при старте ходит в БД, Kafka,
      конфиг-сервис — а те недоступны или медленны, и время уходит
      на таймауты и ретраи. В логах это видно как паузы вокруг подключений.
    - **Миграции БД**: Flyway/Liquibase на старте гоняет тяжёлую миграцию
      или ждёт блокировку таблицы.
    - **Инициализация Spring-контекста**: тысячи бинов, «тяжёлые»
      конструкторы/`@PostConstruct`, которые сами что-то грузят или считают.
      В Spring Boot это раскладывается по шагам актуатором `/actuator/startup`
      (ApplicationStartup) — видно, какие бины сколько создавались.
    - **Прогрев не по адресу**: кто-то на старте жадно грузит кэш/справочники
      целиком, хотя можно лениво или в фоне.
    - Медленный DNS/сеть/диск в конкретном окружении — если на локальной
      машине быстро, а в проде медленно, разница в окружении, а не в коде.

    Хороший ответ отличает уровень: не «увеличу таймаут», а «найду этап,
    пойму, обязателен ли он на старте, и либо ускорю, либо вынесу из старта
    в фон/лень». Отдельный флаг зрелости — упомянуть, что kubernetes
    с жёстким startup probe может убивать медленно стартующее приложение,
    и лимиты проб надо согласовывать с реальным временем старта.

## Языки и платформы

### Kotlin

??? question "В Kotlin есть null safety, но NPE всё равно прилетел. Откуда?"
    Null safety гарантируется компилятором и только **внутри мира Kotlin**.
    Дыры — на границах:

    - **Java-интероп.** Типы из Java-кода — платформенные (`String!`):
      компилятор не знает, бывает ли там `null`, и пропускает обращение без
      проверки. Java-метод вернул `null` — получили классический NPE.
      Лечится аннотациями `@Nullable`/`@NotNull` на Java-стороне.
    - **Оператор `!!`** — явное «уверен, что не null»; если ошибся — NPE.
    - **`lateinit var`** — обращение до инициализации.
    - **Рефлексия и десериализация** — Jackson без Kotlin-модуля или JPA
      могут записать `null` в non-null поле в обход компилятора.

    Хороший ответ: null safety — это проверка компиляции, а не рантайма,
    поэтому всё, что обходит компилятор (Java, рефлексия, `!!`), обходит
    и защиту.

??? question "Что сломается, если сделать JPA-сущность data-классом Kotlin?"
    Формально заработает (с плагинами), но проблем несколько:

    - **Нет конструктора без аргументов**, который требует JPA, — нужен
      плагин `kotlin-jpa`, генерирующий его.
    - **Kotlin-классы `final`**, а Hibernate строит прокси-наследников для
      ленивой загрузки — нужен `kotlin-allopen`.
    - Главное: **сгенерированный `equals`/`hashCode` считается по всем
      полям**, включая `id`, который появляется только при сохранении.
      Объект «меняет хэш» после `persist` — ломается поиск в `HashSet`,
      сравнение до/после сохранения даёт `false`.
    - `equals`/`toString` по всем полям **трогают ленивые связи** — случайная
      инициализация lazy-коллекций или `LazyInitializationException`.

    Идиома: сущности — обычные классы с `equals` по `id`, data-классы —
    для DTO, где авто-`equals` и `copy` как раз уместны.

??? question "Чем корутины отличаются от потоков?"
    Корутина — не поток, а способ **приостанавливать выполнение, не блокируя
    поток**. Тысячи корутин работают на небольшом пуле потоков: дойдя до
    suspend-точки (сетевой вызов, `delay`), корутина освобождает поток, и на нём
    выполняется другая, а не висит заблокированной. Поток ОС дорогой (~1 МБ
    стека), корутина дешёвая (объект в куче).

    Отсюда выгода: держать десятки тысяч одновременных «ожиданий» без десятков
    тысяч потоков. Ключевое: корутины **не ускоряют CPU-работу**, они экономят
    потоки на I/O-ожидании. Под капотом компилятор превращает suspend-функцию
    в state machine с продолжением (continuation); вызвать её можно только
    из другой корутины или suspend-функции.

??? question "Почему в Kotlin классы и методы final по умолчанию?"
    Осознанное решение против «хрупкого базового класса»: наследование, которое
    не спроектировали явно, легко ломается при изменении родителя. Чтобы
    разрешить переопределение, надо явно написать `open` — это заставляет автора
    решить, что класс действительно предназначен для расширения.

    Побочный эффект бьёт по Spring/Hibernate: они строят прокси-наследников,
    а `final`-класс не наследуется. Поэтому для Spring-бинов с `@Transactional`
    и JPA-сущностей нужен плагин `kotlin-allopen` (в spring-boot он уже
    подключён), который делает нужные классы `open`.

### Quarkus

??? question "Почему Quarkus стартует за десятки миллисекунд, а Spring Boot за секунды?"
    Потому что они делают работу фреймворка **в разное время**. Spring Boot
    на каждом старте сканирует classpath, разбирает аннотации, строит
    DI-контекст и прокси через рефлексию — это и есть секунды старта.
    Quarkus выполняет всё это **один раз на этапе сборки**: в артефакте
    лежит уже сгенерированный «прошитый» код, и на старте его остаётся
    только запустить.

    Побочные эффекты того же решения: меньше памяти (рефлексивные метаданные
    не нужны в рантайме) и первоклассная нативная компиляция GraalVM
    (старт за миллисекунды), которая как раз требует отказа от рефлексии
    на старте.

    Стоит добавить честное «но»: для долгоживущего сервиса секунды старта
    обычно не проблема, поэтому само по себе это не причина мигрировать —
    выигрыш важен в serverless и плотных контейнерных окружениях.

## Spring

### Spring Core: IoC, бины, контекст

??? question "Что такое IoC и внедрение зависимостей на самом деле — что делает контейнер?"
    IoC (инверсия управления) — объект не создаёт свои зависимости сам через
    `new`, а получает их извне. Контейнер Spring на старте: сканирует классы,
    строит карту **определений бинов** (`BeanDefinition` — как создать бин, какие
    зависимости, скоуп), затем создаёт бины и связывает их (autowiring).
    «Инверсия» в том, что жизненным циклом и связыванием управляет контейнер,
    а не твой код.

    Что это даёт: код зависит от интерфейсов, а не от конкретных реализаций →
    легко подменять (тесты, окружения), единая точка конфигурации, управление
    скоупами и жизненным циклом. По сути контейнер — большая фабрика с графом
    зависимостей.

??? question "Опиши жизненный цикл бина от создания до уничтожения."
    Этапы:

    1. контейнер создаёт экземпляр (конструктор);
    2. внедряет зависимости (поля/сеттеры);
    3. вызывает `Aware`-интерфейсы (`BeanNameAware` и т.п.);
    4. `BeanPostProcessor.postProcessBeforeInitialization`;
    5. инициализация — `@PostConstruct`, затем `afterPropertiesSet`, затем
       init-method;
    6. `BeanPostProcessor.postProcessAfterInitialization` — **здесь создаются
       AOP-прокси** (бин оборачивается);
    7. бин готов и живёт;
    8. при закрытии контекста — `@PreDestroy`, `DisposableBean.destroy`.

    Ключевое понимание: прокси (`@Transactional` и др.) навешивается на шаге 6,
    поэтому в контейнере лежит не твой объект, а обёртка вокруг него.

??? question "Что такое BeanPostProcessor и почему это «сердце магии» Spring?"
    Это хук, через который контейнер даёт вмешаться в создание **каждого** бина
    — два метода, before и after инициализации. Именно через
    `BeanPostProcessor` реализованы почти все аннотации: автопрокси-креатор
    оборачивает бины в AOP-прокси (`@Transactional`, `@Async`),
    `AutowiredAnnotationBeanPostProcessor` обрабатывает `@Autowired`,
    `CommonAnnotationBeanPostProcessor` — `@PostConstruct`.

    Понимание этого снимает «магию»: Spring не делает ничего сверхъестественного,
    он последовательно прогоняет каждый бин через набор постпроцессоров.

??? question "Конструкторное или полевое внедрение — что и почему?"
    **Конструкторное** (через параметры конструктора) — рекомендуемое. Плюсы:
    зависимости видны и обязательны (нельзя создать полусобранный объект), поле
    можно сделать `final` (неизменяемо, потокобезопасно), класс легко
    инстанцировать в тесте обычным `new` без Spring, циклы вылезают сразу
    на старте как ошибка — это хорошо, сигнал о дизайне.

    **Полевое** (`@Autowired` на поле) — короче, но скрывает зависимости, нельзя
    `final`, тестировать труднее (нужна рефлексия/контекст), маскирует циклы.
    Сеттерное — для опциональных зависимостей. Правило: по умолчанию
    конструкторное.

??? question "Ты внедрил prototype-бин в синглтон. Сколько экземпляров будет создано?"
    **Один.** Внедрение происходит один раз — при создании синглтона:
    контейнер создал prototype-экземпляр, положил в поле, и дальше синглтон
    всегда работает с ним. «Новый экземпляр на каждое обращение» — это про
    обращения **к контейнеру**, а поле синглтона к контейнеру больше
    не обращается.

    Если нужен свежий экземпляр на каждый вызов: внедрить
    `ObjectProvider<T>` и звать `getObject()` в момент использования,
    либо `@Lookup`-метод, либо scoped proxy. И стоит спросить себя, нужен ли
    тут вообще бин — объект без зависимостей из контейнера честнее создавать
    обычным `new`.

??? question "Два бина одного типа в контексте. Что произойдёт при старте?"
    Пока никто не внедряет этот тип — ничего: два бина спокойно живут
    в контексте. Ошибка случится при внедрении по типу:
    `NoUniqueBeanDefinitionException` — контейнер нашёл двух кандидатов
    и не стал угадывать (у Spring Boot будет понятное сообщение
    «expected single matching bean but found 2»).

    Разрешение: `@Qualifier` — явно указать нужный бин; `@Primary` —
    назначить кандидата по умолчанию; или внедрить **всех** —
    `List<T>` / `Map<String, T>`, стандартная идиома «все реализации
    стратегии» (обработчики, валидаторы).

??? question "Два сервиса ссылаются друг на друга. Что сделает Spring и что делать тебе?"
    Spring упадёт на старте с `BeanCurrentlyInCreationException` и нарисует
    цикл: с конструкторным внедрением создать оба бина невозможно — каждому
    нужен готовый другой. Со Spring Boot 2.6 циклы запрещены по умолчанию
    для любого вида внедрения.

    Что делать — в порядке правильности:

    1. **Починить дизайн** — цикл почти всегда означает, что общая
       ответственность двух классов должна жить в третьем, от которого
       зависят оба. Или заменить один из вызовов событием
       (`ApplicationEventPublisher`).
    2. `@Lazy` на одной из зависимостей — внедрится прокси, бин создастся
       при первом вызове. Цикл разомкнут, дизайн-проблема осталась.
    3. `spring.main.allow-circular-references=true` — костыль для легаси.

    На интервью главное — сказать, что цикл это сигнал о дизайне,
    а не техническая неприятность.

??? question "@Configuration или @Component: почему @Bean-метод, вызванный дважды, возвращает один и тот же бин?"
    Потому что `@Configuration`-класс сам оборачивается в **CGLIB-прокси**.
    Когда внутри одного `@Bean`-метода ты вызываешь другой `@Bean`-метод, вызов
    перехватывается прокси и возвращает уже созданный синглтон из контейнера,
    а не выполняет метод заново (это называется full mode).

    Если тот же метод объявить в `@Component` (lite mode) — прокси нет, прямой
    вызов создаст **новый** объект, и синглтон-семантика сломается. Поэтому
    конфигурации с межбиновыми вызовами должны быть `@Configuration`.

??? question "Бины по умолчанию синглтоны. Чем это опасно?"
    Один экземпляр на весь контекст шарится всеми потоками, обрабатывающими
    запросы. Значит, изменяемое **поле-состояние** в бине (счётчик, кэш,
    «текущий пользователь») — это разделяемое состояние без синхронизации →
    гонки и утечка данных между запросами.

    Правило: бины (сервисы, контроллеры) должны быть **stateless**; всё
    состояние запроса — в локальных переменных и параметрах, а не в полях.
    Нужно состояние на запрос — request-скоуп или `ThreadLocal` (с очисткой).
    Это прямое следствие синглтон-скоупа, которое часто забывают.

### AOP и прокси

??? question "Как технически работает AOP-прокси и какие бывают?"
    Spring создаёт вокруг бина объект-**обёртку**, которая перехватывает вызовы
    методов, выполняет «совет» (открыть транзакцию, проверить право, уйти в пул)
    и делегирует настоящему бину. Два вида:

    - **JDK dynamic proxy** — если бин реализует интерфейс: создаётся прокси,
      реализующий тот же интерфейс (проксируются только методы интерфейса);
    - **CGLIB** — если интерфейса нет: генерируется подкласс бина в рантайме
      (поэтому класс и методы не должны быть `final`/`private`).

    Spring Boot по умолчанию использует CGLIB. Главное следствие: в контейнере
    лежит **прокси**, а не твой объект, и перехватываются только вызовы, идущие
    через прокси (снаружи). Отсюда все грабли с самовызовом.

??? question "@Async не выполняется асинхронно. Что общего у этой проблемы с предыдущей?"
    Общее — **механизм AOP-прокси**. `@Async`, как и `@Transactional`,
    работает через обёртку вокруг бина: прокси перехватывает вызов и уводит
    его в пул потоков. Значит, и ломается он так же:

    - **самовызов** — метод того же класса вызван через `this`, мимо прокси;
    - метод `private`/`final`/`static` — прокси не может перехватить;
    - объект создан через `new`, а не контейнером.

    Специфика самого `@Async`: нужен `@EnableAsync` (без него аннотация
    просто игнорируется), а без настроенного executor'а Boot использует
    дефолтный пул — стоит объявить свой с понятными лимитами. И помнить:
    `@Async`-метод выполняется в другом потоке — **вне транзакции
    вызывающего** и без его `ThreadLocal`-контекста (SecurityContext, MDC).

    Красивый ответ формулирует правило: все «магические» аннотации Spring
    (`@Transactional`, `@Async`, `@Cacheable`, `@PreAuthorize`) — это один
    механизм прокси и один набор граблей.

### Транзакции (@Transactional)

??? question "Ты поставил @Transactional, а транзакции нет. Почему?"
    Самая частая причина — **вызов метода изнутри того же класса**.
    `@Transactional` работает через прокси: Spring оборачивает бин, и транзакция
    открывается, когда вызов идёт **через прокси**, снаружи. Если метод `a()`
    вызывает метод `b()` того же объекта, вызов идёт напрямую через `this`,
    минуя прокси, и аннотация на `b()` не срабатывает.

    Другие причины:

    - Метод `private`, `final` или `static` — прокси не может его перехватить.
    - Класс не является бином Spring, объект создан через `new`.
    - Исключение **checked**: по умолчанию откат происходит только на
      `RuntimeException` и `Error`. Для проверяемых нужен `rollbackFor`.
    - Исключение поймано внутри метода и не проброшено — откатывать нечего.

    Ответ показывает, что человек понимает: `@Transactional` — не магия,
    а паттерн Proxy, и у него есть границы применимости.

??? question "REQUIRED, REQUIRES_NEW, NESTED — в чём разница и когда что?"
    - **REQUIRED** (по умолчанию) — присоединиться к существующей транзакции
      или создать новую; всё работает в одной транзакции, откат откатывает всё.
    - **REQUIRES_NEW** — всегда открыть **новую** транзакцию, приостановив
      текущую; коммитится/откатывается независимо. Нужен, когда часть работы
      должна сохраниться независимо от исхода основной — например, запись
      в аудит должна остаться, даже если бизнес-транзакция откатится.
    - **NESTED** — savepoint внутри текущей транзакции: откат отматывает только
      до savepoint, не всю транзакцию (работает не на всех БД/драйверах).

    Важная ловушка REQUIRES_NEW: он берёт **второе** соединение из пула, пока
    первое приостановлено, — под нагрузкой так можно исчерпать пул и получить
    взаимную блокировку на соединениях.

??? question "Куда девается уровень изоляции из @Transactional? Что делает readOnly?"
    Spring не реализует изоляцию сам — он **выставляет её на соединении**
    (`SET TRANSACTION ISOLATION LEVEL`), а обеспечивает БД своим MVCC
    и блокировками. Причём в PostgreSQL `READ_UNCOMMITTED` не существует —
    он молча поднимется до `READ_COMMITTED`.

    `readOnly=true` — прежде всего подсказка ORM: Hibernate отключает dirty
    checking (не сравнивает снапшоты, не флашит изменения) → быстрее
    и безопаснее для чтения; плюс роутер источников данных может направить
    такую транзакцию на реплику. Сама по себе `readOnly` жёстко запись на
    уровне БД не запрещает — это в первую очередь оптимизация.

### Spring MVC

??? question "Что происходит от прихода HTTP-запроса до ответа в Spring MVC?"
    Запрос попадает на встроенный сервер (Tomcat), тот отдаёт его единственному
    **DispatcherServlet** (front controller). Дальше Dispatcher:

    1. через `HandlerMapping` находит метод контроллера по URL и HTTP-методу;
    2. `HandlerAdapter` вызывает его, разрешив аргументы: `@RequestBody`
       десериализуется Jackson'ом через `HttpMessageConverter`,
       `@RequestParam`/`@PathVariable` извлекаются, срабатывает валидация;
    3. метод возвращает объект;
    4. для REST объект сериализуется в JSON тем же конвертером и пишется в ответ;
    5. исключения перехватывает `HandlerExceptionResolver` (`@ControllerAdvice`).

    Всё это в одном потоке из пула Tomcat — модель thread-per-request.

??? question "Модель thread-per-request — что важно понимать для прода?"
    Каждый входящий запрос обрабатывается **одним потоком** из пула сервера
    (Tomcat, по умолчанию ~200) от начала до конца, синхронно. Пока поток ждёт
    БД или внешний вызов — он занят и не обслуживает других.

    Отсюда стабильность прода: если внешняя зависимость затормозила, потоки
    копятся на ожидании, пул исчерпывается, и сервис перестаёт принимать
    запросы — каскадный отказ. Защита: таймауты на все внешние вызовы,
    ограничение пулов, circuit breaker и bulkhead, вынос долгих операций
    в асинхронность. Это же причина, почему блокирующие вызовы в реактивном
    стеке (WebFlux) недопустимы — там потоков мало.

??? question "Фильтр, интерсептор, аспект — чем отличаются и что раньше?"
    Порядок: **фильтр** (Servlet) → DispatcherServlet → **интерсептор**
    (Spring MVC) → **аспект**/контроллер.

    - **Фильтр** работает на уровне сервлет-контейнера, видит сырой
      request/response, ещё до Spring MVC — там живут Spring Security,
      логирование, CORS.
    - **Интерсептор** (`HandlerInterceptor`) знает про выбранный обработчик
      (контроллер), удобен для `preHandle`/`postHandle` вокруг контроллеров.
    - **Аспект** (`@Aspect`) — вокруг любых бинов и методов, самый гранулярный.

    Выбор по уровню: сырой HTTP → фильтр; вокруг контроллеров с доступом
    к handler → интерсептор; вокруг произвольной бизнес-логики → аспект.

### Spring Boot и автоконфигурация

??? question "Как работает автоконфигурация Spring Boot?"
    Boot при старте читает список автоконфигураций из файлов
    `META-INF/spring/…AutoConfiguration.imports` в стартерах (раньше —
    `spring.factories`). Каждая автоконфигурация обвешана условиями
    `@Conditional`: `@ConditionalOnClass` (класс есть в classpath),
    `@ConditionalOnMissingBean` (пользователь не определил свой),
    `@ConditionalOnProperty` и т.п. Применяются только те, чьи условия выполнены.

    Отсюда «магия»: добавил стартер (класс появился в classpath) → сработало
    `@ConditionalOnClass` → бины настроились. А `@ConditionalOnMissingBean`
    гарантирует, что твой бин переопределит дефолтный. Автоконфигурации
    применяются **после** пользовательских бинов.

??? question "Из чего состоит @SpringBootApplication?"
    Это композиция трёх аннотаций:

    - `@SpringBootConfiguration` — тот же `@Configuration` (класс с бинами);
    - `@ComponentScan` — сканировать текущий пакет и ниже на
      `@Component`/`@Service`/…;
    - `@EnableAutoConfiguration` — включить механизм автоконфигурации.

    Понимание нужно, чтобы объяснить, почему бины находятся только в пакете
    приложения и ниже (из-за `@ComponentScan`) и как точечно отключить
    автоконфиг (`exclude` в `@EnableAutoConfiguration`).

??? question "Добавил зависимость — приложение перестало стартовать. Как понять, что сконфигурировалось?"
    Зависимость в Boot — не пассивная библиотека, а **включатель**:
    её появление в classpath активирует автоконфигурации через
    `@ConditionalOnClass`. Классика: добавил `spring-data-jpa` с драйвером —
    Boot стал требовать настроенный `DataSource` и падает без URL базы;
    добавил `starter-security` — эндпоинты закрылись.

    Диагностика штатная:

    1. Прочитать само сообщение об ошибке — у Boot они человеческие
       («Failed to configure a DataSource: url attribute is not specified»).
    2. Запустить с `--debug` — печатается **conditions evaluation report**:
       какие автоконфигурации применились и почему, какие отклонены.
    3. У живого приложения — `/actuator/conditions` и `/actuator/beans`.

    Решения: настроить то, что требует новая конфигурация, переопределить
    бин своим, или точечно исключить —
    `spring.autoconfigure.exclude` / `exclude` в `@SpringBootApplication`.

??? question "Тесты проходят по одному, но падают, когда запускаешь всё сразу. Почему?"
    Тесты **делят состояние**, и порядок/соседство запуска его вскрывает:

    - **Кэш Spring-контекста.** Контекст между тестами переиспользуется,
      и это фича (скорость). Но если тест мутирует бины или использует
      `@MockBean`/другие проперти — конфигурация контекста меняется,
      поднимаются новые контексты, а тесты, полагавшиеся на «свежесть»
      старого, ловят чужое состояние.
    - **Общая БД**: один тест оставил данные — у другого поплыли выборки.
      Лечится транзакционными тестами с откатом (`@Transactional` на тесте)
      или явной очисткой.
    - **Статическое состояние**: static-поля, синглтоны вне Spring,
      системные проперти, фиксированные порты.
    - Реже — реальный порядок выполнения: тесты неявно зависели друг
      от друга.

    Правильный вывод: тест должен сам готовить и убирать своё состояние
    и не полагаться на порядок. `@DirtiesContext` — молоток на крайний
    случай: чинит, но убивает кэширование и время прогона.

??? question "Где в Spring Boot берётся веб-сервер?"
    Он **встроенный** (embedded): Tomcat (по умолчанию), Jetty или Undertow идут
    зависимостью внутри jar, и приложение само поднимает сервер в `main()`,
    а не деплоится во внешний контейнер сервлетов. Поэтому артефакт —
    самодостаточный «fat jar»: `java -jar` и всё.

    Это упрощает контейнеризацию (один процесс = один сервис) и локальный
    запуск. Сменить сервер — исключить `spring-boot-starter-tomcat` и добавить
    стартер нужного.

### Spring Security

??? question "Как устроена аутентификация в Spring Security — путь запроса?"
    Всё висит на **цепочке фильтров** (`SecurityFilterChain`), встроенной
    в общую сервлет-цепочку одним `DelegatingFilterProxy`. Запрос идёт через
    фильтры:

    1. фильтр аутентификации (штатный `UsernamePasswordAuthenticationFilter`
       или свой JWT-фильтр) извлекает креды/токен и передаёт
       `AuthenticationManager`;
    2. тот делегирует списку `AuthenticationProvider`; кто-то проверяет
       (`DaoAuthenticationProvider`: грузит `UserDetails` через
       `UserDetailsService`, сверяет пароль `PasswordEncoder`'ом);
    3. при успехе `Authentication` кладётся в `SecurityContext` (ThreadLocal);
    4. авторизационный фильтр проверяет права по URL, а `@PreAuthorize` —
       на уровне метода через AOP.

    Понимание цепочки объясняет и почему важен порядок фильтров, и почему
    `@Async` теряет контекст (ThreadLocal в другой поток не переносится).

??? question "Эндпоинт закрыт аннотацией @PreAuthorize, но всё равно доступен. Где ошибка?"
    Первый подозреваемый — **нет `@EnableMethodSecurity`**: без него
    `@PreAuthorize` молча игнорируется, никакой ошибки не будет.

    Дальше — стандартные прокси-грабли (методная безопасность работает
    через AOP-прокси, как `@Transactional`):

    - самовызов — метод вызван изнутри того же класса через `this`;
    - метод не public / объект создан через `new`;

    И проверить сам маршрут: возможно, запрос обрабатывает **другой метод
    или другая цепочка фильтров** — например, слишком широкий `permitAll()`
    в URL-правилах не отменяет `@PreAuthorize`, но легко перепутать, какой
    именно эндпоинт дёргается. Отладка: лог Security на DEBUG и проверка,
    каким матчером и методом обработан запрос.

??? question "Твой фильтр не видит аутентифицированного пользователя. Причём тут порядок фильтров?"
    `SecurityContextHolder` заполняется **фильтром аутентификации** внутри
    цепочки Spring Security. Если твой фильтр выполняется **раньше** —
    он видит пустой контекст: аутентификация ещё не произошла.

    Типовые варианты этой ошибки:

    - фильтр зарегистрирован с высоким приоритетом (`@Order`) и встал
      до `springSecurityFilterChain`;
    - фильтр добавлен в цепочку Security до фильтра аутентификации —
      при добавлении нужно явно указывать позицию:
      `http.addFilterAfter(myFilter, AuthorizationFilter.class)`.

    Решение — поставить фильтр после аутентификационного (или использовать
    интерсептор/аспект, которые всегда выполняются после Security).
    Смежная версия той же проблемы: контекст пуст в `@Async`-методе —
    `SecurityContext` живёт в ThreadLocal и в другой поток сам
    не переносится.

??? question "Сессии или JWT — в чём разница и трейдоффы?"
    - **Сессия** — состояние (кто залогинен) хранит сервер у себя, клиент носит
      только id сессии в куке; на каждый запрос сервер смотрит сессию. Просто,
      легко отозвать (удалил сессию — разлогинил), но состояние надо шарить
      между инстансами (sticky sessions или общий Redis) — хуже масштабируется.
    - **JWT** — токен самодостаточен: внутри клеймы (кто, роли, срок) и подпись;
      сервер состояние не хранит, только проверяет подпись → stateless, отлично
      масштабируется и удобно между сервисами.

    Минус JWT: его нельзя легко отозвать до истечения (нужны короткий TTL +
    refresh или чёрный список), а данные в нём видны (base64, не шифр) —
    секреты не класть. Для микросервисов обычно берут JWT.

??? question "Как сервис проверяет JWT и почему ему не нужна база на каждый запрос?"
    Токен состоит из трёх частей: `header.payload.signature`. Сервис берёт
    payload (клеймы: subject, roles, exp) и **проверяет подпись** своим ключом:
    для HMAC — общим секретом, для RSA/EC — публичным ключом выпустившего
    (auth-сервер подписал приватным). Если подпись валидна и срок не истёк —
    доверяет клеймам без похода в БД.

    Именно поэтому JWT stateless и быстрый: и данные, и доказательство
    подлинности — в самом токене. Критично реально проверять подпись и `exp`;
    классические дыры — принять токен, не проверив подпись, или довериться
    алгоритму `none`.

??? question "Когда нужна защита от CSRF, а когда нет?"
    CSRF актуален, когда браузер **автоматически** прикладывает креды к запросу
    — то есть при аутентификации через куки/сессию: злоумышленник с чужого сайта
    шлёт запрос на твой домен, браузер сам добавляет куку, и запрос проходит
    от имени пользователя. Защита — CSRF-токен, который чужой сайт не знает.

    Если аутентификация идёт через заголовок `Authorization: Bearer <token>`
    (JWT), браузер его сам не подставляет — CSRF неактуален, и для stateless
    REST API его обычно отключают. Отсюда правило: сессионное веб-приложение —
    CSRF включён; stateless API на токенах — выключен.

## Данные

??? question "Два пользователя одновременно редактируют одну и ту же строку. Что произойдёт?"
    Без специальных мер — **потерянное обновление** (lost update). Оба прочитали
    строку, оба посчитали новое значение от старого, оба записали. Победит тот,
    кто записал последним, изменения первого исчезнут бесследно.

    Сама база при этом не сломается: на уровне изоляции READ COMMITTED второй
    `UPDATE` просто подождёт, пока первая транзакция отпустит блокировку строки,
    и применится поверх. Проблема не в базе, а в логике «прочитал — посчитал —
    записал».

    Три способа решить:

    - **Оптимистичная блокировка.** В таблице колонка версии, в JPA — поле
      с `@Version`. При обновлении проверяется, что версия не изменилась.
      Если изменилась — `OptimisticLockException`, и приложение решает,
      что делать: показать пользователю конфликт или повторить операцию.
      Подходит, когда конфликты редки.
    - **Пессимистичная блокировка.** `SELECT ... FOR UPDATE` — строка
      блокируется сразу при чтении, второй читатель ждёт. Надёжно, но
      снижает параллелизм и создаёт риск deadlock. Подходит, когда
      конфликты часты.
    - **Атомарное обновление.** Не читать значение в приложение вообще:
      `UPDATE accounts SET balance = balance - 100 WHERE id = ?`. База сама
      выполнит это атомарно. Самый дешёвый вариант, но работает не для
      любой логики.

    Хороший ответ заканчивается вопросом «а что должно произойти с точки
    зрения пользователя» — конфликт данных это в первую очередь продуктовое
    решение, а не техническое.

??? question "Как добавить в таблицу на 50 миллионов строк колонку NOT NULL, не уронив сервис?"
    Одной миграцией — никак. Проблема в двух вещах: долгая блокировка таблицы
    и то, что старая версия кода про новую колонку не знает, а во время
    выкатки в проде работают обе версии одновременно.

    Поэтому в несколько шагов, каждый из которых совместим с обеими версиями кода:

    1. Добавить колонку **nullable**. В PostgreSQL 11+ добавление колонки
       со значением по умолчанию не переписывает всю таблицу, но `NOT NULL`
       на существующие данные — переписывает.
    2. Выкатить код, который **пишет** в новую колонку, но ещё не требует её
       заполненности.
    3. **Забэкфилить** старые строки — не одним `UPDATE` на всю таблицу,
       а батчами, иначе получится гигантская транзакция и раздувание таблицы.
    4. Только теперь навесить ограничение. Чтобы не держать тяжёлую
       блокировку, в PostgreSQL это делают через `CHECK (col IS NOT NULL)
       NOT VALID`, затем `VALIDATE CONSTRAINT` — валидация не блокирует чтение
       и запись.

    Главная мысль ответа: **миграция должна быть обратно совместимой**, потому
    что схема и код обновляются не одновременно.

??? question "Почему индекс есть, а запрос всё равно медленный?"
    Типовые причины:

    - **Функция или приведение типа над колонкой.** `WHERE lower(email) = ?`
      не использует обычный индекс по `email` — нужен индекс по выражению.
    - **Неправильный порядок колонок в составном индексе.** Индекс `(a, b)`
      работает для условия по `a` и по `a, b`, но не для условия только по `b`.
    - **Низкая селективность.** Если условию удовлетворяет половина таблицы,
      планировщик осознанно выберет последовательное чтение — так дешевле.
    - **Устаревшая статистика.** Планировщик оценивает по статистике; если она
      не соответствует данным, он выберет плохой план.
    - **Медленно не из-за поиска.** Запрос тащит слишком много строк или колонок,
      и время уходит на передачу данных, а не на поиск.

    Правильный порядок действий в ответе: не гадать, а посмотреть
    `EXPLAIN ANALYZE` и сравнить ожидаемое количество строк с фактическим.

??? question "Открываешь страницу со списком заказов, а в логах 200 запросов к базе. Что случилось?"
    Проблема **N+1**. Один запрос достал список заказов, а потом на каждый заказ
    Hibernate сходил в базу отдельно — за клиентом или за позициями, потому что
    связь ленивая и обращение к ней произошло уже после основного запроса.

    Как заметить: включить логирование SQL, посмотреть счётчик запросов
    в метриках или написать тест, который считает количество обращений к базе.
    Само по себе это не видно — приложение работает, просто медленно, и тем
    хуже, чем больше данных.

    Как лечить:

    - `JOIN FETCH` в JPQL — забрать связанные сущности одним запросом.
    - `@EntityGraph` на репозитории — то же самое декларативно.
    - Батчи (`@BatchSize`) — вместо 200 запросов будет 20.
    - Проекция или DTO — если нужны три поля, не тащить сущности целиком.

    Важно не отвечать «поставлю EAGER». Это не решение, а перенос проблемы:
    теперь лишние данные будут грузиться всегда, в том числе там, где не нужны.

??? question "Почему LazyInitializationException возникает уже после того, как метод отработал?"
    Ленивая коллекция — это прокси, который сходит в базу при первом обращении.
    Сходить он может только пока жив persistence context, то есть внутри
    транзакции. Если сущность вернули наружу — например, в контроллер, где
    транзакция уже закрыта, — и там обратились к связи, прокси инициализировать
    себя не может.

    Правильный способ — не тащить сущность за пределы сервисного слоя.
    Внутри транзакции собрать DTO с нужными данными и вернуть его. Это заодно
    решает проблему случайной сериализации всей объектной сети в JSON.

    Костыли, которые обычно называют и которые лучше знать как костыли:
    `open-session-in-view` (включён в Spring Boot по умолчанию и маскирует
    проблему, порождая N+1) и `EAGER` на связи.

??? question "Запрос работал быстро, а через месяц стал медленным. Код не менялся. Что произошло?"
    Код тот же, но изменились **данные и статистика**. Основные версии:

    - **Устарела статистика планировщика.** Данных стало кратно больше, а
      `ANALYZE` давно не проходил — планировщик оценивает по старым числам
      и выбирает план, который на новом объёме плох (например, `Nested Loop`
      там, где нужен `Hash Join`). Обновляется `ANALYZE`, обычно это делает
      autovacuum, но он мог не успевать.
    - **Bloat из-за MVCC.** Частые `UPDATE`/`DELETE` оставили массу мёртвых
      версий строк, vacuum не поспевает (часто мешают долгие транзакции) —
      таблица и индексы раздулись, читать нужно больше страниц.
    - **Изменилась селективность.** Раньше условию удовлетворяли единицы строк
      (индекс был выгоден), теперь — заметная доля таблицы, и планировщик
      осознанно ушёл на seq scan.

    Порядок действий тот же: `EXPLAIN ANALYZE`, сравнить оценку с фактом.
    Расходятся — `ANALYZE`; раздулось — разобраться с vacuum и долгими
    транзакциями.

??? question "Транзакция висит и блокирует остальных. Как найти виновника и что делать?"
    Найти — через системные представления PostgreSQL. `pg_stat_activity`
    показывает активные сессии, их `state` (ищем зависшие в `idle in
    transaction` — открыта транзакция, но ничего не делает), `query` и время
    начала. `pg_locks` (и удобный `pg_blocking_pids(pid)`) показывает, **кто
    кого блокирует**.

    Что делать прямо сейчас: аккуратно завершить виновника —
    `pg_cancel_backend(pid)` (отменить запрос) или `pg_terminate_backend(pid)`
    (убить сессию). Транзакция откатится, блокировки отпустятся.

    Почему возникло и как не повторять: чаще всего это **`idle in
    transaction`** — код открыл транзакцию и ушёл в долгую работу/внешний
    вызов, не закоммитив. Лечится дисциплиной коротких транзакций (не держать
    в транзакции сетевые вызовы) и страховкой `idle_in_transaction_session_
    timeout`, которая сама прибивает такие сессии. Долгая висящая транзакция
    вредна ещё и тем, что удерживает снимок и мешает vacuum.

??? question "Две транзакции заблокировали друг друга. Кто разрулит и что увидит приложение?"
    Это **deadlock**: A держит строку 1 и ждёт строку 2, B держит строку 2 и
    ждёт строку 1 — сами они не разойдутся. Разрулит **сам PostgreSQL**: он
    периодически проверяет граф ожиданий, обнаруживает цикл и **убивает одну**
    из транзакций с ошибкой `deadlock detected`. Вторая после этого
    продолжается.

    Приложение у проигравшей транзакции получит исключение (в JPA обёрнутое
    в `CannotAcquireLockException`/подобное) — её нужно **повторить**.

    Профилактика важнее лечения: захватывать блокировки **в одном и том же
    порядке** во всех участках кода (например, всегда по возрастанию `id`) —
    тогда цикл ожидания не образуется в принципе. Плюс короткие транзакции,
    чтобы окно пересечения было меньше.

??? question "Ты проверяешь SELECT-ом, что записи нет, и вставляешь. Почему появляются дубликаты?"
    Классическая **гонка (race condition)** «check-then-act». Между твоим
    `SELECT` (записи нет) и `INSERT` другая транзакция успевает вставить такую
    же запись. На уровне изоляции Read Committed каждая транзакция честно не
    видела чужого незакоммиченного — и обе вставили. Проверка в приложении
    здесь принципиально не спасает: две транзакции проходят её одновременно.

    Правильное решение — переложить гарантию на **базу**, а не на логику:

    - **`UNIQUE`-ограничение** на столбец — база физически не даст вставить
      дубль, вторая вставка упадёт с ошибкой уникальности, которую
      приложение обработает. Это единственная железная гарантия.
    - **`INSERT ... ON CONFLICT DO NOTHING/DO UPDATE`** (upsert) — атомарно
      «вставить или ничего», без гонки.
    - Повышение изоляции до Serializable тоже закроет, но `UNIQUE` проще и
      дешевле.

    Мысль ответа: инварианты уникальности должен держать constraint в БД,
    а не `SELECT`-проверка в коде — «проверил и сделал» в конкурентной среде
    всегда дырявое.

??? question "Удалил 10 миллионов строк, место на диске не освободилось. Почему?"
    Из-за **MVCC**. `DELETE` не стирает строки физически — он лишь помечает их
    версии как мёртвые (проставляет `xmax`), чтобы их перестали видеть новые
    транзакции. Место они всё ещё занимают.

    Освобождает его **VACUUM**: помечает пространство мёртвых версий как
    свободное для переиспользования — но обычный `VACUUM`/autovacuum, как
    правило, **не отдаёт место ОС**, а оставляет его под будущие вставки в эту
    же таблицу. Чтобы реально вернуть место системе, нужен `VACUUM FULL`
    (переписывает таблицу компактно) — но он берёт тяжёлую блокировку и на
    проде опасен; альтернатива — `pg_repack` без долгой блокировки.

    Ещё нюанс: если удаление шло гигантской транзакцией или в базе живёт
    долгая параллельная транзакция, vacuum вообще не сможет убрать версии,
    пока они кому-то «видимы». Отсюда практика — удалять большие объёмы
    батчами.

??? question "Пул соединений к базе исчерпан. Какие причины бывают?"
    Соединений с БД конечное число — приложение берёт их из пула (в Spring
    Boot это **HikariCP**, по умолчанию ~10). «Пул исчерпан» значит: все
    соединения заняты, новый запрос ждёт свободного и по таймауту падает
    (`Connection is not available`). Причины — почти всегда соединения
    **держат дольше, чем нужно**:

    - **Долгие транзакции.** Транзакция удерживает соединение всё своё время.
      Если внутри `@Transactional` есть медленный внешний вызов (HTTP к
      другому сервису, отправка письма) — соединение занято на секунды зря.
      Внешние вызовы выносят за транзакцию.
    - **Утечки — соединение не вернули в пул.** Ручная работа с соединением
      без закрытия, `@Transactional` не там. Обычно это баг, а не нагрузка.
    - **Реальная перегрузка.** Запросов больше, чем пул×скорость обработки;
      часто первопричина — медленные запросы (нет индекса, N+1), которые
      держат соединение дольше.
    - **Слишком маленький пул** под нагрузку — но увеличивать вслепую вредно:
      пул больше, чем БД может обслужить параллельно, только перегружает базу.

    Диагностика: метрики Hikari (активные/ожидающие соединения), логи о
    таймаутах, поиск долгих транзакций и медленных запросов. Порядок починки:
    сначала **сократить время удержания** (быстрые запросы, короткие
    транзакции, вынести внешние вызовы), и только потом трогать размер пула.

    Связка с теорией: соединение живёт под транзакцией и persistence context,
    поэтому «держи транзакции короткими» — это не только про блокировки в БД,
    но и про то, чтобы не выедать пул.

??? question "Пагинация тормозит на глубоких страницах. Почему и чем заменить OFFSET?"
    `LIMIT 20 OFFSET 100000` заставляет БД прочитать и **отбросить** первые
    100 000 строк, прежде чем вернуть 20. Чем глубже страница, тем дороже —
    работа линейно растёт с номером страницы, хотя отдаём всегда 20 строк.

    Замена — **keyset-пагинация** (seek method): запоминать не номер страницы,
    а значение последней строки и продолжать от него:

    ```sql
    SELECT * FROM orders
    WHERE  id < :last_seen_id      -- курсор с прошлой страницы
    ORDER BY id DESC
    LIMIT  20;
    ```

    При индексе по `id` база сразу прыгает в нужное место — стоимость не
    зависит от глубины. Минус — нельзя прыгнуть на «страницу 500» и нужен
    стабильный уникальный ключ сортировки, но для бесконечной ленты/выгрузок
    это как раз то, что нужно. (Подробнее — в теме про пагинацию в Spring
    Data JPA.)

??? question "Как посчитать строки в таблице на 100 миллионов, не положив базу?"
    Важный для PostgreSQL факт: `SELECT count(*)` **точный, но медленный** —
    из-за MVCC база обязана обойти строки и проверить видимость каждой версии
    под текущий снимок, индексом «просто взять число» нельзя. На 100 млн это
    полный проход.

    Варианты по ситуации:

    - **Приблизительно и мгновенно** — взять оценку из статистики планировщика:
      `reltuples` из `pg_class` (или распарсить `EXPLAIN`). Для дашбордов и
      «примерно сколько» этого достаточно, значение поддерживает
      analyze/vacuum.
    - **Точно, но регулярно** — вести **счётчик отдельно**: инкрементировать
      в триггере/коде или пересчитывать по расписанию в отдельную таблицу.
    - Если точный `count` по фильтру нужен часто — правильный индекс хотя бы
      сделает проход index-only, но чуда не будет.

    Суть ответа: в PostgreSQL `count(*)` не бесплатный; для больших таблиц
    выбираем между точностью и скоростью — чаще всего годится приблизительная
    оценка из статистики.

??? question "Кэш и база разошлись. Как это вышло и что делать?"
    Кэш — это **копия** данных, а копия может отстать от источника. Типовые
    причины расхождения:

    - Обновили БД, но **не инвалидировали** кэш (забыли `@CacheEvict`, или
      данные поменял другой сервис/ручной `UPDATE`, о котором кэширующее
      приложение не знает).
    - **Гонка**: между чтением из БД и записью в кэш другой поток успел
      изменить данные — в кэш легло уже устаревшее значение.

    Что делать — по горячим следам сбросить (инвалидировать) затронутые ключи;
    системно — уменьшить вероятность и цену расхождения:

    - **Инвалидировать при записи**: после `UPDATE` удалять ключ (удалять
      надёжнее, чем перезаписывать — следующее чтение подтянет свежее).
    - **TTL как страховка**: даже пропущенная инвалидация протухнет сама.
    - Принять, что кэш даёт **eventual consistency**, и кэшировать только то,
      где короткое окно устаревания допустимо; критичное к свежести не
      кэшировать вовсе.

    Главная мысль: кэш — оптимизация, а не источник правды. Система должна
    оставаться корректной, если кэш сбросить; а идеальной согласованности
    кэша с БД без потери смысла кэша не добиться — выбираем допустимое окно
    устаревания.

??? question "Redis перезапустился. Что стало с данными и почему это может быть нормально?"
    Зависит от роли Redis и настроек персистентности.

    - Если Redis — **кэш**, потеря данных при рестарте **нормальна**: кэш
      просто наполнится заново из БД при следующих запросах (cache-aside).
      Корректность не страдает — только временно просядет скорость, пока кэш
      холодный. Есть побочный риск — **лавина**: весь поток разом идёт в БД,
      пока кэш пуст; смягчается прогревом и разбросом TTL. Именно потому что
      для кэша потеря не страшна, персистентность на нём часто не включают.
    - Если Redis — **источник данных** (сессии, счётчики, очереди), потеря
      недопустима, и должна быть включена персистентность.

    Персистентность Redis: **RDB** — периодический снимок (теряются изменения
    между снимками), **AOF** — журнал команд (потери до ~1 секунды при
    `everysec`, но старт медленнее); часто включают оба. Плюс репликация +
    Sentinel для отказоустойчивости.

    Суть ответа: «нормально» — когда Redis используется как кэш и система
    рассчитана на его пустоту; для данных, которые нельзя потерять, полагаться
    на in-memory без персистентности и репликации нельзя.

??? question "Нужна распределённая блокировка на Redis. Почему это сложнее, чем поставить ключ?"
    «Поставить ключ» кажется достаточным, но наивная реализация дырявая в
    нескольких местах:

    - **Захват должен быть атомарным и с TTL сразу.** `SET key val NX PX
      <ttl>` за одну команду: `NX` — поставить, только если ключа нет, `PX` —
      сразу с TTL. Если ставить ключ и TTL двумя командами, между ними процесс
      может упасть — и замок зависнет навсегда.
    - **TTL — неизбежный компромисс.** Без TTL умерший владелец заблокирует
      ресурс навечно. С TTL — замок может истечь, **пока владелец ещё
      работает**, и его захватит второй: два процесса разом в критической
      секции. Точного TTL не существует.
    - **Снимать замок можно только свой.** Нужен уникальный токен владельца, и
      снятие — «удали ключ, только если значение моё» — атомарно, Lua-скриптом.
      Иначе процесс удалит уже чужой захваченный замок.
    - **В кластере Redis** гарантии слабее: при failover ключ может не доехать
      до новой primary, и замок «раздвоится». Алгоритм Redlock пытается это
      решить несколькими узлами, но он спорный.

    Практический вывод: не писать руками, а брать готовое (**Redisson** —
    `RLock` с watchdog-продлением TTL). И спросить себя, нужна ли вообще
    распределённая блокировка — часто задачу лучше решить идемпотентностью,
    оптимистичной блокировкой или уникальным индексом в БД, которые надёжнее.

??? question "Два сервиса пишут в одну таблицу. Что с этим не так?"
    Это антипаттерн **shared database** — общая таблица делает два сервиса
    скрыто связанными, хотя формально они «независимы»:

    - **Нет владельца схемы.** Любое изменение структуры таблицы одним
      сервисом может сломать второй; менять её страшно, потому что нужно
      согласовывать оба и деплоить синхронно — теряется смысл разделения
      на сервисы.
    - **Размывается бизнес-логика и инварианты.** Правила «как корректно
      менять эти данные» теперь дублируются в двух кодовых базах и легко
      разъезжаются; конкурентные записи двух сервисов дают гонки, которые
      никто целиком не контролирует.
    - **Непонятно, кто отвечает** за консистентность, миграции, нагрузку.

    Как правильно: у каждого куска данных **один сервис-владелец**, который
    даёт остальным доступ через API/события, а не через прямую запись в его
    таблицу. Второму сервису нужно менять данные — он зовёт владельца
    (синхронно через API или асинхронно через сообщение), а не пишет сам.

    Оговорка честности: в монолите или маленькой системе общая таблица —
    норма, и дробить ради дробления не надо. Проблемой это становится именно
    когда сервисы должны разрабатываться и деплоиться независимо.

??? question "Пользователю нужна ссылка на файл в хранилище, но доступ должен быть временным. Как?"
    **Presigned URL** — временная подписанная ссылка на конкретный объект.
    Приложение (у него есть ключи доступа к S3) подписывает запрос к нужному
    файлу с ограниченным сроком и отдаёт клиенту готовую ссылку; клиент идёт по
    ней **прямо в S3**, минуя приложение.

    В ссылку зашиты и хранилищем проверяются: конкретный объект и операция
    (GET на скачивание / PUT на загрузку), срок действия (TTL, скажем 5–15
    минут) и подпись ключами приложения. Ключи в ссылку **не попадают**, бакет
    остаётся приватным — публичный доступ открывать не надо.

    Почему так правильно:

    - **Безопасно** — хранилище приватно, доступ точечный (один объект) и
      временный, права проверяются приложением **до** выдачи ссылки.
    - **Масштабируемо** — тяжёлый файловый трафик идёт напрямую клиент↔S3,
      приложение не пропускает байты через себя и не становится узким местом.

    Нюансы, которые стоит назвать: выданную ссылку до истечения **нельзя
    отозвать**, поэтому TTL держат коротким; presigned работает в обе стороны
    (скачивание и прямая загрузка от клиента); на загрузку размер/тип
    ограничивают политикой, не доверяя клиенту.

    Чего **не** делать: открывать бакет публично или гонять файл через
    приложение ради «ссылки» — первое небезопасно, второе не масштабируется.

    Честно: с S3 в проде не работал, разбирал на пет-проекте — presigned URL
    там ровно для этого сценария и пригодился.

## Взаимодействие

??? question "Consumer прочитал сообщение из Kafka, обработал его и упал, не закоммитив оффсет. Что дальше?"
    После ребалансировки партицию заберёт другой потребитель группы и прочитает
    сообщение **заново** — оффсет-то не сдвинулся. Обработка выполнится второй
    раз со всеми последствиями: второе письмо, второе списание, второй заказ.

    Отсюда главный вывод: Kafka даёт **at-least-once**, а не exactly-once.
    Дубликаты — не аварийная ситуация, а нормальный режим работы, и потребитель
    обязан быть **идемпотентным**:

    - хранить идентификаторы обработанных сообщений и пропускать повторы;
    - выполнять операцию так, чтобы повтор ничего не менял (`UPSERT` вместо
      `INSERT`, установка значения вместо инкремента);
    - опираться на бизнес-ключ и уникальный индекс в базе.

    Обратный порядок — коммитить оффсет до обработки — даёт at-most-once:
    дубликатов не будет, но при падении сообщение потеряется. Для платежей
    это хуже.

??? question "Пользователь дважды кликнул кнопку «Оплатить». Как не списать деньги дважды?"
    Тремя рубежами, а не одним.

    **Ключ идемпотентности.** Клиент присылает вместе с запросом уникальный
    ключ операции. Сервер сохраняет результат по этому ключу; если приходит
    повторный запрос с тем же ключом — возвращает сохранённый ответ, ничего
    не выполняя. Это стандартный подход в платёжных API и он же спасает
    от повторов из-за сетевых ретраев, а не только от двойного клика.

    **Ограничение в базе.** Уникальный индекс по бизнес-ключу операции.
    Последняя линия обороны: даже если логика приложения дала сбой,
    вторая запись просто не вставится. Проверка `SELECT ... if not exists ...
    INSERT` без уникального индекса от гонки не защищает — два потока пройдут
    проверку одновременно.

    **Блокировка на клиенте.** Заблокировать кнопку после нажатия — полезно,
    но это украшение, а не защита: запрос можно повторить и без интерфейса.

    Хороший ответ отдельно проговаривает: повтор запроса приходит не только
    от пользователя. Прокси, балансировщик и библиотека ретраев делают это
    сами, и сервер о причине не знает.

??? question "Что происходит между вводом URL в браузере и появлением страницы?"
    Вопрос не на память, а на то, насколько цельно человек представляет систему.
    Достаточно назвать этапы, не углубляясь в каждый:

    1. Браузер разбирает URL и ищет IP по доменному имени — **DNS**,
       с кэшированием на нескольких уровнях.
    2. Устанавливается **TCP-соединение** с сервером, для HTTPS следом идёт
       **TLS-рукопожатие** с проверкой сертификата.
    3. Уходит **HTTP-запрос**, обычно не напрямую в приложение, а сначала
       в балансировщик или обратный прокси, который выбирает конкретный
       экземпляр.
    4. На стороне бэкенда запрос проходит через контейнер сервлетов,
       цепочку фильтров, диспетчер, попадает в контроллер, оттуда
       в бизнес-логику и в базу.
    5. Ответ возвращается тем же путём, браузер разбирает HTML и
       догружает остальные ресурсы, часть из которых берёт из кэша
       по заголовкам `Cache-Control` и `ETag`.

    Сильный ответ упоминает, что путь запроса — это ещё и место, где всё
    может сломаться: DNS не отвечает, сертификат истёк, балансировщик
    отправил в неготовый под.

## Инженерия и архитектура

??? question "Надо записать в базу и отправить событие в Kafka. Как гарантировать, что произойдёт либо всё, либо ничего?"
    Никак, если делать это в лоб. База и брокер — две разные системы,
    общей транзакции у них нет. Что бы ты ни поставил первым, между двумя
    операциями приложение может упасть: либо данные записаны, а событие
    не ушло, либо событие ушло про изменение, которого не случилось.

    Рабочее решение — **transactional outbox**:

    1. В той же транзакции, что и основные данные, пишем строку в таблицу
       `outbox` — что произошло и какое событие надо отправить.
    2. Транзакция коммитится атомарно: либо есть и данные, и запись
       о событии, либо ничего.
    3. Отдельный процесс читает `outbox` и отправляет события в Kafka,
       помечая отправленные.

    Плата за это — событие уходит **не мгновенно** и может уйти **более одного
    раза** (отправили, но не успели пометить). Поэтому потребитель обязан быть
    идемпотентным. Exactly-once между системами на практике недостижим;
    достижимо at-least-once плюс идемпотентная обработка.

??? question "Сервис A вызывает сервис B. B начал отвечать за 30 секунд вместо 50 миллисекунд. Что произойдёт с A?"
    A ляжет следом, хотя сам полностью исправен. Каждый запрос к B занимает
    поток A на 30 секунд; пул потоков быстро исчерпывается, и A перестаёт
    отвечать вообще — в том числе на запросы, которые к B никакого отношения
    не имеют. Это **каскадный отказ**: один медленный сервис утаскивает
    за собой всю цепочку.

    Что должно быть настроено заранее:

    - **Таймауты** на все внешние вызовы. Вызов без таймаута — это вызов
      с таймаутом «бесконечность».
    - **Circuit breaker.** Если доля ошибок превысила порог, перестать ходить
      в B вообще и сразу возвращать ошибку или заглушку. Периодически
      пробовать снова.
    - **Bulkhead.** Ограничить, сколько потоков может уйти на вызовы B, чтобы
      остальная функциональность продолжала работать.
    - **Деградация.** Решить заранее, что показывать пользователю, если B
      недоступен: кэш, значение по умолчанию, частичный ответ.

    Отдельно стоит упомянуть **retry**: без ограничения попыток и без
    экспоненциальной задержки он не помогает, а добивает уже упавший сервис.

??? question "Релиз надо откатить, но миграция уже применилась. Действия?"
    Зависит от того, была ли миграция **аддитивной** или **разрушительной** —
    и это как раз аргумент, почему миграции проектируют обратно совместимыми.

    - **Миграция аддитивная** (добавили столбец/таблицу/индекс, ничего не
      сломав) — откат кода безопасен: старый код просто не использует новый
      столбец, а тот ему не мешает. Схему **назад не откатывают** — лишний
      риск; новый столбец спокойно живёт неиспользованным до следующего
      релиза. Это идеальный случай, ради которого и держат схему совместимой
      с обеими версиями кода.
    - **Миграция разрушительная** (`DROP COLUMN`, `NOT NULL`, переименование)
      — вот тут больно: старый код, к которому откатываемся, рассчитывает на
      прежнюю схему, а её уже нет. Откатывать схему down-скриптом в проде
      опасно (можно потерять данные), поэтому чаще чинят **forward-фиксом** —
      новой миграцией, возвращающей совместимость, — а не откатом.

    Правильный вывод: разрушительные операции нельзя катить в одном релизе с
    кодом, который на них рассчитывает, — их выносят в следующий релиз, когда
    старого кода уже точно нет (expand-and-contract). Тогда откат любого
    отдельного релиза не встречает несовместимой схемы. То есть проблема
    решается не «как откатить миграцию», а тем, что миграции с самого начала
    делают безопасными к откату.

## Инфраструктура

??? question "Приложение запущено в трёх подах. Внутри — задача по расписанию. Что не так?"
    Она выполнится три раза. `@Scheduled` живёт в каждом экземпляре приложения
    и ничего не знает про соседей. Если задача рассылает письма — пользователь
    получит три письма, если начисляет бонусы — начислит трижды.

    Варианты решения:

    - **Распределённая блокировка.** Перед выполнением задача пытается взять
      блокировку во внешнем хранилище — Redis или таблица в БД. Кто взял,
      тот и работает. Готовое решение — ShedLock.
    - **Вынести из приложения.** Задачу запускает планировщик снаружи:
      CronJob в Kubernetes поднимает отдельный под.
    - **Очередь.** Один инстанс кладёт задание в брокер, обработчики разбирают.

    Отдельно стоит сказать, что задача всё равно должна быть **идемпотентной**:
    блокировка может протухнуть, под — перезапуститься посреди работы.

## Мониторинг и эксплуатация

??? question "Сервис начал тормозить. Твои действия?"
    Главное в ответе — порядок и то, что ты не начинаешь с чтения кода.

    1. **Насколько всё плохо и с какого момента.** Смотрю графики: выросла
       задержка, доля ошибок, нагрузка. Важен момент начала — совпал ли он
       с выкаткой, с ростом трафика, с проблемами у соседнего сервиса.
    2. **Смотрю на перцентили, а не на среднее.** Средние 100 мс могут
       скрывать p99 в три секунды: каждый сотый запрос ждёт три секунды,
       и жалуются именно эти пользователи.
    3. **Все инстансы или один.** Если тормозит один под — дело в нём:
       ресурсы, залипший поток, неудачный узел. Если все — общая причина.
    4. **Где именно время.** По трассировке видно, на каком шаге теряется
       время: своя логика, база, вызов соседа. Без трассировки — по логам
       с идентификатором запроса.
    5. **Типовые подозреваемые.** База: медленные запросы, блокировки,
       исчерпанный пул соединений. Соседний сервис: отвечает медленно
       и держит наши потоки. Память: частые сборки мусора. Внешний сбой:
       выросло время ответа стороннего API.
    6. **Сначала остановить кровь, потом разбираться.** Если совпало
       с выкаткой — откатить, а причину искать после.

??? question "Что нельзя писать в логи?"
    Персональные данные, полные номера карт, пароли, токены и ключи, содержимое
    платёжных документов. Логи хранятся долго, лежат в централизованной системе,
    и доступ к ним есть у широкого круга людей — в банке это прямой инцидент
    безопасности, а не мелкая небрежность.

    Опасность в том, что утечка обычно происходит не намеренно: логируют
    объект запроса целиком, а внутри — токен; ловят исключение и пишут
    тело ответа платёжного шлюза; включают DEBUG на проде «на пять минут».

    Что делать: маскировать чувствительные поля, логировать идентификаторы
    вместо содержимого, следить за уровнями логирования, а вместо полного
    дампа объектов писать осмысленные сообщения. И обязательно —
    идентификатор запроса в каждой записи, иначе в распределённой системе
    логи бесполезны.
